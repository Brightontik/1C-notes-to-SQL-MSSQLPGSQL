# Низкоуровневая трансляция запросов: Из 1С во внутренний SQL СУБД

Этот документ является фундаментальным руководством по работе транслятора платформы 1С (Query Generator). Здесь препарируются высокоуровневые объекты встроенного языка запросов и разбирается их физическое преобразование в реляционный SQL-код для СУБД (MS SQL / PostgreSQL) на уровне страниц данных, индексов и планов выполнения.

---

## Концепт 1: Анатомия, физика и жизненный цикл Временных Таблиц

Временные таблицы (ВТ) — главный инструмент декомпозиции сложных запросов. Однако без понимания физики их работы на уровне дисковой подсистемы сервера СУБД, ВТ могут стать причиной деградации всей базы данных.

### ⚖️ Архитектурный баттл: Менеджер ВТ vs Пакетный Запрос

Способ объявления временной таблицы кардинально меняет поведение системных баз данных СУБД (`tempdb` в MS SQL или схем `pg_temp` в PostgreSQL).



| Критерий | Менеджер ВТ (Программный объект) ❌ | Пакетный запрос (Единый контекст) ✅ |
| :--- | :--- | :--- |
| **Жизненный цикл в СУБД** | Таблица живет в СУБД до тех пор, пока переменная Менеджера не удалена в коде 1С или не вызван метод `.Закрыть()`. | Таблицы создаются, наполняются и принудительно уничтожаются внутри одного серверного вызова СУБД (одна транзакция). |
| **Накладные расходы** | Высокие. Сеанс удерживает дескрипторы таблиц между вызовами серверных процедур, фрагментируя системный кэш СУБД. | Минимальные. Платформа генерирует единый монолитный SQL-скрипт. СУБД мгновенно очищает сеансовую память за один такт. |
| **Риски Highload** | Забытый или вовремя не очищенный менеджер ВТ в длинных циклах вызывает лавинообразное раздувание файлов `tempdb`. | Исключены. Платформа на уровне ядра гарантирует выполнение команд `DROP TABLE` в конце пакета. |

### 🗺️ Физическое именование и изоляция ВТ в СУБД

Когда во встроенном языке 1С выполняется команда `ПОМЕСТИТЬ ВТ_Товары`, транслятор платформы полностью переписывает имя таблицы, изолируя данные на уровне сеанса СУБД.

```text
+-----------------------------------------------------------------------+

| Сервер приложений 1С: Контекст сеанса №142                            |
| Запрос: ПОМЕСТИТЬ ВТ_Товары                                           |
+-----------------------------------------------------------------------+
                                   │
                                   ▼ (Трансляция Query Generator)
+-----------------------------------------------------------------------+

| Сервер СУБД (MS SQL / PostgreSQL)                                     |
|                                                                       |
| 1. Создание физической таблицы в сеансовой схеме:                     |
|    - В MS SQL (tempdb):    CREATE TABLE #tt142_v1 (...);              |
|    - В PostgreSQL (pg_temp): CREATE TEMPORARY TABLE tt142_v1 (...);  |
|                                                                       |
| 2. Изоляция: Другие сеансы СУБД (например, #tt143) не видят данные   |
|    этой таблицы, исключая конфликты параллельного чтения.             |
+-----------------------------------------------------------------------+
```

### ⚡ Физика индексации ВТ: Во что превращается ИНДЕКСИРОВАТЬ ПО

Если временная таблица содержит более нескольких сотен строк и далее участвует в соединениях (`JOIN`), отсутствие явного индекса — это автоматический приговор для СУБД. Вычислительная сложность поиска прыгает до квадратичной **O(N * M)** из-за каскадных `Table Scan` (полное сканирование страниц с диска).

#### 1. Наивный подход (Отсутствие индекса, деградация I/O)
Разработчик помещает большой объем данных во временную таблицу и сразу соединяет её с физической таблицей СУБД.

```bsl
// Вычислительная сложность на уровне СУБД: O(N * M)
// СУБД будет вынуждена перебирать ВТ_Товары вслепую (Table Scan) при каждом сопоставлении строки
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка, Количество ПОМЕСТИТЬ ВТ_Товары ИЗ Документ.Реализация.Товары;

|// ... нет индекса ...
|ВЫБРАТЬ * ИЗ ВТ_Товары КАК ВТ
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном ПО ВТ.Ссылка = Ном.Ссылка";
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Платформа создает «голую» кучу (Heap) без первичного ключа
CREATE TABLE #tt1 (LRef BINARY(16), Qty NUMERIC(15,3)); 
INSERT INTO #tt1 SELECT ... ;

-- При соединении СУБД не может выполнить точечный поиск (Index Seek). 
-- План выполнения: [#tt1] -> TABLE SCAN (последовательное чтение всех страниц памяти) 
-- -> Сброс в HASH MATCH для соединения с Номенклатурой. CPU улетает в 100%.
```

#### 2. Подход Highload-Архитектора (Явная кластеризация ВТ)
Разработчик жестко указывает платформе перестроить структуру временной таблицы под быстрый индексный поиск СУБД.

```bsl
// Вычислительная сложность на уровне СУБД снижается до: O(N log M)
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка, Количество ПОМЕСТИТЬ ВТ_Товары ИЗ Документ.Реализация.Товары

|ИНДЕКСИРОВАТЬ ПО Ссылка; // 👈 Физическая перестройка страниц в СУБД
|// ...
|ВЫБРАТЬ * ИЗ ВТ_Товары КАК ВТ
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном ПО ВТ.Ссылка = Ном.Ссылка";
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
CREATE TABLE #tt1 (LRef BINARY(16), Qty NUMERIC(15,3));
INSERT INTO #tt1 SELECT ... ;

-- Платформа принудительно отдает СУБД команду на генерацию B-Tree структуры:
CREATE CLUSTERED INDEX IX_tt1_Ref ON #tt1 (LRef); -- В MS SQL (физическая сортировка строк)
-- Или CREATE INDEX tt1_ref_idx ON #tt1 (LRef);     -- В PostgreSQL (логический индекс)

-- Теперь при соединении СУБД делает моментальный INDEX SEEK со сложностью O(1) на каждую строку!
```

### 🧠 Протухание Статистики во Временных Таблицах

Самый коварный эффект при трансляции ВТ на больших объемах данных (от 10 000+ строк) — это **гниение плана запроса из-за жесткого отставания статистики** внутри TempDB.

#### Суть проблемы (Внутренние кишки СУБД):
1. Оптимизатор СУБД строит план выполнения всего пакета запросов *до* того, как этот пакет реально выполнился в базе.
2. В момент компиляции плана таблица `#tt1` еще физически пуста (0 строк). Оптимизатор строит гипотезу: *«Данных там нет, значит, для соединения я выберу дешевый Nested Loops (Вложенные циклы)»*.
3. Во время реального выполнения (Runtime) в `#tt1` на лету влетает 200 000 строк. Но сохраненный план СУБД поменять уже не может.
4. В итоге СУБД начинает крутить `Nested Loops` для сотен тысяч строк. Наступает каскадный таймаут, транзакции блокируются.

#### Как с этим бороться архитектурно:
Если во временную таблицу помещается огромный объем данных, пакет необходимо **принудительно разорвать**. Мы завершаем первый запрос, выгружаем данные в менеджер ВТ и стартуем новый запрос с этим же менеджером. При старте второго запроса оптимизатор СУБД заново опрашивает временную базу, видит, что таблица `#tt1` уже заполнена физически, актуализирует статистику распределения данных и строит корректный, быстрый план выполнения (`Hash Match / Merge Join`).

### 🛡️ Способы борьбы с протуханием статистики во Временных Таблицах

Чтобы заставить оптимизатор СУБД пересчитать статистику (Cardinality Estimation) и построить корректный план для наполненной временной таблицы, у Highload-архитектора есть два ключевых паттерна.

---

#### Паттерн 1: Разрыв пакета запросов (Explicit Recompilation)

**Суть метода:** Мы разделяем логику на два (или более) независимых вызова объекта `Запрос`, но связываем их через единый экземпляр `МенеджерВременныхТаблиц`. 

* При первом вызове СУБД наполняет временную таблицу `#tt1` и завершает выполнение. 
* При старте второго вызова оптимизатор СУБД выполняет фазу компиляции заново. Он опрашивает системные таблицы временной базы, видит реальное физическое количество строк в `#tt1` (например, 150 000 строк) и мгновенно перестраивает план с неоптимального `Nested Loops` на производительный `Hash Match`.

##### ❌ Неоптимальный вариант (Единый пакет — слепая статистика)
```bsl
// СУБД строит план для ВСЕГО пакета ОДНОМОМЕНТНО. 
// Оптимизатор думает, что в ВТ_Номенклатура 0 строк, и выбирает Nested Loops для соединения.
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_Номенклатура ИЗ Справочник.Номенклатура ГДЕ ... ;

|ИНДЕКСИРОВАТЬ ПО Ссылка;
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ * ИЗ РегистрСведений.Цены.СрезПоследних ИЗ ВТ_Номенклатура КАК ВТ ...";
Результат = Запрос.Выполнить(); // 💥 Зависание на фазе соединения
```

##### ✅ Оптимальный вариант (Разрыв пакета через Менеджер ВТ)
```bsl
МенеджерВТ = Новый МенеджерВременныхТаблиц;

// Шаг 1: Только наполняем и индексируем ВТ
Запрос1 = Новый Запрос;
Запрос1.МенеджерВременныхТаблиц = МенеджерВТ;
Запрос1.Текст = 
"ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_Номенклатура ИЗ Справочник.Номенклатура ГДЕ ... ;

|ИНДЕКСИРОВАТЬ ПО Ссылка";
Запрос1.Выполнить(); // СУБД физически записала строки в tempdb и создала B-Tree индекс

// Шаг 2: Разрыв пакета. СУБД вынуждена пересчитать статистику перед компиляцией!
Запрос2 = Новый Запрос;
Запрос2.МенеджерВременныхТаблиц = МенеджерВТ;
Запрос2.Текст = 
"ВЫБРАТЬ * ИЗ РегистрСведений.Цены.СрезПоследних КАК Цены 
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТ_Номенклатура КАК ВТ ПО Цены.Номенклатура = ВТ.Ссылка";
// Оптимизатор видит реальный объем ВТ и вместо Nested Loops выбирает Hash Match
Результат = Запрос2.Выполнить(); 
```

#### Паттерн 2: Пакетное наполнение ВТ порциями (Bulk Batching)

**Суть метода:** Метод применяется, когда исходный массив данных для временной таблицы формируется на стороне сервера приложений (например, массив ссылок, переданный из ТЧ документа, или внешняя таблица значений). Попытка загнать в параметр запроса `ГДЕ Ссылка В (&МассивНаМиллионСсылок)` порождает генерацию гигантского SQL-текста с кучей операторов `OR`, что вызывает ошибку `Internal Query Processor Error` или полную деградацию парсера СУБД.

Вместо этого мы берем порции (кванты) данных, наполняем временную таблицу итеративно, а финальный тяжелый запрос выполняем только после полной сборки данных. При этом СУБД динамически обновляет статистику таблицы при превышении порогов модификации строк (`Threshold`).

##### ✅ Оптимальный вариант (Итеративное наполнение порциями)
```bsl
МенеджерВТ = Новый МенеджерВременныхТаблиц;
РазмерПорции = 10000;
Индекс = 0;

Пока Индекс < ИсходныйМассивСсылок.Количество() Цикл
    
    // Вырезаем порцию данных из массива
    Порция = Новый Массив;
    Для Шаг = 1 По РазмерПорции Цикл
        Если Индекс >= ИсходныйМассивСсылок.Количество() Тогда Прервать; КонецЕсли;
        Порция.Добавить(ИсходныйМассивСсылок[Индекс]);
        Индекс = Индекс + 1;
    КонецЦикла;
    
    ЗапросЗаполнения = Новый Запрос;
    ЗапросЗаполнения.МенеджерВременныхТаблиц = МенеджерВТ;
    ЗапросЗаполнения.УстановитьПараметр("ПорцияДанных", Порция);
    
    // Если таблицы еще нет — создаем ее (ПОМЕСТИТЬ), если есть — дописываем (ОБЪЕДИНИТЬ)
    Если Индекс = РазмерПорции Тогда
        ЗапросЗаполнения.Текст = 
            "ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ИтоговыйМассив ИЗ Справочник.Номенклатура ГДЕ Ссылка В (&ПорцияДанных);

            |ИНДЕКСИРОВАТЬ ПО Ссылка";
    Иначе
        // Дописываем данные в существующую таблицу #tt СУБД через промежуточное объединение
        ЗапросЗаполнения.Текст = 
            "ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ВременнаяПорция ИЗ Справочник.Номенклатура ГДЕ Ссылка В (&ПорцияДанных);
            |////////////////////////////////////////////////////////////////////////////////
            |ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_НоваяИтоговая ИЗ ВТ_ИтоговыйМассив

            |ОБЪЕДИНИТЬ ВСЕ
            |ВЫБРАТЬ Ссылка ИЗ ВТ_ВременнаяПорция;
            |////////////////////////////////////////////////////////////////////////////////

            |УНИЧТОЖИТЬ ВТ_ИтоговыйМассив;
            |////////////////////////////////////////////////////////////////////////////////
            |ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ИтоговыйМассив ИЗ ВТ_НоваяИтоговая;

            |ИНДЕКСИРОВАТЬ ПО Ссылка;
            |////////////////////////////////////////////////////////////////////////////////
            |УНИЧТОЖИТЬ ВТ_НоваяИтоговая;

            |УНИЧТОЖИТЬ ВТ_ВременнаяПорция";
    КонецЕсли;
    
    ЗапросЗаполнения.Выполнить();
КонецЦикла;

// Финальный тяжелый расчет выполняется отдельным запросом. 
// СУБД имеет актуальную статистику по всей собранной коллекции!
ФинальныйЗапрос = Новый Запрос;
ФинальныйЗапрос.МенеджерВременныхТаблиц = МенеджерВТ;
ФинальныйЗапрос.Текст = 
    "ВЫБРАТЬ * ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки КАК Остатки
    |ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТ_ИтоговыйМассив КАК Фильтр ПО Остатки.Номенклатура = Фильтр.Ссылка";
Результат = ФинальныйЗапрос.Выполнить(); // СУБД отработает идеально за O(N log M)
```

## Концепт 2: Виртуальные таблицы регистров (Остатки, Обороты, СрезПоследних)

Виртуальные таблицы — это чисто синтаксический сахар платформы 1С. Физически в СУБД никаких «виртуальных» таблиц не существует. Платформа полностью переписывает эти конструкции, собирая данные из двух реальных физических таблиц СУБД: таблицы движений (основная) и таблицы итогов (агрегированная).

### ⚖️ Архитектурный баттл: Физическое устройство таблиц регистров накопления

Посмотрим, из каких кирпичиков транслятор собирает данные на уровне СУБД (для примера взят Регистр Накопления `ТоварыНаСкладах`).



| Компонент | Имя в 1С | Имя в СУБД (MS SQL / PG) | Что хранит физически на диске? |
| :--- | :--- | :--- | :--- |
| **Таблица движений** | `РегистрНакопления.ТоварыНаСкладах` | `_AccumRegXX` | Все исторические записи (приход/расход) с момента запуска системы. Растет бесконечно (**O(N)**). |
| **Таблица итогов** | Нет прямого доступа | `_AccumRegTotalsXX` | Рассчитанные и зафиксированные остатки на начало каждого месяца. Размер относительно стабилен. |

---

### ⚡ Магия параметров ВТ против преступления «ГДЕ» вне параметров

Главная точка деградации СУБД — непонимание разницы между фильтрацией данных *внутри* параметров виртуальной таблицы и наложением условий через классическое секционирование `ГДЕ` за пределами таблицы.

#### 1. Наивный подход (Условие ГДЕ вне параметров — Смерть производительности)
Разработчик обращается к виртуальной таблице остатков, а фильтр по номенклатуре накладывает в конце запроса.

```bsl
// Вычислительная сложность на уровне СУБД: O(N) по всей таблице итогов!
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Номенклатура, Склад, В НаличииОстаток 

|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки КАК Остатки
|ГДЕ Остатки.Номенклатура = &Номенклатура"; // ❌ Архитектурное преступление
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Платформа сначала заставляет СУБД выполнить подзапрос для расчета ВСЕХ остатков
SELECT _Fld101, _Fld102, SUM(_Fld103) 
FROM _AccumRegTotals100 -- 💥 СУБД делает INDEX SCAN или TABLE SCAN всей таблицы итогов!
GROUP BY _Fld101, _Fld102
-- И только ПОТОМ СУБД фильтрует этот гигантский массив в оперативной памяти:
WHERE _Fld101 = @P1; 
```
*Результат:* Если в базе 1 000 000 позиций номенклатуры, СУБД поднимет в RAM остатки по всему миллиону, сгруппирует их, и только ради одной выбранной позиции выбросит 999 999 строк. Диск и CPU улетают в пол.

#### 2. Подход Highload-Архитектора (Строгая фильтрация в параметрах ВТ)
Разработчик жестко ограничивает выборку на этапе генерации запроса внутри круглых скобок.

```bsl
// Вычислительная сложность на уровне СУБД: O(log N) — точечный индексный поиск
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Номенклатура, Склад, В НаличииОстаток 

|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(&Период, Номенклатура = &Номенклатура) КАК Остатки"; // ✅ Стандарт Highload
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Транслятор 1С сразу внедряет условие фильтрации в тело SQL-запроса к СУБД:
SELECT _Fld101, _Fld102, SUM(_Fld103) 
FROM _AccumRegTotals100 
WHERE _Fld101 = @P1 -- 🎯 СУБД делает моментальный INDEX SEEK по кластерному индексу итогов!
GROUP BY _Fld101, _Fld102;
```

### 🗺️ Во что превращается расчет остатков на «нестандартную» дату?

Когда ты запрашиваешь остатки на `ТекущаяДата()` (середина месяца), СУБД не может просто прочитать таблицу итогов, ведь итоги рассчитаны только на начало месяца. Платформа генерирует сложнейшую конструкцию с оператором `UNION ALL`.

Анатомия трансляции выглядит следующим образом:

```text
                               Запрос Остатков на 15-е число
                                             │
                                             ▼
                 +-------------------------------------------------------+
                 |            Query Generator платформы 1С               |
                 +-------------------------------------------------------+
                                             │
                      ┌──────────────────────┴──────────────────────┐
                      ▼ (UNION ALL)                                 ▼ (UNION ALL)
+-------------------------------------------+ +-------------------------------------------+

|          Таблица итогов СУБД              | |         Таблица движений СУБД             |
|            (_AccumRegTotals)              | |              (_AccumReg)                  |
| - Берет срез на начало текущего месяца    | | - Добирает дельту движений с 1-го числа |
| - Быстрый INDEX SEEK за O(1)              | |   месяца по 15-е число                    |
|                                           | | - Ограниченный INDEX RANGE SCAN           |
+-------------------------------------------+ +-------------------------------------------+
                      │                                             │
                      └──────────────────────┬──────────────────────┘
                                             ▼
                              Финальный математический расчет:
                              [Итоги на 1 число] + [Приходы] - [Расходы]
```

Если у тебя в регистре накопления за первые 15 дней месяца записаны миллионы движений (например, крупный ритейл или маркетплейс), расчет остатков на середину месяца «на лету» будет вызывать жесткое торможение. СУБД придется лопатить миллионы строк движений.

**Highload-вывод:** Оперативные остатки в высоконагруженных системах всегда нужно запрашивать на *пустую дату* (когда Период не указан). В этом случае 1С транслирует запрос напрямую к актуальной таблице текущих итогов без привлечения тяжелой таблицы движений и без использования `UNION ALL`.

### 🧠 Оверинжиниринг СрезаПоследних для регистров сведений

Виртуальная таблица `СрезПоследних` для периодических регистров сведений — это самый тяжелый SQL-запрос, который только может сгенерировать платформа.

Когда пишется `РегистрСведений.Цены.СрезПоследних(&Период)`, СУБД выполняет вложенный коррелированный подзапрос:
1. Сначала она делает `SELECT MAX(Период) FROM _InfoRegXX WHERE Период <= @Параметр` с группировкой по измерениям, чтобы найти дату последней записи для каждого товара.
2. Затем она делает повторный `JOIN` к этой же физической таблице по полученным максимальным датам, чтобы вытащить ресурс (саму цену).

**Катастрофа на скриншоте архитекторов:** Если разработчик берет этот и без того тяжелый вложенный подзапрос `СрезПоследних` и соединяет его через `ЛЕВОЕ СОЕДИНЕНИЕ` с другой виртуальной таблицей `Остатки`, СУБД полностью теряет способность использовать индексы. План запроса сваливается в `FULL SCAN + HASH MATCH` [1], оперативная память забивается промежуточными таблицами, а время выполнения уходит в бесконечность.
