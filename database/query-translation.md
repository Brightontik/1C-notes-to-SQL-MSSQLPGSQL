# Низкоуровневая трансляция запросов: Из 1С во внутренний SQL СУБД

Этот документ является фундаментальным руководством по работе транслятора платформы 1С (Query Generator). Здесь препарируются высокоуровневые объекты встроенного языка запросов и разбирается их физическое преобразование в реляционный SQL-код для СУБД (MS SQL / PostgreSQL) на уровне страниц данных, индексов и планов выполнения.

---

## Концепт 1: Анатомия, физика и жизненный цикл Временных Таблиц

Временные таблицы (ВТ) — главный инструмент декомпозиции сложных запросов. Однако без понимания физики их работы на уровне дисковой подсистемы сервера СУБД, ВТ могут стать причиной деградации всей базы данных.

### ⚖️ Архитектурный баттл: Менеджер ВТ vs Пакетный Запрос

Способ объявления временной таблицы кардинально меняет поведение системных баз данных СУБД (`tempdb` в MS SQL или схем `pg_temp` в PostgreSQL).



| Критерий | Менеджер ВТ (Программный объект) ❌ | Пакетный запрос (Единый контекст) ✅ |
| :--- | :--- | :--- |
| **Жизненный цикл в СУБД** | Таблица живет в СУБД до тех пор, пока переменная Менеджера не удалена в коде 1С или не вызван метод `.Закрыть()`. | Таблицы создаются, наполняются и принудительно уничтожаются внутри одного серверного вызова СУБД (одна транзакция). |
| **Накладные расходы** | Высокие. Сеанс удерживает дескрипторы таблиц между вызовами серверных процедур, фрагментируя системный кэш СУБД. | Минимальные. Платформа генерирует единый монолитный SQL-скрипт. СУБД мгновенно очищает сеансовую память за один такт. |
| **Риски Highload** | Забытый или вовремя не очищенный менеджер ВТ в длинных циклах вызывает лавинообразное раздувание файлов `tempdb`. | Исключены. Платформа на уровне ядра гарантирует выполнение команд `DROP TABLE` в конце пакета. |

### 🗺️ Физическое именование и изоляция ВТ в СУБД

Когда во встроенном языке 1С выполняется команда `ПОМЕСТИТЬ ВТ_Товары`, транслятор платформы полностью переписывает имя таблицы, изолируя данные на уровне сеанса СУБД.

```text
+-----------------------------------------------------------------------+

| Сервер приложений 1С: Контекст сеанса №142                            |
| Запрос: ПОМЕСТИТЬ ВТ_Товары                                           |
+-----------------------------------------------------------------------+
                                   │
                                   ▼ (Трансляция Query Generator)
+-----------------------------------------------------------------------+

| Сервер СУБД (MS SQL / PostgreSQL)                                     |
|                                                                       |
| 1. Создание физической таблицы в сеансовой схеме:                     |
|    - В MS SQL (tempdb):    CREATE TABLE #tt142_v1 (...);              |
|    - В PostgreSQL (pg_temp): CREATE TEMPORARY TABLE tt142_v1 (...);  |
|                                                                       |
| 2. Изоляция: Другие сеансы СУБД (например, #tt143) не видят данные   |
|    этой таблицы, исключая конфликты параллельного чтения.             |
+-----------------------------------------------------------------------+
```

### ⚡ Физика индексации ВТ: Во что превращается ИНДЕКСИРОВАТЬ ПО

Если временная таблица содержит более нескольких сотен строк и далее участвует в соединениях (`JOIN`), отсутствие явного индекса — это автоматический приговор для СУБД. Вычислительная сложность поиска прыгает до квадратичной **O(N * M)** из-за каскадных `Table Scan` (полное сканирование страниц с диска).

#### 1. Наивный подход (Отсутствие индекса, деградация I/O)
Разработчик помещает большой объем данных во временную таблицу и сразу соединяет её с физической таблицей СУБД.

```bsl
// Вычислительная сложность на уровне СУБД: O(N * M)
// СУБД будет вынуждена перебирать ВТ_Товары вслепую (Table Scan) при каждом сопоставлении строки
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка, Количество ПОМЕСТИТЬ ВТ_Товары ИЗ Документ.Реализация.Товары;

|// ... нет индекса ...
|ВЫБРАТЬ * ИЗ ВТ_Товары КАК ВТ
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном ПО ВТ.Ссылка = Ном.Ссылка";
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Платформа создает «голую» кучу (Heap) без первичного ключа
CREATE TABLE #tt1 (LRef BINARY(16), Qty NUMERIC(15,3)); 
INSERT INTO #tt1 SELECT ... ;

-- При соединении СУБД не может выполнить точечный поиск (Index Seek). 
-- План выполнения: [#tt1] -> TABLE SCAN (последовательное чтение всех страниц памяти) 
-- -> Сброс в HASH MATCH для соединения с Номенклатурой. CPU улетает в 100%.
```

#### 2. Подход Highload-Архитектора (Явная кластеризация ВТ)
Разработчик жестко указывает платформе перестроить структуру временной таблицы под быстрый индексный поиск СУБД.

```bsl
// Вычислительная сложность на уровне СУБД снижается до: O(N log M)
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка, Количество ПОМЕСТИТЬ ВТ_Товары ИЗ Документ.Реализация.Товары

|ИНДЕКСИРОВАТЬ ПО Ссылка; // 👈 Физическая перестройка страниц в СУБД
|// ...
|ВЫБРАТЬ * ИЗ ВТ_Товары КАК ВТ
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном ПО ВТ.Ссылка = Ном.Ссылка";
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
CREATE TABLE #tt1 (LRef BINARY(16), Qty NUMERIC(15,3));
INSERT INTO #tt1 SELECT ... ;

-- Платформа принудительно отдает СУБД команду на генерацию B-Tree структуры:
CREATE CLUSTERED INDEX IX_tt1_Ref ON #tt1 (LRef); -- В MS SQL (физическая сортировка строк)
-- Или CREATE INDEX tt1_ref_idx ON #tt1 (LRef);     -- В PostgreSQL (логический индекс)

-- Теперь при соединении СУБД делает моментальный INDEX SEEK со сложностью O(1) на каждую строку!
```

### 🧠 Протухание Статистики во Временных Таблицах

Самый коварный эффект при трансляции ВТ на больших объемах данных (от 10 000+ строк) — это **гниение плана запроса из-за жесткого отставания статистики** внутри TempDB.

#### Суть проблемы (Внутренние кишки СУБД):
1. Оптимизатор СУБД строит план выполнения всего пакета запросов *до* того, как этот пакет реально выполнился в базе.
2. В момент компиляции плана таблица `#tt1` еще физически пуста (0 строк). Оптимизатор строит гипотезу: *«Данных там нет, значит, для соединения я выберу дешевый Nested Loops (Вложенные циклы)»*.
3. Во время реального выполнения (Runtime) в `#tt1` на лету влетает 200 000 строк. Но сохраненный план СУБД поменять уже не может.
4. В итоге СУБД начинает крутить `Nested Loops` для сотен тысяч строк. Наступает каскадный таймаут, транзакции блокируются.

#### Как с этим бороться архитектурно:
Если во временную таблицу помещается огромный объем данных, пакет необходимо **принудительно разорвать**. Мы завершаем первый запрос, выгружаем данные в менеджер ВТ и стартуем новый запрос с этим же менеджером. При старте второго запроса оптимизатор СУБД заново опрашивает временную базу, видит, что таблица `#tt1` уже заполнена физически, актуализирует статистику распределения данных и строит корректный, быстрый план выполнения (`Hash Match / Merge Join`).

### 🛡️ Способы борьбы с протуханием статистики во Временных Таблицах

Чтобы заставить оптимизатор СУБД пересчитать статистику (Cardinality Estimation) и построить корректный план для наполненной временной таблицы, у Highload-архитектора есть два ключевых паттерна.

---

#### Паттерн 1: Разрыв пакета запросов (Explicit Recompilation)

**Суть метода:** Мы разделяем логику на два (или более) независимых вызова объекта `Запрос`, но связываем их через единый экземпляр `МенеджерВременныхТаблиц`. 

* При первом вызове СУБД наполняет временную таблицу `#tt1` и завершает выполнение. 
* При старте второго вызова оптимизатор СУБД выполняет фазу компиляции заново. Он опрашивает системные таблицы временной базы, видит реальное физическое количество строк в `#tt1` (например, 150 000 строк) и мгновенно перестраивает план с неоптимального `Nested Loops` на производительный `Hash Match`.

##### ❌ Неоптимальный вариант (Единый пакет — слепая статистика)
```bsl
// СУБД строит план для ВСЕГО пакета ОДНОМОМЕНТНО. 
// Оптимизатор думает, что в ВТ_Номенклатура 0 строк, и выбирает Nested Loops для соединения.
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_Номенклатура ИЗ Справочник.Номенклатура ГДЕ ... ;

|ИНДЕКСИРОВАТЬ ПО Ссылка;
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ * ИЗ РегистрСведений.Цены.СрезПоследних ИЗ ВТ_Номенклатура КАК ВТ ...";
Результат = Запрос.Выполнить(); // 💥 Зависание на фазе соединения
```

##### ✅ Оптимальный вариант (Разрыв пакета через Менеджер ВТ)
```bsl
МенеджерВТ = Новый МенеджерВременныхТаблиц;

// Шаг 1: Только наполняем и индексируем ВТ
Запрос1 = Новый Запрос;
Запрос1.МенеджерВременныхТаблиц = МенеджерВТ;
Запрос1.Текст = 
"ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_Номенклатура ИЗ Справочник.Номенклатура ГДЕ ... ;

|ИНДЕКСИРОВАТЬ ПО Ссылка";
Запрос1.Выполнить(); // СУБД физически записала строки в tempdb и создала B-Tree индекс

// Шаг 2: Разрыв пакета. СУБД вынуждена пересчитать статистику перед компиляцией!
Запрос2 = Новый Запрос;
Запрос2.МенеджерВременныхТаблиц = МенеджерВТ;
Запрос2.Текст = 
"ВЫБРАТЬ * ИЗ РегистрСведений.Цены.СрезПоследних КАК Цены 
|ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТ_Номенклатура КАК ВТ ПО Цены.Номенклатура = ВТ.Ссылка";
// Оптимизатор видит реальный объем ВТ и вместо Nested Loops выбирает Hash Match
Результат = Запрос2.Выполнить(); 
```

#### Паттерн 2: Пакетное наполнение ВТ порциями (Bulk Batching)

**Суть метода:** Метод применяется, когда исходный массив данных для временной таблицы формируется на стороне сервера приложений (например, массив ссылок, переданный из ТЧ документа, или внешняя таблица значений). Попытка загнать в параметр запроса `ГДЕ Ссылка В (&МассивНаМиллионСсылок)` порождает генерацию гигантского SQL-текста с кучей операторов `OR`, что вызывает ошибку `Internal Query Processor Error` или полную деградацию парсера СУБД.

Вместо этого мы берем порции (кванты) данных, наполняем временную таблицу итеративно, а финальный тяжелый запрос выполняем только после полной сборки данных. При этом СУБД динамически обновляет статистику таблицы при превышении порогов модификации строк (`Threshold`).

##### ✅ Оптимальный вариант (Итеративное наполнение порциями)
```bsl
МенеджерВТ = Новый МенеджерВременныхТаблиц;
РазмерПорции = 10000;
Индекс = 0;

Пока Индекс < ИсходныйМассивСсылок.Количество() Цикл
    
    // Вырезаем порцию данных из массива
    Порция = Новый Массив;
    Для Шаг = 1 По РазмерПорции Цикл
        Если Индекс >= ИсходныйМассивСсылок.Количество() Тогда Прервать; КонецЕсли;
        Порция.Добавить(ИсходныйМассивСсылок[Индекс]);
        Индекс = Индекс + 1;
    КонецЦикла;
    
    ЗапросЗаполнения = Новый Запрос;
    ЗапросЗаполнения.МенеджерВременныхТаблиц = МенеджерВТ;
    ЗапросЗаполнения.УстановитьПараметр("ПорцияДанных", Порция);
    
    // Если таблицы еще нет — создаем ее (ПОМЕСТИТЬ), если есть — дописываем (ОБЪЕДИНИТЬ)
    Если Индекс = РазмерПорции Тогда
        ЗапросЗаполнения.Текст = 
            "ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ИтоговыйМассив ИЗ Справочник.Номенклатура ГДЕ Ссылка В (&ПорцияДанных);

            |ИНДЕКСИРОВАТЬ ПО Ссылка";
    Иначе
        // Дописываем данные в существующую таблицу #tt СУБД через промежуточное объединение
        ЗапросЗаполнения.Текст = 
            "ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ВременнаяПорция ИЗ Справочник.Номенклатура ГДЕ Ссылка В (&ПорцияДанных);
            |////////////////////////////////////////////////////////////////////////////////
            |ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_НоваяИтоговая ИЗ ВТ_ИтоговыйМассив

            |ОБЪЕДИНИТЬ ВСЕ
            |ВЫБРАТЬ Ссылка ИЗ ВТ_ВременнаяПорция;
            |////////////////////////////////////////////////////////////////////////////////

            |УНИЧТОЖИТЬ ВТ_ИтоговыйМассив;
            |////////////////////////////////////////////////////////////////////////////////
            |ВЫБРАТЬ Ссылка ПОМЕСТИТЬ ВТ_ИтоговыйМассив ИЗ ВТ_НоваяИтоговая;

            |ИНДЕКСИРОВАТЬ ПО Ссылка;
            |////////////////////////////////////////////////////////////////////////////////
            |УНИЧТОЖИТЬ ВТ_НоваяИтоговая;

            |УНИЧТОЖИТЬ ВТ_ВременнаяПорция";
    КонецЕсли;
    
    ЗапросЗаполнения.Выполнить();
КонецЦикла;

// Финальный тяжелый расчет выполняется отдельным запросом. 
// СУБД имеет актуальную статистику по всей собранной коллекции!
ФинальныйЗапрос = Новый Запрос;
ФинальныйЗапрос.МенеджерВременныхТаблиц = МенеджерВТ;
ФинальныйЗапрос.Текст = 
    "ВЫБРАТЬ * ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки КАК Остатки
    |ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТ_ИтоговыйМассив КАК Фильтр ПО Остатки.Номенклатура = Фильтр.Ссылка";
Результат = ФинальныйЗапрос.Выполнить(); // СУБД отработает идеально за O(N log M)
```

## Концепт 2: Виртуальные таблицы регистров (Остатки, Обороты, СрезПоследних)

Виртуальные таблицы — это чисто синтаксический сахар платформы 1С. Физически в СУБД никаких «виртуальных» таблиц не существует. Платформа полностью переписывает эти конструкции, собирая данные из двух реальных физических таблиц СУБД: таблицы движений (основная) и таблицы итогов (агрегированная).

### ⚖️ Архитектурный баттл: Физическое устройство таблиц регистров накопления

Посмотрим, из каких кирпичиков транслятор собирает данные на уровне СУБД (для примера взят Регистр Накопления `ТоварыНаСкладах`).



| Компонент | Имя в 1С | Имя в СУБД (MS SQL / PG) | Что хранит физически на диске? |
| :--- | :--- | :--- | :--- |
| **Таблица движений** | `РегистрНакопления.ТоварыНаСкладах` | `_AccumRegXX` | Все исторические записи (приход/расход) с момента запуска системы. Растет бесконечно (**O(N)**). |
| **Таблица итогов** | Нет прямого доступа | `_AccumRegTotalsXX` | Рассчитанные и зафиксированные остатки на начало каждого месяца. Размер относительно стабилен. |

---

### ⚡ Магия параметров ВТ против преступления «ГДЕ» вне параметров

Главная точка деградации СУБД — непонимание разницы между фильтрацией данных *внутри* параметров виртуальной таблицы и наложением условий через классическое секционирование `ГДЕ` за пределами таблицы.

#### 1. Наивный подход (Условие ГДЕ вне параметров — Смерть производительности)
Разработчик обращается к виртуальной таблице остатков, а фильтр по номенклатуре накладывает в конце запроса.

```bsl
// Вычислительная сложность на уровне СУБД: O(N) по всей таблице итогов!
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Номенклатура, Склад, В НаличииОстаток 

|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки КАК Остатки
|ГДЕ Остатки.Номенклатура = &Номенклатура"; // ❌ Архитектурное преступление
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Платформа сначала заставляет СУБД выполнить подзапрос для расчета ВСЕХ остатков
SELECT _Fld101, _Fld102, SUM(_Fld103) 
FROM _AccumRegTotals100 -- 💥 СУБД делает INDEX SCAN или TABLE SCAN всей таблицы итогов!
GROUP BY _Fld101, _Fld102
-- И только ПОТОМ СУБД фильтрует этот гигантский массив в оперативной памяти:
WHERE _Fld101 = @P1; 
```
*Результат:* Если в базе 1 000 000 позиций номенклатуры, СУБД поднимет в RAM остатки по всему миллиону, сгруппирует их, и только ради одной выбранной позиции выбросит 999 999 строк. Диск и CPU улетают в пол.

#### 2. Подход Highload-Архитектора (Строгая фильтрация по левому подмножеству индекса)

Чтобы СУБД выполнила максимально производительный точечный поиск `INDEX SEEK`, архитектор обязан передавать параметры фильтрации в виртуальную таблицу строго в порядке их следования в составном индексе СУБД (который равен порядку измерений в конфигураторе 1С).

Предположим, измерения регистра расположены в порядке: **Склад** (первое), **Номенклатура** (второе). Соответственно, составной индекс итогов в СУБД имеет структуру: `Период + Склад + Номенклатура`.

```bsl
// Вычислительная сложность на уровне СУБД: O(log N) — честный Index Seek.
// Передаем параметры строго по порядку следования полей в составном индексе!
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Номенклатура, Склад, В НаличииОстаток 
|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(&Период, Склад = &Склад И Номенклатура = &Номенклатура) КАК Остатки";
```

**Что летит в СУБД под капотом (Низкий грейд):**
```sql
-- Платформа транслирует фильтр, полностью удовлетворяющий левое подмножество составного индекса:
SELECT _Fld101, _Fld102, SUM(_Fld103) 
FROM _AccumRegTotals100 
WHERE _Period = @P1 AND _Fld102 = @P2 AND _Fld101 = @P3 -- Склад (_Fld102) обязан стоять перед Номенклатурой (_Fld101)!
GROUP BY _Fld101, _Fld102;

-- 🎯 План выполнения СУБД: INDEX SEEK по составному некластерному или кластерному индексу.
```

> **Важное архитектурное правило:** Если в параметрах ВТ указать только `Номенклатура = &Номенклатура`, пропустив Склад, который идет в структуре регистра первым, СУБД не сможет использовать составной индекс для поиска. Оптимизатор будет вынужден переключиться на **`INDEX SCAN`**, последовательно вычитывая страницы индекса в поисках нужного значения, что при больших объемах данных мгновенно вызовет деградацию I/O.

### 🗺️ Во что превращается расчет остатков на «нестандартную» дату?

Когда ты запрашиваешь остатки на `ТекущаяДата()` (середина месяца), СУБД не может просто прочитать таблицу итогов, ведь итоги рассчитаны только на начало месяца. Платформа генерирует сложнейшую конструкцию с оператором `UNION ALL`.

Анатомия трансляции выглядит следующим образом:

```text
                               Запрос Остатков на 15-е число
                                             │
                                             ▼
                 +-------------------------------------------------------+

                 |            Query Generator платформы 1С               |
                 +-------------------------------------------------------+
                                             │
                      ┌──────────────────────┴──────────────────────┐
                      ▼ (UNION ALL)                                 ▼ (UNION ALL)
+-------------------------------------------+ +-------------------------------------------+

|          Таблица итогов СУБД              | |         Таблица движений СУБД             |
|            (_AccumRegTotals)              | |              (_AccumReg)                  |
| - Берет срез на начало текущего месяца    | | - Добирает дельту движений с 1-го числа   |
| - INDEX SEEK по B-Tree со сложностью      | |   месяца по 15-е число                    |
|   O(log N) по количеству записей итогов   | | - INDEX RANGE SCAN (Сложность O(K), где    |
|                                           | |   K — количество движений за 15 дней)     |
+-------------------------------------------+ +-------------------------------------------+
                      │                                             │
                      └──────────────────────┬──────────────────────┘
                                             ▼
                              Финальный математический расчет:
                              [Итоги на 1 число] + [Приходы] - [Расходы]

```

Если у тебя в регистре накопления за первые 15 дней месяца записаны миллионы движений (например, крупный ритейл или маркетплейс), расчет остатков на середину месяца «на лету» будет вызывать жесткое торможение. СУБД придется лопатить миллионы строк движений.

**Highload-вывод:** Оперативные остатки в высоконагруженных системах всегда нужно запрашивать на *пустую дату* (когда Период не указан). В этом случае 1С транслирует запрос напрямую к актуальной таблице текущих итогов без привлечения тяжелой таблицы движений и без использования `UNION ALL`.

### 🧠 Оверинжиниринг СрезаПоследних для регистров сведений

Виртуальная таблица `СрезПоследних` для периодических регистров сведений — это самый тяжелый SQL-запрос, который только может сгенерировать платформа.

Когда пишется `РегистрСведений.Цены.СрезПоследних(&Период)`, СУБД выполняет вложенный коррелированный подзапрос:
1. Сначала она делает `SELECT MAX(Период) FROM _InfoRegXX WHERE Период <= @Параметр` с группировкой по измерениям, чтобы найти дату последней записи для каждого товара.
2. Затем она делает повторный `JOIN` к этой же физической таблице по полученным максимальным датам, чтобы вытащить ресурс (саму цену).

**Катастрофа на скриншоте архитекторов:** Если разработчик берет этот и без того тяжелый вложенный подзапрос `СрезПоследних` и соединяет его через `ЛЕВОЕ СОЕДИНЕНИЕ` с другой виртуальной таблицей `Остатки`, СУБД полностью теряет способность использовать индексы. План запроса сваливается в `FULL SCAN + HASH MATCH` [1], оперативная память забивается промежуточными таблицами, а время выполнения уходит в бесконечность.

## 🔁 Повторение и закрепление: Анатомия физических выборок СУБД

Чтобы раз и навсегда зафиксировать в памяти логику работы транслятора 1С (Query Generator), сведем все правила сборки таблиц в одну жесткую координатную сетку. Запомни: СУБД выполняет только то, что продиктовано физическим устройством таблиц платформы.

---

### 🗺️ Матрица трансляции: Во что превращается код 1С на уровне СУБД


| Конструкция в 1С | Какая физическая таблица СУБД читается? | Какая SQL-операция происходит под капотом? | Риск деградации (Highload-эффект) |
| :--- | :--- | :--- | :--- |
| **Остатки на пустую дату** *(Конец времен)* | Только таблица текущих итогов `_AccumRegTotals` | Прямой одиночный `SELECT` с `INDEX SEEK` или `INDEX SCAN`. | **Минимальный.** Самый легкий способ получить срез остатков. |
| **Остатки на границу месяца** *(Например, 01.06.2026 00:00:00)* | Только таблица итогов за конкретный месяц `_AccumRegTotals` | Прямой одиночный `SELECT` по индексу периода за **O(log N)**. | **Низкий.** СУБД считывает уже рассчитанный и сохраненный агрегат. |
| **Остатки на середину месяца** *(Например, 15.06.2026)* | **Две таблицы:** Итоги за начало месяца `_AccumRegTotals` **+** Движения `_AccumReg` | **`UNION ALL`** подзапросов от обеих таблиц с последующим `SUM` и `GROUP BY`. | **Высокий.** Если за 15 дней по регистру прошли миллионы документов, СУБД захлебнется в `INDEX RANGE SCAN` таблицы движений. |
| **Обращение через точку** *(Разыменование: Ссылка.Контрагент.Город)* | Физические таблицы связанных справочников/документов | Каскад скрытых, неявных операторов **`LEFT JOIN`**. | **Катастрофический.** Каждая точка — это новый `LEFT JOIN`. Цепочка из 3–4 точек вызовет перегрузку CPU из-за неоптимальных `Nested Loops`. |
| **СрезПоследних (Периодический РС)** | Физическая таблица регистра сведений `_InfoReg` | Вложенный коррелированный подзапрос с **`MAX(Период)`** + повторный **`JOIN`** к себе же. | **Крайне высокий.** Самый сложный одиночный запрос платформы. Соединение двух таких таблиц «в лоб» гарантированно вешает СУБД. |

### 🧠 Визуальный симулятор: Как транслятор видит твой код

Давай визуализируем два полярных состояния системы, чтобы ты навсегда запомнил разницу между **«Вертикальной сборкой» (UNION ALL)** и **«Горизонтальной сборкой» (LEFT JOIN)**.

#### 1. Ситуация «Вертикальный пирог» (UNION ALL)
Происходит, когда ты заставляешь 1С вычислять состояние данных, которое физически не записано в одной точке. Платформа склеивает прошлое и настоящее по вертикали.

```text
Запрос: РегистрНакопления.ТоварыНаСкладах.Остатки(На Середину Месяца)
                  
                  ┌───────────────────────────────┐
                  │      Транслятор 1С (SQL)      │
                  └───────────────┬───────────────┘
                                  │
         ┌────────────────────────┴────────────────────────┐
         ▼          (Строка к строке по вертикали)         ▼
+-----------------------------------+             +-----------------------------------+

| Таблица Итогов (_AccumRegTotals)  |             | Таблица Движений (_AccumReg)      |
| "Что было на 1-е число месяца"    |             | "Что наторговали с 1 по 15 число" |
+-----------------------------------+             +-----------------------------------+
                                  │                 │
                                  └────────┬────────┘
                                           ▼
                    [ Итог СУБД: UNION ALL + Свертка в памяти ]
```

#### 2. Ситуация «Горизонтальный паровоз» (LEFT JOIN)
Происходит, когда ты вытягиваешь реквизиты связанных объектов через точку или пишешь явные соединения. Платформа прицепляет таблицы справа налево.

```text
Запрос: Выбрать ТекстовоеПоле Из РеализацияТоваровУслуг (Ссылка.Контрагент.ОсновнойДоговор.Валюта)

+-----------------------+      +-----------------------+      +-----------------------+

|  Таблица Документа    |      |  Таблица Контрагента  |      |   Таблица Договора    |
|   (_DocumentXX)       |      |    (_ReferenceXX)     |      |    (_ReferenceYY)     |
+-----------------------+      +-----------------------+      +-----------------------+
            │                              │                              │
            └───────[ LEFT JOIN ]──────────┴───────[ LEFT JOIN ]──────────┘
```

---

### 📌 Архитектурное правило-шпаргалка

1. **Видишь `UNION ALL`** — думай о количестве строк. СУБД будет лопатить таблицы вглубь (историю движений). Твоя задача — минимизировать временной интервал между точкой расчета и точкой рассчитанных итогов.
2. **Видишь `LEFT JOIN`** (или точки в запросе) — думай о ширине и индексах. СУБД будет лопатить таблицы вширь. Твоя задача — проверить, чтобы все поля соединения (Левое и Правое) были проиндексированы, а цепочки разыменования не превышали одного уровня.

## 🎭 Скрытые LEFT JOIN внутри Виртуальных Таблиц

Вопреки расхожему мнению, оператор `LEFT JOIN` генерируется платформой не только при разыменовании полей в секции `ВЫБРАТЬ`. Транслятор 1С вынужден прибегать к **горизонтальным скрытым соединениям** непосредственно на этапе сборки виртуальных таблиц в трех критических сценариях.

---

### Сценарий 1. Измерения с Составным типом данных

Когда измерение регистра имеет составной тип (например, `Справочник.Контрагенты` + `Справочник.ФизическиеЛица`), СУБД на физическом уровне хранит в таблице итогов или движений универсальный бинарный идентификатор и маркер типа.

#### Что происходит под капотом СУБД:
Чтобы выдать текстовые представления или сопоставить типы при базовом чтении остатков, транслятор платформы генерирует скрытые `LEFT JOIN` ко **всем** физическим таблицам СУБД, которые входят в этот составной тип.

```text
Запрос: РегистрНакопления.Взаиморасчеты.Остатки() 
        (Где измерение "Субконто1" имеет составной тип из 3-х справочников)

+-----------------------------+      +-----------------------------+


|  Таблица итогов регистра    |      | Физическая таблица Спр.1    |
|    (_AccumRegTotalsXX)      |      |      (_ReferenceYY)         |
+-----------------------------+      +-----------------------------+
               │                                    │
               ├───────────[ Скрытый LEFT JOIN ]────┘
               │
               ├───────────[ Скрытый LEFT JOIN ]────► [Физическая таблица Спр.2] (_ReferenceZZ)
               │
               └───────────[ Скрытый LEFT JOIN ]────► [Физическая таблица Спр.3] (_ReferenceQQ)
```

**Highload-вывод:** Использование составных типов данных в измерениях высоконагруженных регистров — это добровольное умножение количества `LEFT JOIN` в каждом запросе. Измерения должны быть строго типизированы.

---

### Сценарий 2. Разыменование полей (Точка) внутри параметров ВТ

Разработчик пытается написать «правильный» запрос с фильтрацией внутри круглых скобок, но использует обращение к реквизиту через точку.

```bsl
// Разработчик думает, что делает легкий запрос

|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(, Номенклатура.ВидНоменклатуры = &Вид) КАК Остатки
```

#### Что летит в СУБД под капотом (Низкий грейд):
СУБД физически не может проверить условие по `ВидНоменклатуры`, так как этого поля нет в таблице итогов регистра `_AccumRegTotals`. Платформа сначала строит скрытое соединение, и только потом накладывает фильтр в секции `WHERE`.

```sql
SELECT ... 
FROM _AccumRegTotals100 T1
LEFT JOIN _Reference50 T2 ON T1._Fld101RRef = T2._IDRRef -- 💥 Скрытый JOIN к справочнику Номенклатура
WHERE T2._Fld205RRef = @P1; -- Фильтрация по Виду номенклатуры происходит только ПОСЛЕ соединения!
```

**Highload-вывод:** Этот запрос ничем не лучше, чем условие `ГДЕ` вне параметров. Оптимизатор СУБД свалится в `INDEX SCAN` таблицы итогов регистра, потому что он не может сделать `INDEX SEEK` по значению, которого нет в индексе регистра. 
*Как исправить:* Если требуется частая фильтрация по свойствам номенклатуры, это свойство (например, `ВидНоменклатуры`) необходимо выносить в отдельное физическое измерение регистра или использовать предварительно подготовленную проиндексированную Временную Таблицу.

---

### Сценарий 3. Виртуальная таблица «Обороты» с детализацией по Периодичности

При использовании виртуальной таблицы `Обороты` с параметром периодичности (например, `Периодичность = День`) и выводе полей регистратора.

```bsl
|ИЗ РегистрНакопления.Продажи.Обороты(,,, Периодичность = День) КАК Обороты
```

#### Что происходит под капотом СУБД:
Таблица движений хранит точную дату и время (`_Period` с точностью до секунды). Чтобы сгруппировать данные по дням и сопоставить их со служебной информацией документов, транслятор 1С во многих сценариях генерирует неявный `LEFT JOIN` к **Журналу Документов** (`_DocumentJournalXX`) или системным календарным таблицам платформы. 

**Highload-вывод:** Запрос к таблице `Обороты` с жесткой периодичностью и последующим соединением с другими таблицами порождает сложнейшие SQL-конструкции. Если вам нужны чистые обороты за период без привязки к календарной сетке дней, всегда используйте `Периодичность = Период` (значение по умолчанию) — это избавит СУБД от паразитных горизонтальных соединений.

## Концепт 3: Физика Соединений (JOIN), Подзапросов и Агрегации в СУБД

Когда во встроенном языке 1С пишется ключевое слово `СОЕДИНЕНИЕ`, для транслятора это лишь декларация связи. Оптимизатор СУБД на основе накопленной статистики преобразует эту связь в один из трех низкоуровневых физических операторов выполнения. Задача highload-архитектора — заставить СУБД выбрать наиболее эффективный оператор под текущий объем данных.

---

### ⚖️ Физические операторы JOIN под капотом СУБД

Разберем, как СУБД сопоставляет строки двух таблиц (Ведущей/Внешней и Ведомой/Внутренней) на уровне процессора.


| Оператор СУБД | Как работает алгоритм? | Оптимален для... | Вычислительная сложность | Катастрофа в Highload (Риски) |
| :--- | :--- | :--- | :--- | :--- |
| **Nested Loops** *(Вложенные циклы)* | Для каждой строки первой таблицы СУБД ищет совпадение во второй таблице по индексу. | Малых выборок (до пары тысяч строк) при наличии точного индекса. | **O(N * log M)** | **Смерть от Index Scan.** Если во второй таблице нет индекса, сложность превращается в квадратичную **O(N * M)**, вызывая 100% загрузку CPU. |
| **Hash Match** *(Хэш-соединение)* | СУБД сканирует меньшую таблицу, строит хэш-таблицу в RAM, а затем сканирует вторую таблицу, сверяя хэши. | Больших объемов несимметричных данных без подходящих индексов. | **O(N + M)** | **Spill to TempDB.** Если хэш-таблице не хватает выделенной RAM (Memory Grant), СУБД сбрасывает ее на медленный диск, парализуя систему. |
| **Merge Join** *(Соединение слиянием)* | СУБД берет два предварительно отсортированных по полю связи потока данных и параллельно сравнивает их «в один проход». | Огромных таблиц, которые **уже жестко отсортированы** по индексам. | **O(N + M)** | **Дорогая сортировка.** Если данные не отсортированы индексом, оператор `Sort` в плане запроса сожрет все ресурсы процессора. |

---

### ⚡ Каскадный ад разыменования полей через точку

Самый популярный способ убить производительность сервера СУБД в 1С — это написание длинных цепочек через точку (разыменование ссылок) в секциях выборки или условий.

#### 1. Наивный подход (Длина цепи, парализующая CPU)
Разработчик пишет простой линейный запрос к таблице документов, вытягивая свойства контрагента и договора.

```bsl
// Для разработчика это ОДНА таблица. 
// Для СУБД — это каскад из ТРЕХ скрытых неявных LEFT JOIN!
Запрос = Новый Запрос("ВЫБРАТЬ Ссылка, Контрагент.ОсновнойДоговор.Валюта ИЗ Документ.РеализацияТоваровУслуг");
```

**Что летит в СУБД под капотом (Низкий грейд):**
Транслятор 1С генерирует громоздкий SQL-текст с кучей псевдонимов (`T1`, `T2`, `T3`...):
```sql
SELECT T1._IDRRef, T3._Fld505RRef 
FROM _Document150 T1
LEFT OUTER JOIN _Reference20 T2 ON T1._Fld101RRef = T2._IDRRef -- 💥 Неявный JOIN 1 (Контрагент)
LEFT OUTER JOIN _Reference30 T3 ON T2._Fld202RRef = T3._IDRRef -- 💥 Неявный JOIN 2 (Договор)
-- Оптимизатор СУБД вынужден строить каскад из Nested Loops, умножая накладные расходы на RAM.
```

#### 2. Подход Highload-Архитектора (Явная декомпозиция через ВТ)
Архитектор принудительно разрывает скрытые связи, предварительно выбирая ссылки во временную таблицу и выполняя контролируемые соединения по проиндексированным полям.

```bsl
// Вычислительная сложность жестко контролируется архитектором
Запрос = Новый Запрос;
Запрос.Текст = 
"ВЫБРАТЬ Контрагент ПОМЕСТИТЬ ВТ_УникальныеКонтрагенты ИЗ Документ.РеализацияТоваровУслуг СГРУППИРОВАТЬ ПО Контрагент;
```

### 🧠 Подзапросы в секции условий: `В (ВЫБРАТЬ...)` против Соединений

Конструкция встроенного языка `ГДЕ Ссылка В (ВЫБРАТЬ Ссылка...)` транслируется платформой по-разному в зависимости от контекста и версии платформы.

#### Вариант А: Трансляция в оператор `IN` или `EXISTS`
Если подзапрос возвращает неиндексированный набор данных, СУБД будет выполнять внутренний подзапрос для *каждой* строки внешней таблицы (коррелированный подзапрос), что превращает скорость выполнения в **O(N * M)**.

#### Вариант Б: Автоматическая трансформация в `INNER JOIN` (Semi-Join)
Современные версии платформы 1С и оптимизаторы СУБД часто пытаются раскрыть подзапрос `В (...)` и превратить его в неявное внутреннее соединение. Но если внутри подзапроса используется ключевое слово `РАЗЛИЧНЫЕ` (`DISTINCT`), оптимизатор СУБД может запутаться, построить неверную оценку количества строк (`Cardinality`) и выбрать тяжелый `Hash Match` вместо индексированного слияния.

**Правило Highload-разработки:** Категорически избегай тяжелых вложенных подзапросов в секциях `ГДЕ ... В (...)`. Всегда выгружай данные подзапроса в промежуточную Временную Таблицу с явным вызовом `ИНДЕКСИРОВАТЬ ПО` и соединяй ее через явный `ВНУТРЕННЕЕ СОЕДИНЕНИЕ`. Это гарантирует выбор стабильного плана `Merge Join` или `Nested Loops`.

### ⚡ Физика группировок: `СГРУППИРОВАТЬ ПО` и сброс данных на диск (Spill)

Когда в 1С пишется `СГРУППИРОВАТЬ ПО Номенклатура`, СУБД должна физически собрать одинаковые записи вместе.

1. Если данные уже упорядочены на диске индексом по полю `Номенклатура`, СУБД применит дешевый оператор **Stream Aggregate** — она просто идет по порядку индекса и суммирует ресурсы за **O(N)**.
2. Если индекса нет, СУБД задействует оператор **Hash Aggregate** — начнет строить хэш-таблицу группировки в памяти. 

#### Главный риск Highload-систем: `Spill to TempDB`
Если группируется таблица на миллионы строк с большим количеством уникальных ключей, СУБД запрашивает у операционной системы квант памяти (Memory Grant). Если реальный объем хэш-таблицы превышает выделенный квант, СУБД уходит в состояние **Spill** — начинает сбрасывать промежуточные неотсортированные данные в файлы `tempdb` на жесткий диск. Скорость запроса падает в сотни раз, дисковая полка улетает в 100%, вешая всю информационную базу.

**Как защититься:** Никогда не делай группировок (`СГРУППИРОВАТЬ ПО`) по тяжелым текстовым полям или реквизитам неограниченной длины. Группировка должна идти исключительно по проиндексированным ссылочным полям, числам или коротким строкам фиксированной длины.
