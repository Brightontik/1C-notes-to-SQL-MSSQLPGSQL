# Оптимизация алгоритмов: Борьба с квадратичной сложностью $O(N^2)$ в задачах партионного учета (FIFO/LIFO)

В Highload-системах операции распределения остатков в памяти (списание партий, резервирование, распределение затрат) являются главными точками отказа, если они написаны без учета теории сложности алгоритмов.

---

## 🛑 Анатомия проблемы: Квадратичный взрыв в памяти

При классическом обходе двух таблиц («Что нужно списать» и «Что есть на остатке») разработчики часто используют метод `ТаблицаЗначений.НайтиСтроки()`. 

### Почему это ложит прод?
* Объект `ТаблицаЗначений` в платформе 1С:Предприятие по умолчанию **не имеет индексов**.
* Каждый вызов метода `НайтиСтроки()` принудительно запускает **полное сканирование (линейный поиск)** таблицы с самого начала ($O(M)$, где $M$ — количество строк остатков).
* Если на входе мы имеем $N$ строк потребностей, то общая сложность алгоритма превращается в **$O(N \times M)$**. При однородных данных (например, распределение одной большой позиции по тысячам партий) это чистый квадрат — **$O(N^2)$**.

> **Метрика катастрофы:** На порциях данных в 10 000 строк алгоритм со сложностью $O(N)$ выполнит ~10 000 операций, а алгоритм $O(N^2)$ — **100 000 000 операций**, что намертво заблокирует рабочий процесс кластера 1С по CPU.

---

## 🛠 Паттерн оптимизации: Индексация через Хеш-таблицу

Чтобы снизить сложность до линейной — **$O(N + M)$**, — применяется паттерн предварительной индексации данных в оперативную память с помощью объекта `Соответствие`. Поиск по ключу в `Соответствии` в 1С реализован через хеш-таблицу и отрабатывает за константное время **$O(1)$**.

### Алгоритм:
1. **Первый проход ($O(M)$):** В один цикл обходим таблицу остатков и группируем строки в `Соответствие`, где ключ — это уникальный идентификатор аналитики (составной ключ), а значение — `Массив` подходящих строк-партий.
2. **Второй проход ($O(N)$):** Обходим таблицу потребностей и вместо тяжелого поиска по всей таблице мгновенно достаем готовый массив партий из хеш-таблицы за $O(1)$.

```bsl
    СтрокиПартий = ТаблицаОстатков.НайтиСтроки(Отбор); 
    
    Для Каждого СтрПартии Из СтрокиПартий Цикл
        // Логика распределения...
    КонецЦикла;
КонецЦикла;
```

## ⏪ Исходный код (Было)
*Сложность O(N * M), уязвимость к затягиванию транзакций и неявным запросам.*
```bsl
Функция РаспределитьПартии(ТаблицаОстатков, МассивПараметров, МассивГруппировки, МассивСуммирования, Отказ, ТаблицаНужд = Неопределено) Экспорт
	ТЗ_Движения = Новый ТаблицаЗначений;
	Для Каждого Параметр Из МассивПараметров Цикл
		ТЗ_Движения.Колонки.Добавить(Параметр);
	КонецЦикла;
	
	СтрокаГруппировки = СтрСоединить(МассивГруппировки, ", ");
	СтрокаСуммирования = СтрСоединить(МассивСуммирования, ", ");
	
	Если ТаблицаНужд <> Неопределено Тогда
		ТЗ_Нужды = ТаблицаНужд.Скопировать();
	Иначе
		ТЗ_Нужды = ТаблицаОстатков.Скопировать();
	КонецЕсли;
	
	ИмяПоляПотребности = МассивСуммирования[0];
	
	Для Каждого СтрНужды Из ТЗ_Нужды Цикл
		ОсталосьСписать = СтрНужды[ИмяПоляПотребности];
		ТекущаяСебестоимость = 0;
		
		Отбор = Новый Структура;
		Для Каждого ПолеГруппировки Из МассивГруппировки Цикл
			Отбор.Вставить(ПолеГруппировки, СтрНужды[ПолеГруппировки]);
		КонецЦикла;
		
		СтрокиПартий = ТаблицаОстатков.НайтиСтроки(Отбор);
		
		Для Каждого СтрПартии Из СтрокиПартий Цикл
			Если ОсталосьСписать <=0 Тогда Прервать; КонецЕсли;
			Если СтрПартии.КолОстаток = 0 Тогда Продолжить; КонецЕсли; 
			ВзятьИзПартии = Мин(ОсталосьСписать, СтрПартии.КолОстаток);
			Если ВзятьИзПартии = СтрПартии.КолОстаток Тогда
				СуммаСписания = СтрПартии.СуммаОстаток;
			Иначе
				СуммаСписания = Окр(СтрПартии.СуммаОстаток * ВзятьИзПартии / СтрПартии.КолОстаток, 2);
			КонецЕсли;
			
			НоваяСтрока = ТЗ_Движения.Добавить();
			ЗаполнитьЗначенияСвойств(НоваяСтрока, СтрПартии);
			НоваяСтрока.Количество = ВзятьИзПартии;
			НоваяСтрока.Сумма = СуммаСписания;
			НоваяСтрока.ВидДвижения = ВидДвиженияНакопления.Расход;
			
			ТекущаяСебестоимость = ТекущаяСебестоимость + СуммаСписания;
			ОсталосьСписать = ОсталосьСписать - ВзятьИзПартии;
		КонецЦикла;
		Если ОсталосьСписать > 0 Тогда
			ИмяПоляНоменклатуры = МассивГруппировки[0]; 
			ТекстОшибки = СтрШаблон("Недостаточно Продуктов '%1'. Нужно еще %2 ед.", 
                                СтрНужды[ИмяПоляНоменклатуры], ОсталосьСписать);
		ОбщегоНазначенияКлиентСервер.СообщитьПользователю(ТекстОшибки, ,"Продукты",,Отказ);
		КонецЕсли;
	КонецЦикла;
	Возврат Новый Структура("ТаблицаДляРегистра", ТЗ_Движения);
КонецФункции
```

## 🎯 Моя прикладная реализация (Универсальный алгоритм)

Ниже представлен мой рабочий код функции `РаспределитьПартии` с оптимизированным обходом коллекций и защитой от квадратичной сложности.

```bsl
// Сложность: O(N + M) вместо прежней O(N * M). 
// Полностью убрано сканирование таблицы остатков через НайтиСтроки() внутри цикла.
Функция РаспределитьПартии(ТаблицаОстатков, МассивПараметров, МассивГруппировки, МассивСуммирования, Отказ, ТаблицаНужд = Неопределено) Экспорт
	
	ТЗ_Движения = Новый ТаблицаЗначений;
	Для Каждого Параметр Из МассивПараметров Цикл
		ТЗ_Движения.Колонки.Добавить(Paraметр);
	КонецЦикла;
	
	// Если ТаблицаНужд не передана, берем копию остатков
	Если ТаблицаНужд <> Неопределено Тогда
		ТЗ_Нужды = ТаблицаНужд.Скопировать();
	Иначе
		ТЗ_Нужды = ТаблицаОстатков.Скопировать();
	КонецЕсли;
	
	ИмяПоляПотребности = МассивСуммирования[0];
	КоличествоПолейГруппировки = МассивГруппировки.Количество();
	
	// ------------------------------------------------------------------------
	// ШАГ 1. Индексация таблицы остатков (Построение Хеш-таблицы в памяти)
	// Сложность: O(M), где M - количество строк в ТаблицаОстатков
	// ------------------------------------------------------------------------
	ИндексОстатков = Новый Соответствие;
	
	Для Каждого СтрПартии Из ТаблицаОстатков Цикл
		
		// Для универсального составного ключа используем фиксированный массив значений группировок.
		// В 1С Массив может быть ключом Соответствия. Это быстрее, чем собирать уникальную строку.
		КлючАналитики = Новый Массив(КоличествоПолейГруппировки);
		Для Индекс = 0 По КоличествоПолейГруппировки - 1 Цикл
			КлючАналитики[Индекс] = СтрПартии[МассивГруппировки[Индекс]];
		КонецЦикла;
		
		СтрокиПартийАналитики = ИндексОстатков[КлючАналитики];
		Если СтрокиПартийАналитики = Неопределено Тогда
			СтрокиПартийАналитики = Новый Массив;
			ИндексОстатков.Вставить(КлючАналитики, СтрокиПартийАналитики);
		КонецЕсли;
		
		СтрокиПартийАналитики.Добавить(СтрПартии);
		
	КонецЦикла;
	
	// ------------------------------------------------------------------------
	// ШАГ 2. Распределение потребностей по партиям через обращение к Хеш-таблице
	// Сложность: O(N), где N - количество строк в ТЗ_Нужды
	// ------------------------------------------------------------------------
	
	// Выносим создание служебного массива для поиска из цикла, чтобы не нагружать GC (сборщик мусора)
	КлючПоиска = Новый Массив(КоличествоПолейГруппировки);
	
	Для Каждого СтрНужды Из ТЗ_Нужды Цикл
    
    ОсталосьСписать = СтрНужды[ИмяПоляПотребности];
    ТекущаяСебестоимость = 0;
    
    // Быстро собираем массив УИД-ов/значений (это дешево по памяти)
    МассивКомпонентовКлюча = Новый Массив(КоличествоПолейГруппировки);
    Для Индекс = 0 По КоличествоПолейГруппировки - 1 Цикл
        ТекущееЗначение = СтрНужды[МассивГруппировки[Индекс]];
        
        // Защита от неявных запросов: берем УИД, если это ссылка
        Если XMLТипЗнч(ТекущееЗначение) <> Неопределено И ЗначениеЗаполнено(ТекущееЗначение) Тогда
            МассивКомпонентовКлюча[Индекс] = ВетвьДанныхУИД(ТекущееЗначение); // или просто ПолучитьУИД(ТекущееЗначение)
        Иначе
            МассивКомпонентовКлюча[Индекс] = Строка(ТекущееЗначение);
        КонецЕсли;
    КонецЦикла;
    
    // Превращаем в надежный строковый ключ за ОДНУ операцию выделения памяти
    КлючПоискаСтрока = СтрСоединить(МассивКомпонентовКлюча, "_|_");
    
    // Поиск по строке работает со 100% надежностью за O(1)
    СтрокиПартий = ИндексОстатков[КлючПоискаСтрока];
    
    Если СтрокиПартий <> Неопределено Тогда
			
			Для Каждого СтрПартии Из СтрокиПартий Цикл
				
				Если ОсталосьСписать <= 0 Тогда 
					Прервать; 
				КонецЕсли;
				
				// Важно: Проверяем реальный остаток, так как строка могла быть частично списана на прошлых итерациях ТЗ_Нужды
				Если СтрПартии.КолОстаток <= 0 Тогда 
					Продолжить; 
				КонецЕсли; 
				
				ВзятьИзПартии = Мин(ОсталосьСписать, СтрПартии.КолОстаток);
				
				Если ВзятьИзПартии = СтрПартии.КолОстаток Тогда
					СуммаСписания = СтрПартии.СуммаОстаток;
				Иначе
					СуммаСписания = Окр(СтрПартии.СуммаОстаток * ВзятьИзПартии / СтрПартии.КолОстаток, 2);
				КонецЕсли;
				
				НоваяСтрока = ТЗ_Движения.Добавить();
				ЗаполнитьЗначенияСвойств(НоваяСтрока, СтрПартии);
				НоваяСтрока.Количество = ВзятьИзПартии;
				НоваяСтрока.Сумма = СуммаСписания;
				НоваяСтрока.ВидДвижения = ВидДвиженияНакопления.Расход;
				
				ТекущаяСебестоимость = ТекущаяСебестоимость + СуммаСписания;
				ОсталосьСписать = ОсталосьСписать - ВзятьИзПартии;
				
				// Уменьшаем остаток в исходной таблице, чтобы учесть списание для следующих строк нужд
				СтрПартии.КолОстаток = СтрПартии.КолОстаток - ВзятьИзПартии;
				СтрПартии.СуммаОстаток = СтрПартии.СуммаОстаток - СуммаСписания;
				
			КонецЦикла;
			
		КонецЕсли;
		
		// Проверка на нехватку остатка
		Если ОсталосьСписать > 0 Тогда
			ИмяПоляНоменклатуры = МассивГруппировки[0]; 
			ТекстОшибки = СтрШаблон("Ошибка проведения! Недостаточно товаров по аналитике '%1'. Требуется еще %2 ед.", 
				СтрНужды[ИмяПоляНоменклатуры], ОсталосьСписать);
			
			ВызватьИсключение ТекстОшибки; 
			
		КонецЕсли;
		
	КонецЦикла;
	
	Возврат Новый Структура("ТаблицаДляРегистра", ТЗ_Движения);
КонецФункции

```

## 💎 Micro-оптимизации на Highload (Важно помнить)

1. **Вынос конструирования объектов из цикла:** Конструкторы `Новый Структура` или динамическое добавление колонок, выполняемые внутри больших циклов, сильно нагружают сборщик мусора (Garbage Collector) платформы 1С. Объекты отборов или массивы-буферы нужно инициализировать *до* начала цикла и переиспользовать внутри, очищая или меняя только свойства.
2. **Оптимизация составных ключей:** Операции сложения строк через оператор `+` или `СтрШаблон()` внутри циклов неявным образом перевыделяют оперативную память. Использование предварительно инициализированного фиксированного массива в сочетании с методом `СтрСоединить()` работает в разы быстрее. Для экстремального Highload (миллионы итераций) применяют вложенные хеш-таблицы (Соответствие соответствий), чтобы полностью уйти от строковых операций.
3. **Исключение клиентских методов и затягивания транзакций:** Использование интерфейсных функций (например, `СообщитьПользователю`) внутри серверных транзакций проведения ломает работу фоновых заданий и веб-сервисов. При обнаружении нехватки остатков необходимо немедленно вызывать `ВызватьИсключение`. Это прерывает выполнение кода и мгновенно отправляет команду `ROLLBACK` в СУБД, освобождая заблокированные таблицы для других пользователей и предотвращая технологические тайм-ауты (`TTIMEOUT`).
