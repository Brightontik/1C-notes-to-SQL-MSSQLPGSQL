# Роли и права доступа в 1С

Обеспечение безопасности данных и разграничение полномочий — критически важная часть любого корпоративного прикладного решения. В платформе 1С:Предприятие эта задача решается с помощью гибкой ролевой модели, которая тесно интегрирована со всеми объектами метаданных и напрямую влияет на логику формирования SQL-запросов к СУБД.

---

# 1. Введение: Что такое роли в 1С

**Роли** — это объекты метаданных общего назначения, которые определяют полномочия пользователей на выполнение различных операций в рамках всей системы. Каждая роль представляет собой структурированный набор разрешений (прав) как на общесистемные действия (например, администрирование, запуск тонкого клиента, вывод на печать), так и на интерактивные или программные манипуляции с конкретными объектами конфигурации (справочниками, документами, регистрами).

### Два уровня ограничения прав

Механизм безопасности платформы 1С разделяет проверку прав на два независимых и последовательных этапа.

#### Статические права (объектный уровень / метаданные)
Это базовый уровень авторизации, который жестко определяет доступность того или иного типа данных в принципе. Платформа проверяет, имеет ли право пользователь (на основании назначенных ему ролей) совершать базовые CRUD-операции над таблицей объекта.
* **Основные права**: Чтение (`Read`), Добавление (`Insert`), Изменение (`Update`), Удаление (`Delete`).
* **Специфические права**: Проведение документа (`Posting`), Отмена проведения (`Unposting`), Просмотр (`View`), Интерактивное добавление (`InteractiveInsert`).
* **Логика работы**: Если у пользователя ни в одной из ролей нет права «Чтение» на документ `РеализацияТоваровУслуг`, платформа полностью заблокирует доступ к этой таблице. Пользователь не сможет открыть форму списка, а любые попытки программного обращения (через запросы или объектную технику) вызовут исключение нарушения прав доступа еще до обращения к физическим таблицам СУБД.

#### Динамические права (уровень записей / Row-Level Security / RLS)
Если статические права разрешают доступ к объекту в целом, то динамические права определяют доступность конкретных строк (записей) внутри физических таблиц СУБД на основании контекста и бизнес-логики.
* **Как это работает**: Ограничение настраивается внутри роли для конкретного объекта метаданных с помощью специальных шаблонов ограничений (условий на языке запросов 1С).
* **Пример реализации**: Пользователю статически разрешено изменять документ `ЗаказПокупателя`. Однако настроенное правило RLS динамически ограничивает это право условием: `ГДЕ Ответственный = &ПараметрСеанса.ТекущийПользователь`. В результате менеджер сможет увидеть и отредактировать только те документы, которые он создал сам, в то время как документы коллег останутся для него невидимыми или недоступными для модификации.

# 2. Роли на уровне СУБД: Физическая структура

### Прямой ответ: Создает ли каждая роль отдельную таблицу в SQL?
**Нет.** Роли в платформе 1С — это логические метаданные. Никаких выделенных SQL-таблиц (вроде `_RoleBase` или `_RoleSales`) под каждую новую роль в базе данных не создается. Сами по себе роли не хранят прикладных данных, поэтому с точки зрения классической реляционной схемы СУБД они не требуют персональных сущностей для хранения записей.

### Где хранится структура ролей

Все свойства ролей, их иерархия и матрица статических прав доступа (флаги «Разрешено/Запрещено» для CRUD-операций над справочниками и документами) компилируются в единую структуру метаданных прикладного решения.

* **Таблица-хранилище**: Полное текстовое описание прав и скомпилированные внутренние правила валидации сериализуются, сжимаются и записываются в бинарное BLOB-поле таблицы **`Config`** (основная конфигурация).
* **Механизм поставки**: Структура прав мигрирует вместе с файлом метаданных конфигурации (`.cf`) и применяется ко всей базе данных сразу, а не к отдельным учетным записям.

### Как СУБД узнает, у кого какая роль

Связь между конкретным пользователем и назначенными ему правами реализуется через специальную служебную таблицу платформы на уровне СУБД.

#### Регистрация ролей в таблице `V8USERS`
Список всех учетных записей пользователей информационной базы и настройки их аутентификации находятся в служебной таблице **`V8USERS`**. Назначенные роли распределяются через следующие механизмы внутри этой таблицы:

1. **Поле `RolesID`**: Содержит уникальный идентификатор (GUID) совокупности ролей, назначенных пользователю. Это позволяет платформе быстро определять профили прав.
2. **Поле `Data`**: В этом двоичном поле (BLOB) в упакованном формате XML/JSON хранится расширенная служебная информация об учетной записи, включая детальный плоский список уникальных идентификаторов (GUID) всех ролей, отмеченных флагами в Конфигураторе для данного пользователя.
3. **Поле `AdmRole`**: Отдельный бинарный флаг (0 или 1), сигнализирующий СУБД и платформе о наличии у пользователя роли «Администратор» для быстрого прохождения базовых проверок прав доступа.

#### Взаимодействие с памятью и изоляция СУБД

```text
 Авторизация пользователя при старте сеанса 1С

                       |
                       v
         +---------------------------+
         | Считывание таблицы СУБД   |
         |        [V8USERS]          |
         +---------------------------+

                       |
                       v
 +-------------------------------------------+
 | Кэширование набора GUID ролей в RAM       |
 | Рабочего процесса сервера 1С (rphost)     |
 +-------------------------------------------+

                       |
                       +-----------------------------------+
                       |                                   |
                       v (Операция Чтения/Записи)          v (Формирование SQL-запроса)
 +-------------------------------------------+   +-------------------------------------------+

 | Сервер приложений 1С сам валидирует права |   | Платформа модифицирует T-SQL/PL-pgSQL     |
 | (До СУБД доходят только одобренные вызовы)|   | подставляя параметры RLS из памяти rphost |
 +-------------------------------------------+   +-------------------------------------------+
```

* **Кэширование на старте**: При успешном прохождении аутентификации платформа выполняет запрос к `V8USERS` всего один раз. Набор GUID назначенных ролей считывается, десериализуется и кэшируется в оперативной памяти (RAM) рабочего процесса **`rphost`**, привязанного к сеансу.
* **Отсутствие прямой валидации в СУБД**: Сама СУБД (будь то MS SQL или PostgreSQL) ничего не знает о ролевой модели 1С и не выполняет внутреннюю проверку прав примитивными SQL-командами. Проверку объектных прав берет на себя сервер приложений 1С: он сверяет операцию сеанса с кэшем прав в RAM и либо сразу выдает ошибку «Нарушение прав доступа», либо на лету перестраивает SQL-запрос (в случае RLS) перед отправкой его на исполнение в СУБД.

# 3. Механизм RLS (Row Level Security): Превращение ролей в SQL-код

### Трансляция условий RLS в СУБД

Когда у пользователя активирована роль с ограничением на уровне записей (RLS), платформа 1С перестает отправлять в СУБД «чистые» SQL-запросы. Архитектурно это устроено как **динамический препроцессинг**: платформа перехватывает любой запрос на встроенном языке (или интерактивное действие в интерфейсе), анализирует ограничения текущего набора ролей в RAM сервера `rphost` и компилирует итоговый T-SQL/PL-pgSQL код, внедряя условия ограничения непосредственно в тело запроса.

### Пример модификации запроса

Рассмотрим классический пример, когда менеджеру разрешено видеть контрагентов только своей организации. 

1. **Исходный запрос разработчика на встроенном языке 1С:**
   ```bsl
   Запрос = Новый Запрос;
   Запрос.Текст = "ВЫБРАТЬ Код, Наименование ИЗ Справочник.Контрагенты";
   Результат = Запрос.Выполнить();
   ```

2. **Модифицированный SQL-запрос, который платформа отправляет в СУБД:**
   При наличии RLS по доступным организациям (данные о которых берутся из параметров сеанса или служебных регистров прав БСП) простой селект превращается в тяжелую конструкцию с подзапросами:

   ```sql
   SELECT 
       T1._Code AS P1,
       T1._Description AS P2
   FROM 
       _Reference12 T1 -- Физическая таблица справочника Контрагенты
   WHERE 
       -- Платформа принудительно внедряет условие проверки прав:
       T1._Fld34_RRef IN (
           SELECT 
               T2._Fld57_RRef 
           FROM 
               _InfoReg56 T2 -- Служебный регистр сведений прав доступа (например, ТаблицаПрав)
           WHERE 
               T2._Fld58_RRef = @P1 -- Идентификатор текущего пользователя из параметров сеанса
               AND T2._Fld59 = 0x01 -- Флаг разрешения на чтение
       )
   ```

#### Критический оверхед для СУБД:
Из этого примера видно, что для SQL-сервера объем работы увеличивается кратно. Вместо прямого чтения индекса по таблице `_Reference12` СУБД вынуждена выполнять операцию `JOIN` или вложенный подзапрос `IN (SELECT...)` к таблицам прав для каждой сканируемой строки. Если служебные таблицы прав (`_InfoReg56`) не оптимизированы или не имеют актуальной статистики, оптимизатор СУБД может ошибиться с планом, что приведет к полному сканированию (`Table Scan`) и высокой утилизации CPU.

### Планы обмена и регистрация изменений

Роли и права доступа оказывают прямое влияние на подсистему синхронизации данных (планы обмена) и связанные с ними физические таблицы регистрации изменений.

* **Физические таблицы изменений**: Для каждого объекта метаданных, включенного в план обмена, СУБД создает отдельную служебную таблицу регистрации изменений с префиксом `_DocumentChngR<n>` (для документов) или `_ReferenceChngR<n>` (для справочников). Она хранит ссылки на измененные объекты и узлы-получатели.
* **Влияние прав на изменение (`Update`)**: Если у роли пользователя нет статических или динамических прав на изменение (`Update`/`Edit`) объекта, платформа на программном уровне блокирует запись. СУБД даже не пытается инициировать транзакцию записи, соответственно, в таблицу регистрации изменений `_DocumentChngR<n>` новые записи не добавляются.
* **Rls при выгрузке данных (Чтение изменений)**: Сама по себе таблица регистрации изменений не знает про RLS. Однако при выгрузке данных (когда вызывается метод `ВыбратьИзменения()`), платформа выполняет чтение зарегистрированных ссылок. Если у текущего сеанса (например, у служебного пользователя интеграции) настроен RLS, то SQL-запрос к таблице изменений будет точно так же модифицирован подзапросами ограничений. Объекты, к которым нет доступа на чтение, будут отфильтрованы и не попадут в пакет обмена, даже если они были зарегистрированы к выгрузке.

# 4. Влияние ролей на производительность и СУБД (Критично для DBA)

Для системного администратора баз данных (DBA) ролевая модель 1С, обремененная ограничениями RLS, является главным источником нестабильности планов запросов и необоснованной утилизации ресурсов сервера.

### Взрывной рост планов запросов (Парадокс RLS)

Один и тот же программный код (например, типовой отчет или алгоритм подбора номенклатуры), запущенный разными пользователями, демонстрирует диаметрально противоположную производительность на уровне СУБД:

* **Под «Полными правами»**: Платформа полностью отключает препроцессинг RLS. В СУБД уходит линейный, чистый SQL-запрос. Оптимизатор (MS SQL или PostgreSQL) строит идеальный план, используя эффективный поиск по индексам (`Index Seek`). Время выполнения — **доли секунды**.
* **Под ролью «Менеджер» (с активным RLS)**: Тот же самый запрос превращается в монструозную конструкцию с множеством вложенных `LEFT JOIN` и `WHERE IN (SELECT...)` к таблицам прав доступа и параметров сеанса. 
* **Результат для СУБД**: Сложность дерева запроса возрастает экспоненциально. Оптимизатор СУБД часто упирается в лимит времени расчета плана (в MS SQL это состояние `Reason for Early Termination = Time Out`) и выдает первый попавшийся, неоптимальный план. Вместо быстрого поиска СУБД переходит на полное сканирование таблиц (`Table Scan`/`Index Scan`) и уходит в тяжелые хэш-соединения (`Hash Match`), из-за чего скорость выполнения падает в **10–100 раз**.

### Проблема кэширования планов (Parameter Sniffing)

Сложные RLS-ограничения в ролях делают систему крайне уязвимой к эффекту **Parameter Sniffing** (подглядывание параметров) при кэшировании планов запросов в СУБД.

1. **Первый запуск (Плохой сценарий)**: Пользователь с жестко ограниченными правами (доступна всего 1 организация из 100) открывает форму. Платформа генерирует RLS-запрос, где параметром выступает ID этой одной организации. Оптимизатор СУБД видит, что данных мало, строит под них план (например, через `Nested Loops`) и сохраняет этот план в кэш (Plan Cache).
2. **Второй запуск (Катастрофа)**: В систему заходит крупный руководитель, у которого в параметрах сеанса открыт доступ к 95 организациям из 100, и выполняет то же действие. СУБД берет из кэша старый план, рассчитанный на *один* идентификатор, и пытается применить его к огромному массиву данных. 
3. **Итог**: `Nested Loops` на огромной выборке превращаются в бесконечный цикл. Процессор сервера базы данных утилизируется в 100%, дисковая очередь растет, а у пользователей зависают интерфейсы.

### Методы оптимизации для DBA

Чтобы предотвратить коллапс производительности СУБД, вызванный тяжелыми ролями и RLS, применяются два фундаментальных инженерных подхода.

#### 1. Перенос проверок в Привилегированные общие модули
Большинство транзакционных бизнес-операций (проведение документов, списание остатков, начисление резервов) не требуют контроля RLS для каждой строки «на лету». 
* **Решение**: Логику проведения и массовой обработки данных изолируют в общих модулях с флагом `Привилегированный`. 
* **Эффект для СУБД**: В момент записи данных RLS-препроцессор отключается. СУБД получает легкие, параметризованные запросы без подзапросов прав. Проверка прав пользователя при этом переносится на верхний (интерфейсный) уровень 1С всего один раз перед запуском процесса, полностью разгружая СУБД от транзакционного оверхеда.

#### 2. Проактивное индексирование полей RLS
Поля, по которым в ролях настроены ограничения RLS (например, Измерение `Организация` или `Подразделение`), должны быть гарантированно проиндексированы на уровне СУБД.

```text
Типичная структура физического индекса для таблицы с RLS:

+------------------------------------------------------------+

| [Индекс СУБД] = _Fld150_RRef (Организация) + ... (Реквизиты)|
+------------------------------------------------------------+
         |
         v
 Позволяет оптимизатору СУБД принудительно выполнять Index Seek 
 по значению из Параметров Сеанса, отсекая 90% мусорных строк
```

* **Штатное индексирование**: В конфигураторе 1С для таких реквизитов необходимо обязательно взводить свойство `Индексировать` или `Индексировать с доп. упорядочиванием`.
* **Прямое вмешательство DBA (SQL-индексы)**: Если штатных индексов 1С недостаточно (например, RLS использует композитное условие по двум полям), DBA может создать покрывающие некодовые индексы непосредственно в СУБД. Индекс, начинающийся с физического поля организации (`_Fld<n>_RRef`), позволит оптимизатору СУБД моментально сопоставлять входящие из параметров сеанса GUID-идентификаторы, минуя дорогостоящие операции сканирования основных таблиц данных.
