# Общие модули в 1С

# 1. Введение: Что такое общие модули в 1С

**Общие модули** — это объекты метаданных общего назначения, которые представляют собой контейнеры для хранения переиспользуемых процедур и функций. Они позволяют реализовать принцип повторного использования кода (DRY — Don't Repeat Yourself), выносить общую бизнес-логику и сервисные алгоритмы в единое место и вызывать их из любых других модулей конфигурации (модулей объектов, форм, менеджеров и т.д.).

### Ключевые свойства (флаги компиляции и исполнения)

Поведение, доступность и контекст выполнения кода в общем модуле полностью определяются его свойствами в палитре свойств (флагами). 

#### Клиент / Сервер (Директивы компиляции)
Эти флаги определяют, в какой конфигурации среды выполнения и на каком физическом компьютере будет скомпилирован и запущен код общего модуля.
* **Сервер**: Код выполняется на стороне сервера приложений «1С:Предприятие». Доступен полный спектр операций с базой данных, запросы и объектная техника.
* **Клиент (управляемое приложение)**: Код выполняется на стороне клиентского рабочего места (в тонком или толстом клиенте). Предназначен для интерактивной работы, открытия форм и работы с локальными файлами.

#### Внешнее соединение
* Этот флаг определяет доступность процедур и функций общего модуля при подключении к базе данных из внешних систем через COM-соединение (`COMConnector`). Если флаг выключен, внешняя программа не сможет вызвать код из этого модуля.

#### Глобальный
* При включении этого флага модуль становится частью глобального контекста. Экспортные процедуры и функции такого модуля можно вызывать из других мест напрямую (например, просто `МояПроцедура()`), без обязательного указания имени самого модуля через точку (`ИмяОбщегоМодуля.МояПроцедура()`). 

#### Привилегированный
* Один из важнейших флагов безопасности и производительности. Код привилегированного модуля выполняется на сервере **без проверки прав доступа** текущего пользователя (включая ограничения RLS — Row-Level Security). Это необходимо для выполнения служебных операций (например, запись в защищенный регистр сведений), на которые у обычного пользователя нет прямых прав.

---

### Матрица применимости свойств общего модуля

| Флаг / Свойство | Место исполнения | Проверка прав (RLS) | Особенности вызова / Назначение |
| :--- | :--- | :--- | :--- |
| **Сервер** | Сервер 1С | Да | Стандартный серверный контекст для работы с БД. |
| **Клиент** | Рабочее место | Да | Ограничен в работе с данными, предназначен для интерфейсной логики. |
| **Внешнее соединение** | Процесс внешнего подключения | Да | Позволяет вызывать функции модуля внешним программам через COM/OLE. |
| **Привилегированный** | Только Сервер | **Нет** | Игнорирует ограничения прав доступа для служебных операций. |
| **Глобальный** | Зависит от Клиент/Сервер | Да | Расширяет глобальный контекст (вызов функций без префикса `ИмяМодуля.`). |


# 2. Общие модули на уровне СУБД: Физическая реальность

### Прямой ответ: Создают ли общие модули таблицы или колонки в SQL?
**Нет.** Общие модули — это исключительно алгоритмическая сущность конфигурации. В реляционной базе данных не создается никаких физических таблиц или колонок (вроде `_CommonModule`), предназначенных для хранения промежуточных вычислений или результатов работы процедур и функций. 

С точки зрения SQL-сервера, выполнение стандартного кода общего модуля (не содержащего явных запросов или транзакционных команд записи данных) незаметно, так как обработка инструкций ложится на плечи сервера приложений 1С или клиента.

### Где хранится сам исходный код

Весь программный код конфигурации, включая текст общих модулей, является частью метаданных. Он сериализуется платформой, сжимается алгоритмом дефляции (Deflate) и записывается в базу данных в виде бинарных данных (BLOB).

* **Таблица `Config`**: Служит основным хранилищем для работающей конфигурации базы данных.
* **Поиск по UUID**: Каждому общему модулю платформа присваивает уникальный идентификатор (UUID). Записи в таблице `Config` идентифицируются по строковым ключам (поле `FileName`), содержащим этот UUID.
* **Состав хранимых данных**: Для каждого общего модуля в СУБД хранятся две связанные структуры:
  1. **Исходный текст модуля** — текстовый программный код на языке 1С, написанный разработчиком.
  2. **Скомпилированный байт-код** — внутреннее промежуточное представление кода, подготовленное платформой для быстрого исполнения виртуальной машиной 1С.

### Взаимодействие с памятью и кэширование

Процесс обработки и выполнения кода общих модулей оптимизирован с использованием оперативной памяти серверов.

```text
+--------------------------------------------------------------------------+

|                        Оперативная память (RAM)                          |
|  Рабочий процесс rphost / Сеанс клиента                                  |
|  +--------------------------------------------------------------------+  |
|  | [Кэш байт-кода модулей] <- Мгновенное исполнение виртуальной машиной|  |
|  +--------------------------------------------------------------------+  |
+--------------------------------------------------------------------------+
                                     ^

                                     | (Загрузка при первом вызове/старте)
                                     |
+--------------------------------------------------------------------------+

|                             Уровень СУБД                                 |
|  Таблица [Config]                                                        |
|  +--------------------------------------------------------------------+  |
|  | Строка модуля (Поле Data: Сжатый бинарный исходный код + байт-код) |  |
|  +--------------------------------------------------------------------+  |
+--------------------------------------------------------------------------+
```

1. **Первичная загрузка**: При старте сервера приложений 1С (процесса `rphost`) или при первом обращении к экспортному методу общего модуля в сеансе, платформа выполняет SQL-запрос к таблице `Config`.
2. **Распаковка и кэширование**: Сжатые бинарные данные извлекаются, декомпрессируются, и готовый к исполнению байт-код помещается в оперативную память (RAM) рабочего процесса `rphost` (или в память клиента, если модуль скомпилирован под клиентский контекст).
3. **Последующие вызовы**: Все повторные обращения к функциям этого общего модуля происходят напрямую из оперативной памяти, минуя обращения к СУБД, что гарантирует максимальное быстродействие. Кэш сбрасывается только при обновлении конфигурации базы данных или перезапуске рабочих процессов.


# 3. Назначение, свойства и примеры использования

В промышленной разработке общие модули классифицируются по архитектурным ролям. Разумное сочетание их свойств позволяет минимизировать нагрузку на СУБД и значительно ускорить работу системы.

### Серверный код общего назначения

Такие модули компилируются исключительно на сервере (`Сервер = Истина`, `Клиент = Ложь`) и аккумулируют в себе всю бизнес-логику системы, тяжелые вычисления, транзакционные операции и работу с запросами.

* **Типичный пример (`ОбщегоНазначенияСервер`)**: Содержит универсальные функции (например, получение среза последних для списка номенклатуры, расчет себестоимости, чтение реквизитов группы объектов одним запросом).
* **Контекстная изоляция**: Вынесение логики из модулей форм на сервер снижает сетевой трафик между клиентом и сервером приложений, так как на клиент возвращаются только готовые примитивные типы данных (числа, строки) или структуры, а не тяжелые ссылки и таблицы.

### Привилегированные модули как инструмент оптимизации

Флаг `Привилегированный` у общего модуля — это не только инструмент обхода ограничений безопасности, но и важнейший механизм оптимизации работы с СУБД.

* **Зачем нужны**: Они позволяют выполнять операции под полными правами, полностью игнорируя настройки RLS (Row-Level Security — ограничение доступа на уровне записей). Это критично при проведении документов, когда рядовой менеджер (с ограниченным доступом) должен сделать запись в общие регистры остатков или взаиморасчетов.
* **Профит для СУБД**: При включенном RLS платформа 1С при каждом запросе к таблице динамически модифицирует SQL-текст, дописывая к нему тяжелые подзапросы (инструкции `WHERE EXISTS...` с проверкой прав по группам доступа). Это приводит к:
  1. Разрастанию планов запросов в SQL-сервере.
  2. Избыточному сканированию индексов служебных таблиц прав.
  3. Резкому росту нагрузки на CPU сервера базы данных.
* **Результат оптимизации**: Вызов запроса внутри **привилегированного** модуля заставляет платформу генерировать «чистый» SQL-запрос без условий RLS. План выполнения такого запроса становится оптимальным, он отрабатывает в разы быстрее, а нагрузка на СУБД и CPU снижается до минимума.

### Повторное использование значений (Кэширование)

Свойство общего модуля **«Повторное использование возвращаемых значений»** (в коде часто называемое *кэшем функций*) — мощный инструмент борьбы со «спам-запросами» к базе данных. Он имеет два режима работы:

1. **На время вызова**: Кэш хранит результаты функций до тех пор, пока выполняется запустившая их процедура (в рамках текущего стека вызова).
2. **На время сеанса**: Кэш сохраняет данные в оперативной памяти рабочего процесса для конкретного пользователя на все время его работы (или до принудительного сброса).

#### Влияние на снижение холостых обращений к СУБД

Представьте цикл обработки 1000 строк документа, где для каждой строки вызывается функция `ПолучитьСтавкуНДСНоменклатуры(Товар)`. Без кэширования платформа выполнит 1000 одинаковых или похожих SQL-запросов к СУБД. 

При включенном повторном использовании платформа перехватывает вызов:


```text
Вызов функции: ПолучитьСтавкуНДС(ТоварСсылка)

       |
       v
+------------------------------------+
| Проверка Кэша в оперативной памяти |
+------------------------------------+

       |
       +---> [Данные НАЙДЕНЫ] -------> Возврат значения из RAM (0 мс к СУБД)
       |
       +---> [Данные НЕ найдены] ----> Запрос к СУБД -> Запись в Кэш -> Возврат
```

* **Первый вызов**: Платформа выполняет реальный SQL-запрос к СУБД, получает значение и сохраняет пару `Ключ (Параметры функции) -> Значение (Результат)` в оперативной памяти (RAM).
* **Последующие вызовы**: Для всех строк с аналогичным товаром платформа мгновенно возвращает результат из RAM. Обращение к СУБД блокируется. Это ликвидирует избыточную нагрузку на дисковую подсистему и сеть.


# 4. Влияние на производительность и СУБД (Резюме для DBA)

Для администратора баз данных (DBA) общие модули — это ключевой объект контроля, поскольку именно в них сосредоточена основная масса T-SQL/PL-pgSQL запросов, генерируемых платформой 1С.

### Анализ логов (Технологический журнал 1С)

При расследовании проблем производительности (высокий CPU, длительные ожидания на замках, медленные транзакции) вызовы методов общих модулей сопоставляются с событиями СУБД через Технологический журнал (ТЖ) 1С. Основной интерес представляют два типа событий: `CALL` (вызов метода общего модуля) и `DBMSSQL` / `DBPOSTGRS` (непосредственное выполнение SQL-запроса).

* **Взаимосвязь в логе**: События выполнения SQL-запросов всегда вложены внутрь контекста выполнения метода общего модуля. Платформа 1С фиксирует полное свойство `Context` для каждого тяжелого SQL-запроса, что позволяет DBA мгновенно локализовать проблемную процедуру.
* **Пример сопоставления в ТЖ**:
  ```text
  05:12.123001-20,CALL,2,Context=ОбщийМодуль.ПроведениеСервер.Модуль.ОбработкаПроведения
  05:12.124005-15,DBPOSTGRS,3,Context=ОбщийМодуль.ПроведениеСервер.Модуль.ОбработкаПроведения,sql='SELECT ... FROM _Document150 ...'
  ```
  По логу ТЖ видно, что общая длительность метода составила 20 мс, из которых 15 мс занял конкретный запрос к СУБД, инициированный внутри модуля `ПроведениеСервер`.

### Нагрузка на СУБД через неоптимальный код

Общие модули являются главным местом концентрации архитектурных ошибок разработчиков 1С, которые напрямую транслируются в деградацию производительности СУБД. Самая критичная из них — **запрос в цикле**.

#### Механизм деградации при запросе в цикле
Когда разработчик помещает вызов универсальной функции общего модуля (содержащей SQL-запрос) внутрь цикла обхода строк (например, по 10 000 строкам ТЧ документа), для СУБД это оборачивается катастрофой:

1. **Лавинообразный рост контекстных переключений (Context Switches)**: Вместо одного эффективного пакетного запроса сервер приложений отправляет к СУБД 10 000 одиночных мелких запросов. СУБД тратит ресурсы CPU не на обработку данных, а на постоянное переключение контекста потоков, парсинг входящих SQL-текстов и валидацию прав.
2. **Переполнение пула соединений**: При высокой параллельной работе пользователей лавина однотипных запросов из общих модулей мгновенно утилизирует доступные соединения из пула (`Max Pool Size`), вызывая у других сеансов ожидания типа `db_pool_wait`.
3. **Неэффективное использование кэша планов СУБД**: Если запросы внутри модуля формируются динамически (без параметризации), кэш планов (Plan Cache / Shared Buffers) СУБД забивается тысячами уникальных «мусорных» планов, вытесняя действительно важные структуры.

#### Сравнительный анализ паттернов выполнения для DBA



| Метрика на уровне СУБД | Антипаттерн: Запрос в цикле общего модуля | Оптимально: Пакетная выборка в общем модуле |
| :--- | :--- | :--- |
| **Количество SQL-запросов** | Равно числу итераций цикла ($N$) | Строго **1 запрос** |
| **Утилизация CPU на СУБД** | Высокая (из-за оверхеда на сетевой Round-Trip и парсинг) | Минимальная (СУБД быстро отдает результат) |
| **Нагрузка на дисковую подсистему** | Высокая (множественные мелкие чтения/Index Seek) | Оптимальная (последовательное чтение по индексу) |
| **Пропускная способность сети** | Критическое падение из-за тысяч мелких пакетов | Максимальная (данные передаются одним пулом) |
