# Параметры сеанса в 1С

# 1. Введение: Что такое параметры сеанса в 1С

**Параметры сеанса** — это специализированные объекты метаданных общего назначения, представляющие собой именованные глобальные переменные, которые жестко привязаны к конкретному сеансу работы пользователя. Они инициализируются при старте приложения, существуют в оперативной памяти на протяжении всего времени активности сеанса и уничтожаются автоматически при его завершении. 

Главная особенность параметров сеанса заключается в том, что их значения уникальны для каждого отдельного подключения и доступны из любой точки конфигурации (как на сервере, так и на клиенте через серверный вызов).

### Главные функции параметров сеанса

* **Хранение контекста текущего пользователя**
  Параметры сеанса выступают централизованным хранилищем ключевых реквизитов текущего сеанса. В них записываются ссылки на авторизовавшегося сотрудника (`ТекущийПользователь`), его подразделение (`ТекущееПодразделение`) или выбранное по умолчанию юрлицо (`ТекущаяОрганизация`). Это избавляет разработчика от необходимости постоянно передавать эти данные между процедурами.

* **Кэширование персональных настроек и данных**
  Использование параметров сеанса позволяет один раз при запуске программы считать из базы данных тяжелые настройки пользователя (например, доступные склады, права на редактирование цен или константы) и сохранить их в памяти. Это полностью исключает повторные холостые запросы к СУБД при открытии каждой новой формы или проведении документов.

* **Критическая роль в механизме RLS (Row-Level Security)**
  Параметры сеанса являются фундаментом для работы ограничений прав доступа на уровне записей. Они выступают в качестве «входных переменных» в шаблонах ограничений СУБД. На их основе платформа 1С динамически модифицирует SQL-запросы, подставляя условия вида `ГДЕ Организация = &ПараметрСеанса.ТекущаяОрганизация`.

---

### Архитектурные ограничения и особенности



| Характеристика | Описание поведения платформы |
| :--- | :--- |
| **Область видимости** | Ограничена строго текущим сеансом (сеанс №1 не имеет доступа к параметрам сеанса №2). |
| **Доступные типы данных** | Могут содержать как примитивные типы (числа, строки, булево), так и ссылки на объекты БД. |
| **Инициализация** | Выполняется перехват в модуле сеанса (`УстановкаПараметровСеанса`) при первом обращении. |
| **Попытка чтения до инициализации** | Вызывает критическую ошибку платформы: *«Параметр сеанса не установлен»*. |

# 2. Параметры сеанса на уровне СУБД: Где физически лежат данные

### Прямой ответ: Существуют ли постоянные SQL-таблицы для параметров сеанса?
**Нет.** В структуре базы данных СУБД не создается отдельных постоянных таблиц или персональных колонок под каждый объявленный в конфигураторе параметр сеанса. Вы не найдете физических таблиц с именами вроде `_SessionParameters` или колонок `_Param_ТекущийПользователь`. Платформа абстрагирует хранение этих объектов от стандартной реляционной схемы.

### Где они живут во время работы

В режиме активной эксплуатации прикладного решения значения всех инициализированных параметров сеанса **хранятся в оперативной памяти (RAM) сервера приложений 1С**.

* **Контейнер хранения**: Данные изолированно удерживаются внутри адресного пространства конкретного рабочего процесса **`rphost`**, который обслуживает текущую сессию пользователя.
* **Скорость доступа**: Благодаря хранению в RAM, чтение параметров сеанса на сервере происходит мгновенно (0 мс на обращение к СУБД). Это критически важно, так как параметры непрерывно считываются платформой при выполнении любых SQL-запросов, задействующих механизм RLS.

### Механизм сброса на диск (След в СУБД)

Несмотря на энергозависимый характер хранения, параметры сеанса могут оставлять след в физической базе данных. Это происходит при активации встроенных механизмов оптимизации кластера 1С, таких как **пассивизация («засыпание») сеансов**, или при обеспечении отказоустойчивости.

Если пользователь долгое время не проявляет активности, кластер 1С высвобождает оперативную память `rphost`. Платформа сериализует весь контекст сеанса (включая параметры сеанса) в бинарный поток (BLOB) и временно сохраняет его на диск.

#### Куда сбрасываются данные в зависимости от архитектуры:

```text
+-------------------------------------------------------------------------+

|                  Сеанс пользователя (В состоянии покоя)                 |
|  Кластер 1С инициирует пассивизацию для освобождения RAM процесса rphost|
+-------------------------------------------------------------------------+

                                     |
                                     v
                  +--------------------------------------+
                  | Механизм сериализации контекста 1С  |
                  +--------------------------------------+

                                     |
                 +-------------------+-------------------+
                 |                                       |
                 v (Вариант 1)                           v (Вариант 2)
+----------------------------------+   +----------------------------------+

| Таблица базы данных [V8USERS]    |   | Централизованная БД сессий       |
| (Служебное двоичное поле Data)   |   | (В СУБД или на сессионном сервере|
| Используется в старых версиях    |   | Кластера 1С:Предприятия)         |
+----------------------------------+   +----------------------------------+
```

1. **В системную таблицу `V8USERS`**: В ряде архитектурных сценариев и версий платформы сериализованные данные пользовательского контекста сохраняются непосредственно в системной таблице базы данных `V8USERS` (в бинарном поле).
2. **Во временное хранилище базы сессий**: В современных отказоустойчивых кластерах 1С параметры сеанса при пассивизации или миграции сеанса между рабочими процессами `rphost` сбрасываются в служебную **базу данных сессий кластера** (которая может быть развернута как отдельная техническая база на SQL-сервере).

При возобновлении активности пользователя (например, при нажатии кнопки в интерфейсе) платформа выполняет обратное чтение из СУБД, десериализует BLOB-структуру и мгновенно возвращает параметры сеанса в RAM процесса `rphost`.

# 3. Назначение и практические примеры использования

В реальной разработке параметры сеанса делятся на две основные категории: **скалярные** (для хранения одиночных значений контекста) и **коллекции** (для хранения массивов разрешенных объектов). Их правильная инициализация позволяет свести к нулю лишние запросы к СУБД.

### Авторизация и контекст (Скалярные параметры)

Самый частый сценарий использования параметров сеанса — фиксация данных об авторизованном пользователе и его базовых рабочих настройках.

* **Параметр `ТекущийПользователь`** (Тип: `СправочникСсылка.Пользователи`)
  * **Как это работает**: При старте сеанса в модуле сеанса один раз срабатывает событие `УстановкаПараметровСеанса`. Платформа определяет Windows/OpenID-логин или имя пользователя 1С, находит соответствующую ссылку в справочнике `Пользователи` и записывает её в параметр сеанса.
  * **Профит для системы**: В процессе работы пользователь может проводить сотни документов в минуту. В каждом документе в типовых конфигурациях заполняется реквизит `Ответственный`. Вместо того чтобы при каждой записи документа делать тяжелый SQL-запрос `SELECT` к таблице справочника пользователей для определения текущего сотрудника, платформа мгновенно считывает готовую ссылку из оперативной памяти (`ПараметрыСеанса.ТекущийПользователь`).

### Списки разрешенных объектов (Параметры-коллекции)

Параметры сеанса могут хранить не только одиночные значения, но и массивы данных. Это активно используется для ограничения видимости данных в разрезе организационной структуры или географии.

* **Параметр `РазрешенныеПользователюОрганизации`** (Тип: `ФиксированныйМассив` из ссылок `СправочникСсылка.Организации`)
  * **Как это работает**: При входе пользователя в систему программный код считывает его права и роли, определяет, к каким филиалам или юридическим лицам он имеет доступ, и формирует неизменяемый `ФиксированныйМассив` ссылок.
  * **Применение в RLS**: Этот массив напрямую передается в подсистему ограничения доступа. В результате при формировании списков документов (например, в журнале "Документы закупа") СУБД выполняет быструю фильтрацию по уже готовому списку идентификаторов организаций, находящемуся в RAM сервера 1С, не заставляя SQL-сервер каждый раз динамически вычислять дерево доступных пользователю подразделений через JOIN.

---

### Различия в использовании типов данных



| Имя параметра сеанса | Тип данных платформы | Оптимизационный эффект |
| :--- | :--- | :--- |
| **`ТекущийПользователь`** | `СправочникСсылка.Пользователи` | Устраняет повторные обращения к СУБД при заполнении реквизитов "Автор"/"Ответственный". |
| **`ОсновнаяОрганизация`** | `СправочникСсылка.Организации` | Ускоряет подстановку значений по умолчанию в новых формах документов. |
| **`РазрешенныеОрганизации`**| `ФиксированныйМассив` | Передает готовый массив ID в SQL-сервер для мгновенной фильтрации по RLS. |

# 4. Влияние на производительность и СУБД: Архитектура RLS (Критично для DBA)

Для системного администратора баз данных параметры сеанса в контексте RLS (Row-Level Security) являются главным управляющим элементом, который определяет, насколько сложным, тяжелым и оптимальным будет итоговый T-SQL или PL/pgSQL запрос.

### Как параметры сеанса управляют планами SQL-запросов

Когда пользователь выполняет запрос к таблице, на которую наложены ограничения доступа (например, к журналу документов продаж), платформа 1С осуществляет динамическую трансформацию исходного текста запроса. В этот момент значения из параметров сеанса передаются напрямую на уровень СУБД.

* **Параметризация на уровне SQL**: Платформа преобразует ссылки или массивы ссылок из параметров сеанса в стандартные типизированные SQL-параметры (например, `@P1`, `@P2`, `@P3`).
* **Внедрение в план выполнения**: Если в параметре сеанса хранится `ФиксированныйМассив` разрешенных организаций, платформа транслирует его в SQL-конструкцию `WHERE _Fld123 IN (@P1, @P2)`. 
* **Преимущество для оптимизатора СУБД**: Благодаря передаче готовых значений в качестве параметров, оптимизатор SQL-сервера (Query Optimizer) строит стабильный и предсказуемый план выполнения (Execution Plan). СУБД может эффективно использовать операторы `Index Seek` вместо полного сканирования таблиц (`Index Scan` / `Table Scan`), так как ей сразу известны точные границы выборки.

### Проблема неинициализированных параметров

Попытка неявного обращения платформы к параметру сеанса, который еще не был инициализирован (в нем нет данных), способна спровоцировать каскадный сбой производительности и вызвать деградацию транзакционной системы.

```text
Внутри транзакции 1С инициируется SQL-запрос с RLS

                          |
                          v
Платформа обнаруживает: ПараметрСеанса НЕ ИНИЦИАЛИЗИРОВАН!
                          |
                          v
[Аварийное прерывание потока] -> Вызов процедуры модуля сеанса УстановкаПараметровСеанса()

                          |
                          v
Выполнение тяжелого кода внутри транзакции (Запросы к СУБД, чтение настроек)
                          |
                          v
Результат для СУБД: Рост времени транзакции -> Зависание блокировок (Lock Waits) -> Ошибки по таймауту (Timeout)
```

1. **Неявный вызов обработчика**: Если во время выполнения бизнес-логики (особенно внутри транзакции проведения документа) происходит обращение к пустому параметру сеанса, платформа аварийно приостанавливает текущую операцию и запускает обработчик `УстановкаПараметровСеанса()` в модуле сеанса.
2. **Опасность для транзакций**: Код инициализации параметров часто содержит тяжелые запросы для вычисления прав пользователя. Выполнение этих запросов **прямо посреди открытой транзакции** лавинообразно увеличивает время её удержания.
3. **Последствия для DBA**: В СУБД резко возрастает время монопольных блокировок таблиц. Другие пользователи, пытающиеся записать данные в эти же таблицы, встают в очередь ожиданий (`LCK_M_X` в MS SQL или `ExclusiveLock` в PostgreSQL), что приводит к массовым взаимоблокировкам (Deadlocks) и ошибкам времени ожидания (Lock Timeout).

### Объем памяти (RAM) на сервере приложений vs DISK I/O СУБД

Архитектурный выбор размера данных, помещаемых в параметры сеанса — это всегда классический компромисс (Trade-off) между утилизацией оперативной памяти сервера приложений 1С и нагрузкой на дисковую подсистему сервера баз данных.

* **Раздувание RAM процессов `rphost`**: Если в конфигурации настроена сложная иерархическая структура прав, и для каждого пользователя в параметр сеанса записываются огромные массивы (например, списки из 10 000 разрешенных номенклатурных групп или контрагентов), это приводит к существенному росту потребления оперативной памяти рабочими процессами `rphost`. При высокой плотности сессий (сотни пользователей) это может вызвать дефицит RAM на сервере приложений.
* **Снижение нагрузки на DISK I/O СУБД**: С другой стороны, хранение этих тысяч идентификаторов в оперативной памяти полностью избавляет СУБД от необходимости при каждом RLS-запросе динамически вычислять права пользователя через сложные соединения (`JOIN`) с таблицами прав. SQL-сервер получает готовый плоский массив значений, выполняет моментальную фильтрацию в памяти, сводя к минимуму дисковые чтения и снижая нагрузку на CPU и очередь диска (Disk Queue Length).
