
# Архитектура прикладных решений на платформе 1С:Предприятие 8.3

Проектирование любой современной конфигурации 1С базируется на принципах модульности, масштабируемости и удобства конечного пользователя. Одним из фундаментальных объектов метаданных, обеспечивающих реализацию этих принципов в среде разработки, является подсистема. Начиная с управляемого интерфейса (платформа 8.2 и 8.3), роль подсистем кардинально трансформировалась: из простого инструмента группировки в Конфигураторе они превратились в основу визуального представления бизнес-логики программы.

---

# 1. Введение: Что такое подсистема в 1С

**Подсистема** — это подчиненный объект метаданных общего назначения, предназначенный для логического разделения и структурирования прикладного решения. Она объединяет различные объекты конфигурации (справочники, документы, регистры, обработки) по функциональному признаку.

### Главные функции подсистем

* **Формирование командного интерфейса**
  Подсистемы первого (верхнего) уровня напрямую определяют состав **Панели разделов** основного окна приложения в режиме «1С:Предприятие». Дочерние (подчиненные) подсистемы формируют внутреннюю структуру этих разделов: группы навигации, меню базы данных и распределение стандартных команд. Если в конфигурации не включено свойство видимости ни у одной подсистемы, панель разделов пользователю вообще не отобразится.

* **Визуальная фильтрация объектов для разработчика**
  В режиме «Конфигуратор» при разрастании дерева метаданных становится сложно ориентироваться в тысячах объектов. Разработчик может включить штатный фильтр по конкретной подсистеме. После этого в окне конфигурации будут отображаться только те сущности, которые привязаны к выбранному функциональному блоку (например, ERP-модуль «Производство»).

* **Разграничение прав и динамическая видимость**
  Платформа использует подсистемы для гибкой настройки интерфейса под конкретные роли пользователей. При ограничении прав доступа на уровне ролей или программной настройке командного интерфейса платформа автоматически скрывает целые разделы и входящие в них элементы, к которым у текущего сотрудника нет доступа.

* **Управление обновлением, поставкой и поддержкой**
  При создании файлов поставки (`.cf`) или модульных обновлений (`.cfu`) разработчик может отбирать объекты на основании их принадлежности к подсистемам. Это позволяет выставлять индивидуальные правила поддержки (так называемые «замки поставщика» и правила поддержки для измененных объектов) для отдельных подсистем, изолируя кастомный функционал от типового.

# 2. Подсистема на уровне СУБД: Архитектурный секрет

### Прямой ответ: Существует ли физическая таблица подсистем в SQL?
**Нет.** В физической структуре реляционной базы данных (MS SQL Server, PostgreSQL, Oracle или IBM DB2) подсистемы не имеют собственных выделенных таблиц. Вы не найдете таблиц с именами вида `_Subsystem` или `_ReferenceSubsystem`, в которых хранились бы строки с перечнем разделов. 

С точки зрения СУБД, подсистема — это абстракция. Она не оперирует физическими данными, не содержит записей (строк) и не влияет на транзакционную нагрузку (LOCK, INSERT, UPDATE) при работе пользователей с документами или справочниками.

### Где и как хранятся подсистемы

Подсистема является логическим описанием (конфигурацией) прикладного решения. Вся информация о подсистемах — это **мета-информация самой платформы 1С**. Она хранится исключительно в системных таблицах СУБД, которые платформа создает для управления структурой базы данных.

#### Механизм сериализации
Состав подсистем, их иерархическая структура, иконки, настройки видимости и списки входящих объектов сериализуются платформой 1С в XML/JSON-подобные структуры, сжимаются (Deflate) и записываются в виде бинарных данных (BLOB) в следующие таблицы:


| Системная таблица в СУБД | Роль в хранении подсистем | Описание содержимого |
| :--- | :--- | :--- |
| **`Config`** | Основная рабочая конфигурация | Хранит текущую конфигурацию базы данных (режим «Предприятие»). Подсистемы находятся внутри общего потока метаданных в упакованном бинарном виде. |
| **`ConfigSave`** | Конфигурация разработчика | Хранит изменения, которые разработчик внес в Конфигураторе, но еще не обновил базу данных (не нажал клавишу F7). |
| **`DBSchema`** | Структура базы данных | Внутренняя схема метаданных, с которой сверяется платформа при запуске и обновлении. Хранит информацию о соответствии объектов 1С физическим таблицам СУБД. |

#### Схема архитектурного разделения

```text
+-----------------------------------------------------------------------+

|                       Режим «1С:Предприятие 8.3»                      |
|  [Интерфейс пользователя] <--- (Динамическое построение меню)        |
+-----------------------------------------------------------------------+

                                   |
                                   v
+-----------------------------------------------------------------------+
|                      Платформа «1С:Предприятие»                       |
|  Парсит бинарные данные, строит дерево метаданных, кэширует структуру |
+-----------------------------------------------------------------------+

                                   |
                                   v
+-----------------------------------------------------------------------+
|                             Уровень СУБД                              |
|  +-----------------------------------------------------------------+  |
|  | Таблица [Config] / [DBSchema]                                   |  |
|  | -> Поле [BinaryData] (Сжатый BLOB: Иерархия, Состав, Картинки)   |  |
|  +-----------------------------------------------------------------+  |
|  | Физические таблицы бизнес-данных (Например: _Reference32)       |  |
|  | (Ничего не знают о подсистемах)                                 |  |
|  +-----------------------------------------------------------------+  |
+-----------------------------------------------------------------------+
```

### Как это влияет на производительность и обслуживание

1. **Быстрый старт приложения:** При запуске сеанса «1С:Предприятие» платформа считывает метаданные из таблицы `Config` один раз (и кэширует их на сервере приложений), чтобы мгновенно построить интерфейс для пользователя в соответствии с его правами.
2. **Нулевой оверхед на SQL:** Поскольку связи объектов с подсистемами не проверяются на уровне SQL-запросов (через JOIN), деление конфигурации на 5 или на 500 подсистем никак не снижает скорость выполнения запросов к СУБД.

# 3. Назначение и практические примеры использования

В реальной разработке подсистемы классифицируются по функциональному назначению. Их делят на две ключевые категории: прикладные (бизнес-архитектурные) и служебные (технические).

### Бизнес-архитектура (Прикладные подсистемы)

Прикладные подсистемы служат для зеркального отображения реальной структуры бизнес-процессов предприятия. Они строятся на верхнем уровне метаданных и разделяют конфигурацию (особенно класса ERP) на крупные функциональные блоки. 

* **«Продажи» (`_Продажи`)**: Объединяет документы `ЗаказПокупателя`, `РеализацияТоваровУслуг`, справочники `Клиенты`, `СоглашенияСКлиентами` и регистры накопления по взаиморасчетам.
* **«Закупки» (`_Закупки`)**: Группирует сущности для работы с поставщиками — документы `ЗаказПоставщику`, `ПоступлениеТоваровУслуг`, справочник `НоменклатураПоставщиков`.
* **«Склад» / «Запасы» (`_Склад`)**: Отвечает за внутреннее товародвижение. Включает справочник `Склады`, документы `ПеремещениеТоваров`, `ИнвентаризацияТоваров` и оборотные регистры остатков на складах.

### Служебные подсистемы (на примере БСП)

Служебные подсистемы не всегда видны конечному пользователю в интерфейсе. Их задача — обеспечивать работу сквозного, сервисного функционала, который переносится между различными конфигурациями. Ярким примером является **Библиотека Стандартных Подсистем (БСП)**.

* **`КонтактнаяИнформация`**: Предоставляет стандартизированные механизмы для хранения и валидации адресов, телефонов и email-адресов для любых объектов конфигурации (контрагенты, физические лица, пользователи).
* **`ВариантыОтчетов`**: Отвечает за интеграцию системы компоновки данных (СКД), сохранение пользовательских настроек отчетов, их рассылку по расписанию и вывод в панели разделов.
* **`ОбновлениеВерсииИБ`**: Автоматически выполняет служебные обработчики обновления данных при переходе на новые релизы конфигурации.

### Управление доступом и интеграция с ролями

Подсистемы выступают в качестве естественных границ для настройки прав доступа. Это реализуется на двух уровнях:

1. **Права на саму подсистему:** В роли (объекте прав доступа) флаг `Просмотр` для конкретной подсистемы определяет, увидит ли пользователь данный раздел в своей панели разделов или нет. Если у роли забрать право просмотра подсистемы «Закупки», весь раздел полностью исчезнет из интерфейса пользователя, даже если у него есть права на отдельные документы внутри него.
2. **Логическая связь ролей:** При разработке модулей принято создавать роли, жестко привязанные к границам подсистем.

   
| Подсистема | Профильная роль в конфигурации | Права доступа в рамках границы |
| :--- | :--- | :--- |
| **Продажи** | `МенеджерПоПродажам` | Чтение и запись документов продаж, просмотр отчетов по выручке. |
| **Закупки** | `МенеджерПоЗакупкам` | Чтение и запись документов снабжения, формирование заказов поставщикам. |
| **Служебная (БСП)** | `БазовыеПраваБСП` | Доступ к чтению общих классификаторов (валюты, банки, адресный классификатор). |
