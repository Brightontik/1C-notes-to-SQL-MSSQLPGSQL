# Реализация паттернов Outbox и Inbox в 1С (Физический слой СУБД)

**Область применения:** Архитектура гарантированной доставки сообщений, Event-Driven системы, интеграционный Highload.
**Цель правила:** Обеспечить максимальную пропускную способность (RPS) при регистрации и чтении сообщений, исключить взаимные блокировки (Deadlocks) с основными документами и предотвратить раздувание индексов СУБД.

---

## 1. Архитектура очередей на стороне 1С (Физический слой)

Выбор объекта метаданных 1С, который на уровне СУБД будет выступать в роли таблицы-очереди Outbox (для отправки) или Inbox (для приема), критически влияет на параллельность работы всей системы.

### 1.1. Сравнительный анализ объектов метаданных


| Критерий на уровне СУБД | Регистр сведений (Управляемый) | Планы обмена | Бизнес-процессы и Задачи |
| :--- | :--- | :--- | :--- |
| **Структура таблиц в SQL** | Плоская, компактная таблица (`_InfoRegXX`). Минимум избыточных системных полей. | Сложная структура: базовая таблица + скрытые таблицы регистрации изменений (`_ChangeRecXX`). | Гигантская структура: таблицы заголовков, табличные части, системные индексы точек маршрута. |
| **Избыточные индексы** | Индексы строятся строго по измерениям. Легко контролировать и тюнить. | Платформа автоматически строит тяжелые индексы по номерам сообщений и ссылкам объектов. | Огромное количество платформенных индексов (по состоянию, по исполнителю, по времени). |
| **Поведение при записи (DML)** | Быстрый `INSERT` / `UPDATE`. Минимальное количество блокируемых страниц СУБД. | Каждая запись блокирует системные таблицы регистрации. Высокий риск коллизий. | Тяжелый `INSERT` с каскадным обновлением системных индексов и таблиц задач. |
| **Пригодность для Highload** | **Идеально (Эталон)** | **Ограниченно** (Только для монолитных миграций) | **Категорически запрещено** |

### 1.2. Физика процессов на уровне СУБД

#### 1. Регистр сведений (Управляемый режим) — Выбор Архитектора
* **Почему это эталон:** Платформа транслирует управляемый регистр сведений в плоскую SQL-таблицу. Если сделать регистр независимым и непериодическим, мы получаем аналог классической key-value таблицы. 
* **Преимущества для Outbox/Inbox:** Позволяет выполнять мгновенную запись в одной ACID-транзакции с основным документом. Индексы по измерениям (например, `ИдентификаторСообщения` или `КлючИдемпотентности`) обеспечивают скорость поиска за $O(1)$ (Index Seek) без сканирования всей таблицы.

#### 2. Планы обмена — Встроенный инструмент платформы
* **Почему не подходит для Event-Driven архитектуры:** Планы обмена проектировались фирмой «1С» в начале 2000-х годов под концепцию пакетных монолитных миграций (раз в сутки / раз в час). 
* **В чём проблема под нагрузкой:** Механизм регистрации изменений (`ПланыОбмена.ЗарегистрироватьИзменения()`) работает через скрытые таблицы СУБД вида `_ChangeRecXX`. При высокой частоте событий (сотни документов в минуту) постоянная запись в эти таблицы приводит к жестким взаимным блокировкам (Deadlocks) и начинает тормозить проведение самих документов-источников. Кроме того, планы обмена не умеют нативно хранить метаданные события (Routing Key, Headers, payload JSON) без костылей в виде привязки к дополнительным объектам.

#### 3. Бизнес-процессы и Задачи — Архитектурный провал
* **Почему это непригодно для Highload очередей:** Некоторые разработчики пытаются использовать объект «Задача» как элемент очереди (ведь там есть готовые статусы и исполнители). 
* **В чём катастрофа для СУБД:** Объект «Задача» — это тяжелый ссылочный объект конфигурации. На одну запись задачи СУБД выполняет колоссальное количество дисковых и процессорных операций: генерируется GUID, пишутся системные таблицы бизнес-процессов, обновляются многочисленные индексы по точкам маршрута и адресации. При нагрузке в 50-100 RPS база данных моментально уйдет в 100% утилизацию CPU и дисков (I/O Bottleneck), остановив работу пользователей.

## 1.3. Проектирование структуры регистра-очереди (Параметры Highload)

Для реализации Outbox/Inbox необходимо создавать **Независимый Непериодический** регистр сведений. Распределение полей по категориям (Измерения/Ресурсы/Реквизиты) должно строго соответствовать правилам индексации СУБД, чтобы индексы не раздувались, а поиск работал за $O(logN)$.

### Состав полей и типы данных:

* **Измерения (Входят в кластерный индекс таблицы СУБД):**
  * `ИмяИнтеграции` (Строка, длина до 50-100 знаков, индексировать). Позволяет делить очередь по направлениям обмена в одном регистре.
  * `ИдентификаторСообщения` (Тип: `УникальныйИдентификатор`). GUID события, обеспечивающий 100% уникальность строки и выступающий ключом дедупликации.
* **Ресурсы (Данные, которые часто меняются воркером):**
  * `СтатусОбработки` (Тип: `ПеречислениеСсылка.СтатусыОчередиИнтеграции`). Содержит значения: `Новый`, `ВРаботе`, `Ошибка`, `Обработано`.
  * `КоличествоПопыток` (Тип: `Число`, длина 2, точность 0). Счетчик ретраев для отсечки «токсичных» пакетов.
  * `ДатаСледующегоСтарта` (Тип: `Дата`). Временная метка для алгоритма Экспоненциального повтора (Exponential Backoff). Воркер игнорирует запись, если текущее время меньше этой даты.
  * `ТекстОшибки` (Тип: `Строка`, неограниченная длина). Сюда сбрасывается стек ошибки при аварии для Parking Lot.
* **Реквизиты (Тяжелые данные, не участвующие в поиске):**
  * `ТелоСообщения` (Тип: `Строка`, неограниченная длина). Хранит сырой Payload (JSON/XML).

---

### ⚠️ Критичные low-level правила для Архитектора

#### 1. Почему ТелоСообщения должно быть строго в Реквизитах?
В платформе 1С поля категории «Реквизит» регистра сведений физически **никогда не включаются в индексы таблицы СУБД**. Если по ошибке сделать `ТелоСообщения` Ресурсом, СУБД будет тратить колоссальные ресурсы на пересчет и обслуживание индексов при каждой вставке, что приведет к катастрофической деградации производительности (дисковый I/O Bottleneck).

#### 2. Строка(неограниченная) vs ХранилищеЗначений (Специфика СУБД)
Платформа 1С позволяет хранить бинарные данные и длинные тексты либо в неограниченной строке, либо в объекте `ХранилищеЗначений` (BLOB).

* **Для PostgreSQL (Рекомендуемый метод):**
  Строку неограниченной длины Postgres транслирует в физический тип данных **`text`**. В PostgreSQL тип `text` оптимизирован идеально: если строка короткая, она хранится прямо внутри страницы данных таблицы. Если строка становится длинной, Postgres автоматически и незаметно для 1С уносит её в специальную структуру **TOAST** (The Oversized-Attribute Storage Technique), сжимая её на лету. Поэтому для баз на PostgreSQL в качестве `ТелаСообщения` **всегда выбирается Строка (неограниченная)**, содержащая JSON.
* **Для MS SQL Server:**
  Платформа транслирует длинную строку в тип `NVARCHAR(MAX)`. Работа с ним эффективна, но если данные интеграции содержат бинарные файлы или сжатые ZIP-потоки, то оправдано использование `ХранилищаЗначений` (тип `VARBINARY(MAX)` в SQL), чтобы избежать оверхеда на текстовую конвертацию Base64.

## 1.4. Стратегии хранения Payload (Тела сообщения): Матрица выбора Архитектора

Размер передаваемых данных (вес сообщения) — главный фактор, определяющий архитектуру хранения очереди, выделяются 4 основные стратегии.

### Стратегия 1: Прямое хранение в Реквизите РС (Строка неограниченной длины)
* **Когда использовать:** Вес одного сообщения **заведомо мал (до 50-100 КБ)**. Это 90% стандартных EDA-событий (создание контрагента, изменение статуса заказа, быстрая накладная).
* **Физика СУБД:** В PostgreSQL данные ложатся в компактный тип `text` и при превышении лимитов уходят в TOAST-таблицы. В MS SQL — в `NVARCHAR(MAX)`.
* **Плюсы:** Максимальная скорость работы. Чтение и запись происходят в один такт. Нет накладных расходов на создание сопутствующих ссылочных объектов.

### Стратегия 2: Сжатие на платформе (ХранилищеЗначений в Реквизите РС)
* **Когда использовать:** Сообщение имеет средний размер **(от 100 КБ до 10-20 МБ)** и представляет собой чистый текст (большой JSON-реестр остатков, XML-выгрузка прайсов), при этом выносить данные за пределы базы нельзя по соображениям безопасности или инфраструктурным ограничениям.
* **Физика СУБД:** Платформа транслирует `ХранилищеЗначений` в бинарный тип `VARBINARY(MAX)` (MS SQL) или `BYTEA` (PostgreSQL), принудительно сжимая текстовые данные встроенным алгоритмом Deflate на стороне `rphost` перед отправкой в СУБД.
* **Плюсы:** Жесткая экономия сетевого трафика между `rphost` и СУБД, а также дискового пространства базы данных (текст сжимается в 3-5 раз).
* **Минусы:** Высокая нагрузка на CPU серверов приложений 1С из-за постоянных операций сжатия/декомпрессии при записи и чтении каждой строки воркером.

### Стратегия 3: Вынос в служебный Справочник (Гибридный метод)
* **Когда использовать:** Сообщения **тяжелые (от 20 до 100+ МБ)**, но бизнес-требования жестко обязывают хранить их **строго внутри единого транзакционного контура СУБД** (например, для упрощения резервного копирования базы или обеспечения 100% ACID-гарантий).
* **Физика процесса:** В регистр сведений (очередь) вставляется только легкая `Ссылка` на элемент специально созданного служебного справочника (например, `втОчередьИнтеграцииПрисоединенныеФайлы`). Само тяжелое тело сообщения пишется в реквизит этого справочника.
* **Почему нельзя писать напрямую в РС:** Постоянное обновление («апдейт») строк в РС, где лежат мегабайтные бинарники, приводит к дикой фрагментации индексов и раздуванию логов транзакций СУБД при смене статусов (`Новый` $\rightarrow$ `ВРаботе` $\rightarrow$ `Обработано`). Вынос в Справочник изолирует тяжелые данные: мы один раз делаем `INSERT` тяжелого элемента справочника, а в РС-очереди быстро и дешево меняем только статусы у легкой строки со ссылкой. После обработки элемент справочника удаляется.

### Стратегия 4: Вынос во Внешнее Хранилище (Файлы / Shared NAS / S3) — Эталон Highload
* **Когда использовать:** Пакеты данных **экстремально тяжелые (от 100 МБ до нескольких гигабайт)**. Типичный пример ритейла: ночная выгрузка терабайтных исторических данных, медиа-контент, тяжелые архивы.
* **Физика процесса:** Тело сообщения вообще не пересекает границу базы данных 1С и СУБД. `ЗаписьJSON/XML` через файловый поток пишет данные напрямую в общую сетевую папку (NFS/SMB) или выталкивает в S3-совместимое объектное хранилище (MinIO / Yandex Object Storage). В регистр-очередь РС записывается только плоская текстовая строка — **путь к файлу или URL-ссылка на S3-бакет**.
* **Плюсы:** База данных 1С не раздувается и не деградирует. Исключены риски падения `rphost` по Out of Memory. Внешняя система-приемник скачивает тяжелый файл напрямую из S3 по ссылке, полностью снимая с 1С нагрузку по веб-отдаче тяжелого контента.
