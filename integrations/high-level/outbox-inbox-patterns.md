# Реализация паттернов Outbox и Inbox в 1С (Физический слой СУБД)

**Область применения:** Архитектура гарантированной доставки сообщений, Event-Driven системы, интеграционный Highload.
**Цель правила:** Обеспечить максимальную пропускную способность (RPS) при регистрации и чтении сообщений, исключить взаимные блокировки (Deadlocks) с основными документами и предотвратить раздувание индексов СУБД.

---

## 1. Архитектура очередей на стороне 1С (Физический слой)

Выбор объекта метаданных 1С, который на уровне СУБД будет выступать в роли таблицы-очереди Outbox (для отправки) или Inbox (для приема), критически влияет на параллельность работы всей системы.

### 1.1. Сравнительный анализ объектов метаданных


| Критерий на уровне СУБД | Регистр сведений (Управляемый) | Планы обмена | Бизнес-процессы и Задачи |
| :--- | :--- | :--- | :--- |
| **Структура таблиц в SQL** | Плоская, компактная таблица (`_InfoRegXX`). Минимум избыточных системных полей. | Сложная структура: базовая таблица + скрытые таблицы регистрации изменений (`_ChangeRecXX`). | Гигантская структура: таблицы заголовков, табличные части, системные индексы точек маршрута. |
| **Избыточные индексы** | Индексы строятся строго по измерениям. Легко контролировать и тюнить. | Платформа автоматически строит тяжелые индексы по номерам сообщений и ссылкам объектов. | Огромное количество платформенных индексов (по состоянию, по исполнителю, по времени). |
| **Поведение при записи (DML)** | Быстрый `INSERT` / `UPDATE`. Минимальное количество блокируемых страниц СУБД. | Каждая запись блокирует системные таблицы регистрации. Высокий риск коллизий. | Тяжелый `INSERT` с каскадным обновлением системных индексов и таблиц задач. |
| **Пригодность для Highload** | **Идеально (Эталон)** | **Ограниченно** (Только для монолитных миграций) | **Категорически запрещено** |

### 1.2. Физика процессов на уровне СУБД

#### 1. Регистр сведений (Управляемый режим) — Выбор Архитектора
* **Почему это эталон:** Платформа транслирует управляемый регистр сведений в плоскую SQL-таблицу. Если сделать регистр независимым и непериодическим, мы получаем аналог классической key-value таблицы. 
* **Преимущества для Outbox/Inbox:** Позволяет выполнять мгновенную запись в одной ACID-транзакции с основным документом. Индексы по измерениям (например, `ИдентификаторСообщения` или `КлючИдемпотентности`) обеспечивают скорость поиска за $O(1)$ (Index Seek) без сканирования всей таблицы.

#### 2. Планы обмена — Встроенный инструмент платформы
* **Почему не подходит для Event-Driven архитектуры:** Планы обмена проектировались фирмой «1С» в начале 2000-х годов под концепцию пакетных монолитных миграций (раз в сутки / раз в час). 
* **В чём проблема под нагрузкой:** Механизм регистрации изменений (`ПланыОбмена.ЗарегистрироватьИзменения()`) работает через скрытые таблицы СУБД вида `_ChangeRecXX`. При высокой частоте событий (сотни документов в минуту) постоянная запись в эти таблицы приводит к жестким взаимным блокировкам (Deadlocks) и начинает тормозить проведение самих документов-источников. Кроме того, планы обмена не умеют нативно хранить метаданные события (Routing Key, Headers, payload JSON) без костылей в виде привязки к дополнительным объектам.

#### 3. Бизнес-процессы и Задачи — Архитектурный провал
* **Почему это непригодно для Highload очередей:** Некоторые разработчики пытаются использовать объект «Задача» как элемент очереди (ведь там есть готовые статусы и исполнители). 
* **В чём катастрофа для СУБД:** Объект «Задача» — это тяжелый ссылочный объект конфигурации. На одну запись задачи СУБД выполняет колоссальное количество дисковых и процессорных операций: генерируется GUID, пишутся системные таблицы бизнес-процессов, обновляются многочисленные индексы по точкам маршрута и адресации. При нагрузке в 50-100 RPS база данных моментально уйдет в 100% утилизацию CPU и дисков (I/O Bottleneck), остановив работу пользователей.

## 1.3. Проектирование структуры регистра-очереди (Параметры Highload)

Для реализации Outbox/Inbox необходимо создавать **Независимый Непериодический** регистр сведений. Распределение полей по категориям (Измерения/Ресурсы/Реквизиты) должно строго соответствовать правилам индексации СУБД, чтобы индексы не раздувались, а поиск работал за $O(logN)$.

### Состав полей и типы данных:

* **Измерения (Входят в кластерный индекс таблицы СУБД):**
  * `ИмяИнтеграции` (Строка, длина до 50-100 знаков, индексировать). Позволяет делить очередь по направлениям обмена в одном регистре.
  * `ИдентификаторСообщения` (Тип: `УникальныйИдентификатор`). GUID события, обеспечивающий 100% уникальность строки и выступающий ключом дедупликации.
* **Ресурсы (Данные, которые часто меняются воркером):**
  * `СтатусОбработки` (Тип: `ПеречислениеСсылка.СтатусыОчередиИнтеграции`). Содержит значения: `Новый`, `ВРаботе`, `Ошибка`, `Обработано`.
  * `КоличествоПопыток` (Тип: `Число`, длина 2, точность 0). Счетчик ретраев для отсечки «токсичных» пакетов.
  * `ДатаСледующегоСтарта` (Тип: `Дата`). Временная метка для алгоритма Экспоненциального повтора (Exponential Backoff). Воркер игнорирует запись, если текущее время меньше этой даты.
  * `ТекстОшибки` (Тип: `Строка`, неограниченная длина). Сюда сбрасывается стек ошибки при аварии для Parking Lot.
* **Реквизиты (Тяжелые данные, не участвующие в поиске):**
  * `ТелоСообщения` (Тип: `Строка`, неограниченная длина). Хранит сырой Payload (JSON/XML).

---

### ⚠️ Критичные low-level правила для Архитектора

#### 1. Почему ТелоСообщения должно быть строго в Реквизитах?
В платформе 1С поля категории «Реквизит» регистра сведений физически **никогда не включаются в индексы таблицы СУБД**. Если по ошибке сделать `ТелоСообщения` Ресурсом, СУБД будет тратить колоссальные ресурсы на пересчет и обслуживание индексов при каждой вставке, что приведет к катастрофической деградации производительности (дисковый I/O Bottleneck).

#### 2. Строка(неограниченная) vs ХранилищеЗначений (Специфика СУБД)
Платформа 1С позволяет хранить бинарные данные и длинные тексты либо в неограниченной строке, либо в объекте `ХранилищеЗначений` (BLOB).

* **Для PostgreSQL (Рекомендуемый метод):**
  Строку неограниченной длины Postgres транслирует в физический тип данных **`text`**. В PostgreSQL тип `text` оптимизирован идеально: если строка короткая, она хранится прямо внутри страницы данных таблицы. Если строка становится длинной, Postgres автоматически и незаметно для 1С уносит её в специальную структуру **TOAST** (The Oversized-Attribute Storage Technique), сжимая её на лету. Поэтому для баз на PostgreSQL в качестве `ТелаСообщения` **всегда выбирается Строка (неограниченная)**, содержащая JSON.
* **Для MS SQL Server:**
  Платформа транслирует длинную строку в тип `NVARCHAR(MAX)`. Работа с ним эффективна, но если данные интеграции содержат бинарные файлы или сжатые ZIP-потоки, то оправдано использование `ХранилищаЗначений` (тип `VARBINARY(MAX)` в SQL), чтобы избежать оверхеда на текстовую конвертацию Base64.

## 1.4. Стратегии хранения Payload (Тела сообщения): Матрица выбора Архитектора

Размер передаваемых данных (вес сообщения) — главный фактор, определяющий архитектуру хранения очереди, выделяются 4 основные стратегии.

### Стратегия 1: Прямое хранение в Реквизите РС (Строка неограниченной длины)
* **Когда использовать:** Вес одного сообщения **заведомо мал (до 50-100 КБ)**. Это 90% стандартных EDA-событий (создание контрагента, изменение статуса заказа, быстрая накладная).
* **Физика СУБД:** В PostgreSQL данные ложатся в компактный тип `text` и при превышении лимитов уходят в TOAST-таблицы. В MS SQL — в `NVARCHAR(MAX)`.
* **Плюсы:** Максимальная скорость работы. Чтение и запись происходят в один такт. Нет накладных расходов на создание сопутствующих ссылочных объектов.

### Стратегия 2: Сжатие на платформе (ХранилищеЗначений в Реквизите РС)
* **Когда использовать:** Сообщение имеет средний размер **(от 100 КБ до 10-20 МБ)** и представляет собой чистый текст (большой JSON-реестр остатков, XML-выгрузка прайсов), при этом выносить данные за пределы базы нельзя по соображениям безопасности или инфраструктурным ограничениям.
* **Физика СУБД:** Платформа транслирует `ХранилищеЗначений` в бинарный тип `VARBINARY(MAX)` (MS SQL) или `BYTEA` (PostgreSQL), принудительно сжимая текстовые данные встроенным алгоритмом Deflate на стороне `rphost` перед отправкой в СУБД.
* **Плюсы:** Жесткая экономия сетевого трафика между `rphost` и СУБД, а также дискового пространства базы данных (текст сжимается в 3-5 раз).
* **Минусы:** Высокая нагрузка на CPU серверов приложений 1С из-за постоянных операций сжатия/декомпрессии при записи и чтении каждой строки воркером.

### Стратегия 3: Вынос в служебный Справочник (Гибридный метод)
* **Когда использовать:** Сообщения **тяжелые (от 20 до 100+ МБ)**, но бизнес-требования жестко обязывают хранить их **строго внутри единого транзакционного контура СУБД** (например, для упрощения резервного копирования базы или обеспечения 100% ACID-гарантий).
* **Физика процесса:** В регистр сведений (очередь) вставляется только легкая `Ссылка` на элемент специально созданного служебного справочника (например, `втОчередьИнтеграцииПрисоединенныеФайлы`). Само тяжелое тело сообщения пишется в реквизит этого справочника.
* **Почему нельзя писать напрямую в РС:** Постоянное обновление («апдейт») строк в РС, где лежат мегабайтные бинарники, приводит к дикой фрагментации индексов и раздуванию логов транзакций СУБД при смене статусов (`Новый` $\rightarrow$ `ВРаботе` $\rightarrow$ `Обработано`). Вынос в Справочник изолирует тяжелые данные: мы один раз делаем `INSERT` тяжелого элемента справочника, а в РС-очереди быстро и дешево меняем только статусы у легкой строки со ссылкой. После обработки элемент справочника удаляется.

### Стратегия 4: Вынос во Внешнее Хранилище (Файлы / Shared NAS / S3) — Эталон Highload
* **Когда использовать:** Пакеты данных **экстремально тяжелые (от 100 МБ до нескольких гигабайт)**. Типичный пример ритейла: ночная выгрузка терабайтных исторических данных, медиа-контент, тяжелые архивы.
* **Физика процесса:** Тело сообщения вообще не пересекает границу базы данных 1С и СУБД. `ЗаписьJSON/XML` через файловый поток пишет данные напрямую в общую сетевую папку (NFS/SMB) или выталкивает в S3-совместимое объектное хранилище (MinIO / Yandex Object Storage). В регистр-очередь РС записывается только плоская текстовая строка — **путь к файлу или URL-ссылка на S3-бакет**.
* **Плюсы:** База данных 1С не раздувается и не деградирует. Исключены риски падения `rphost` по Out of Memory. Внешняя система-приемник скачивает тяжелый файл напрямую из S3 по ссылке, полностью снимая с 1С нагрузку по веб-отдаче тяжелого контента.

## 2. Подводные камни реализации Outbox (Продюсер на 1С)

Реализация слоя отправки (Producer) требует жесткого соблюдения транзакционных границ платформы 1С и СУБД. Ошибки проектирования на этом этапе мгновенно приводят к деградации параллельной работы пользователей.

### 2.1. Dual Write Антипаттерн (Двойная запись)
**Катастрофа архитектуры:** Попытка выполнить HTTP-вызов к внешнему сервису или отправить сообщение в шину (RabbitMQ/Kafka) напрямую из обработчиков событий объекта (`ПередЗаписью`, `ПриЗаписи`, `ОбработкаПроведения`).

#### Почему это приводит к падению продакшена:
1. **Дикое удлинение транзакции СУБД:** В момент работы обработчика `ПриЗаписи` в СУБД уже открыта физическая транзакция и наложены исключительные (X) блокировки на строки документа. Сетевой вызов (HTTP/REST) выполняется в среднем от 50 мс до нескольких секунд. Все эти секунды СУБД держит транзакцию открытой, намертво блокируя параллельных пользователей и вызывая лавину таймаутов.
2. **Несогласованность данных (Split-Brain):** Если сеть моргнет после отправки HTTP-запроса, внешняя система зафиксирует изменения, а транзакция 1С откатится (например, из-за нехватки остатков на следующем шаге проведения). В итоге на сайте заказ проведен и оплачен, а в 1С его физически нет.
3. **Решение:** Процедура отправки должна быть разделена на два шага. Шаг 1 — моментальная фиксация события в Outbox-регистре в рамках транзакции документа. Шаг 2 — асинхронная отправка сообщения внешним воркером за рамками транзакции документа.

---

### 2.2. Атомарность записи и защита от избыточных блокировок

**Архитектурное правило:** При регистрации нового события в Outbox-очереди ручное выставление управляемых блокировок (`Новый БлокировкаДанных`) категорически **запрещено**. Это избыточная операция, снижающая производительность системы.

#### Физика процесса:
Поскольку для каждого сообщения генерируется `Новый УникальныйИдентификатор`, платформа 1С транслирует метод `Записать()` в оператор СУБД **`INSERT`**. СУБД на низком уровне самостоятельно накладывает исключительную блокировку на новую строку. Так как потоки разных пользователей пишут строки с разными GUID, они обрабатываются полностью параллельно и не блокируют друг друга.

*✅ Правильно (Высоконагруженная атомарная регистрация события без оверхеда):*
```bsl
Процедура ПриЗаписи(Отказ)
	
	Если Отказ Или ОбменДанными.Загрузка Тогда
		Возврат;
	КонецЕсли;
	
	// Нам гарантирована открытая транзакция платформой 1С в этом обработчике.
	// Блокировка здесь НЕ нужна, так как мы делаем чистый INSERT новой строки с уникальным GUID.
	
	ИдентификаторСобытия = Новый УникальныйИдентификатор;
	
	МенеджерЗаписи = РегистрыСведений.втОчередьOutbox.СоздатьМенеджерЗаписи();
	МенеджерЗаписи.ИмяИнтеграции       = "ВыгрузкаЗаказовНаСайт";
	МенеджерЗаписи.ИдентификаторСообщения = ИдентификаторСобытия;
	МенеджерЗаписи.СтатусОбработки     = Перечисления.СтатусыОчередиИнтеграции.Новый;
	МенеджерЗаписи.ТелоСообщения       = СформироватьБыстрыйJsonЗаказа(ЭтотОбъект);
	МенеджерЗаписи.КоличествоПопыток   = 0;
	МенеджерЗаписи.ДатаСледующегоСтарта = ТекущаяДатаСеанса();
	
	// Запись строго в транзакции документа. Если документ не запишется — строка в Outbox сотрется СУБД (Rollback)
	МенеджерЗаписи.Записать(Истина); 
	
КонецПроцедуры
```

### 2.3. Предотвращение Lock Escalation (Эскалации блокировок СУБД)
**Низкоуровневая проблема MS SQL Server:** Когда сотни пользователей одновременно проводят документы, они генерируют сотни параллельных `INSERT` команд в таблицу Outbox-регистра (`_InfoRegXX`). Если количество блокировок строк (Row Locks) в рамках одной транзакции или общего пула превышает системные лимиты, MS SQL Server включает **Lock Escalation** и блокирует **всю таблицу регистра целиком**. 

В этот момент запись для всех остальных пользователей останавливается, и система ловит массовые таймауты.

#### Архитектурные способы борьбы с эскалацией при записи:

1. **Кластеризация индексов по составному ключу:**
   По умолчанию платформа 1С строит кластерный индекс регистра сведений по порядку его измерений. Измерение `ИмяИнтеграции` (строка) должно идти **первым** в списке измерений, а `ИдентификаторСообщения` (GUID) — **вторым**. Это физически группирует записи одного потока в СУБД на одних страницах данных, снижая вероятность перекрытия блокировок разных потоков обмена.
2. **Партиционирование очередей на уровне метаданных:**
   Если один регистр Outbox обслуживает абсолютно все интеграции компании, риск эскалации максимален. Для Highload-систем применяется разделение очередей: создаются физически разные регистры под ключевые бизнес-направления (например, `втОчередьОтправкиЧеков`, `втОчередьСинхронизацииНСИ`). Это физически разносит `INSERT`-потоки в разные SQL-таблицы, исключая взаимное влияние процессов.
3. **Использование PostgreSQL вместо MS SQL:**
   Поскольку PostgreSQL физически не имеет механизма эскалации блокировок строк до уровня таблицы, базы на Postgres по умолчанию более устойчивы к массовой параллельной вставке в одну таблицу-очередь, упираясь только в объем выделенной `Shared Memory` под Lock Table (что регулируется инфраструктурно).

## 3. Способы выгребания Outbox-очереди (Реализация реле)

После того как сообщения атомарно зафиксированы в СУБД, специализированный воркер (реле) должен максимально быстро считать их и передать в брокер или сеть. Существует два полярных архитектурных подхода.

### 3.1. Вариант А: Polling Publisher (Регламентные задания 1С)

Классический подход, при котором опрос таблицы Outbox выполняется силами самой платформы 1С через фоновые процессы.

* **Плюсы:** 100% чистый код на встроенном языке 1С, полное отсутствие сторонней инфраструктуры, легкая отладка и поддержка штатными разработчиками.
* **Минусы:** Высокий RTT (Round-Trip Time) из-за накладных расходов платформы, постоянная паразитная нагрузка на `rphost` и СУБД частыми запросами `SELECT` (даже когда очередь пуста).

#### Как делать правильно (Алгоритм конкурентного батчинга):
Чтобы несколько параллельных фоновых заданий могли эффективно выгребать одну очередь, не ловя взаимных дедлоков и не отправляя дубли сообщений в сеть, код обязан строго соблюдать три правила:

1. **Использование порций (`ВЫБРАТЬ ПЕРВЫЕ N`):** Воркер никогда не должен пытаться прочитать всю очередь целиком за один такт — это вызовет лавинообразный расход памяти и гарантированный Lock Escalation на MS SQL. Оптимальный размер батча — от 100 до 500 записей.
2. **Явная управляемая блокировка в режиме Исключительный:** Перед чтением пачки записей воркер обязан наложить блокировку на этот диапазон, чтобы другие параллельные воркеры его не увидели.
3. **Фильтрация по дате старта и количеству попыток:** Запрос должен игнорировать токсичные пакеты и записи, чье время повтора еще не подошло.

*✅ Правильный код воркера-поллера на встроенном языке 1С:*
```bsl
Процедура ВыгрестиОчередьОтправки() Экспорт
	
	РазмерПачки = 200;
	ТекущаяДата = ТекущаяДатаСеанса();
	ИмяПотока   = "ВыгрузкаЗаказовНаСайт";
	
	НачатьТранзакцию();
	
	Попытка
		// ШАГ 1: Блокируем ВСЁ пространство имен этой интеграции ОДНИМ НАЖАТИЕМ.
		// Параллельный воркер наткнется на эту блокировку ДО запроса.
		// Чтобы он не ждал, а сразу завершал работу (ведь первый воркер уже занят делом),
		// можно использовать БлокировкаДанных, но 1С всегда ждет таймаут. 
		// Для честного распределения по потокам в Highload используют Измерение "НомерПотока" (0..9).
		Блокировка = Новый БлокировкаДанных;
		ЭлементБлокировки = Блокировка.Добавить("РегистрСведений.втОчередьOutbox");

		ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
		Блокировка.Заблокировать();
		
		// ШАГ 2: Теперь запрос гарантированно выберет актуальные данные, 
		// и никто параллельно в эту секунду их не перехватит.
		Запрос = Новый Запрос;
		Запрос.Текст = 
		"ВЫБРАТЬ ПЕРВЫЕ " + Формат(РазмерПачки, "ЧГ=0") + "

		|	Очередь.ИдентификаторСообщения КАК ID

		// ... (заполнение параметров как у вас) ...
		ВыборкаИД = Запрос.Выполнить().Выгрузить().ВыгрузитьКолонку("ID");

			ОтменитьТранзакцию();
			Возврат;

		// ШАГ 3: Массовое обновление статуса (Пакетный UPDATE вместо цикла по МенеджеруЗаписи)
		// Для Highload правильнее использовать НаборЗаписей с отбором по массиву, 
		// либо выполнить быстрый UPDATE-запрос через выполнение RecordSet (если это внешняя СУБД),
		// но в 1С стандарт - обход выборки Набора Записей. 
		// Для простоты переводим в статус "ВРаботе":
		Для Каждого ТекID Из ВыборкаИД Цикл

	ОтправитьПачкуВСеть(ИмяПотока, ВыборкаИД);
```

### 3.1.1. Паттерн динамического надзора за потоками (Отказоустойчивое шардирование)

**Архитектурный компромисс:** 
Жесткое разделение очереди Outbox/Inbox по измерению `НомерПотока` (например, от 0 до 9) гарантирует отсутствие `Deadlocks` и полную независимость параллельных фоновых заданий на уровне СУБД. Однако в Highload-системах этот паттерн порождает критическую уязвимость: **точку отказа «Умерший воркер»**. 

Если фоновое задание Потока №3 аварийно завершится (утечка памяти `rphost`, сетевой сбой, `SIGKILL` от операционной системы), то все сообщения, регистрируемые в СУБД с меткой `НомерПотока = 3`, останутся заблокированными (или необработанными). Начнется лавинообразный рост лага очереди конкретного шарда, а часть бизнес-данных (например, чеки конкретного региона) перестанет улетать на сайт.

#### Механизм отказоустойчивости: Введение роли «Надзирателя» (Supervisor/Watchdog)

Для ликвидации этой уязвимости применяют **двухуровневую архитектуру воркеров**, добавляя дополнительный управляющий контур.

1. **Контур Исполнителей (Workers):** 10 изолированных фоновых заданий, которые просто в бесконечном цикле молотят свои потоки (каждый свой `НомерПотока`), монопольно блокируя их через `БлокировкаДанных` и выполняя отправку.
2. **Контур Надзирателя (Supervisor / Диспетчер):** Отдельное, единичное регламентное задание с минимальным интервалом запуска (например, раз в 30–60 секунд). Оно не занимается отправкой данных, его единственная задача — **телеметрия и перераспределение мертвых шардов**.

---

#### Алгоритм работы Диспетчера очереди (Self-Healing логика):

* **Шаг 1: Сбор метрик и детекция зависания.** 
  Диспетчер делает легкий запрос к таблице-очереди, собирая агрегированные данные: сколько записей находится в статусе `ВРаботе` или `Ошибка` в разрезе каждого `НомерПотока`, и какая максимальная разница между текущим временем и `ДатаСледующегоСтарта`.
* **Шаг 2: Фиксация смерти потока.**
  Если Диспетчер видит, что у Потока №3 более 10 минут висит пачка записей в статусе `ВРаботе`, а новые записи в статусе `Новый` продолжают копиться — это маркер аварии (воркер мертв, либо транзакция «залипла» в СУБД).
* **Шаг 3: Динамический перехват шарда (Failover).**
  Диспетчер инициирует процедуру «Оживления»:
  * Напрямую через объект `ФоновыеЗадания` находит и принудительно завершает зависший процесс Потока №3.
  * На уровне СУБД единым пакетным запросом сбрасывает статус всех зависших записей Потока №3 из `ВРаботе` обратно в `Новый`.
  * **Ребалансировка (Костыль выживания):** Если перезапустить Поток №3 невозможно (например, упала конкретная нода кластера 1С, за которой он был закреплен через требования назначения функционала), Диспетчер временно меняет у накопившихся в СУБД записей `НомерПотока = 3` на номера живых потоков (например, распределяет их поровну между `0, 1, 2`), принудительно заставляя другие фоновые задачи взять на себя чужой объем очереди.

#### Вывод Архитектора:
Этот паттерн существенно увеличивает сложность кодовой базы конфигурации и повышает риски инфраструктурных ошибок при отладке. Однако в Highload-системах уровня топ-ритейла это **единственный рабочий способ** обеспечить непрерывность бизнес-процессов (High Availability) силами самой платформы 1С без перехода на внешние брокеры сообщений.

### 3.2. Вариант Б: Transaction Log Mining / CDC (Debezium / Стриминг логов логов СУБД)

Ультимативный паттерн для экстремального Highload (мировой стандарт проектирования). Процесс отправки полностью выносится за пределы 1С на уровень СУБД и внешних коннекторов.

* **Плюсы:** Абсолютно нулевая нагрузка на сервер приложений 1С (`rphost`) и встроенный язык. Скорость реакции и передачи сообщения в брокер — единицы миллисекунд с момента `COMMIT` документа. Полное отсутствие блокировок таблиц запросами `SELECT`.
* **Минусы:** Высокая сложность развертывания и администрирования инфраструктуры. Главная боль — необходимость ручного низкоуровневого декодирования проприетарной структуры метаданных 1С на уровне физических таблиц СУБД.

#### Как делать правильно (Стратегия CDC для 1С):
1. **Инфраструктурный стек:** На сервере базы данных настраивается связка **Debezium + Apache Kafka Connect**. Debezium непрерывно в режиме реального времени асинхронно читает бинарный журнал транзакций СУБД (WAL-логи в PostgreSQL или Transaction Log в MS SQL).
2. **Маппинг таблиц и полей:** В конфигурационном файле Debezium настраивается фильтр строго на отслеживание команды `INSERT` в физическую таблицу нашего регистра сведений (например, таблицу `_InfoReg12345`).
3. **Декодирование системных полей:** Внешний микросервис-трансформатор считывает бинарный поток изменений из Kafka, переводит физические имена колонок СУБД (например, `_Fld12345_RRef`) в понятные бизнес-имена метаданных (`PartnerID`), на лету преобразует 16-байтные бинарные GUID 1С (`binary(16)`) в канонические текстовые строки UUID и отправляет чистый, валидный JSON в боевые топики брокера для систем-потребителей.

### 3.2.1. Инфраструктурные стратегии обхода лицензионных ограничений 1С при CDC

**Архитектурное ограничение вендора:** Согласно Лицензионному соглашению фирмы «1С», прямое чтение и анализ таблиц СУБД сторонними программными средствами (минуя сервер приложений `rphost`) запрещены. Прямое подключение Debezium/Kafka Connect к боевой СУБД нарушает правила эксплуатации и создает паразитную нагрузку на диски и процессор мастер-базы.

Для легального и безопасного использования Change Data Capture (CDC) в Highload-контурах применяются три легальные стратегии.

---

#### Стратегия А: Чтение логов с Read-Only Реплики (Физическая репликация)
Самый распространенный и надежный способ разделения нагрузок.

* **Физика процесса:** На уровне СУБД настраивается встроенная асинхронная или синхронная потоковая репликация (например, *Streaming Replication* в PostgreSQL или *AlwaysOn AlwaysOn Availability Groups* в MS SQL). Боевая база (Master) занимается только проведением документов и записью Outbox. Все бинарные логи транзакций мгновенно улетают на второй сервер — реплику (Slave).
* **Точка подключения CDC:** Коннектор Debezium подключается к **Slave-серверу (реплике)** и читает WAL-логи оттуда. 
* **Результат:** 
  * Нагрузка на Master-базу равна абсолютному нулю.
  * Лицензионная политика 1С формально не нарушается на продуктовом сервере. 
  * В случае сбоя сети на стороне Kafka брокер никак не затормозит боевой rphost.

---

#### Стратегия Б: Вынос CDC во внешнюю транзиционную БД (Sidecar DB)
Логическое разделение данных на этапе записи.

* **Физика процесса:** При записи документа 1С, вместо записи JSON-строки в регистр сведений основной базы, платформа через механизм **Внешних источников данных** (или прямой коннект через `ADO/ODBC` из кода 1С) выполняет `INSERT` в абсолютно независимую, чистую стороннюю базу данных (например, легкую инстанцию PostgreSQL, развернутую на соседнем сервере специально под нужды очередей).
* **Точка подключения CDC:** Debezium настраивается на эту внешнюю транзитную БД.
* **Результат:** 
  * Продуктовая база 1С вообще не содержит таблиц очередей, её индексы не фрагментируются, а бэкапы весят меньше.
  * Полное соответствие лицензии 1С: основная база не читается напрямую, а запись во внешний источник является штатным и документированным механизмом платформы.

---

#### Стратегия В: Логическая репликация через логический слот (Специфика PostgreSQL)
Реализация стриминга без прямого SQL-доступа к таблицам.

* **Физика процесса:** В PostgreSQL настраивается механизм логической репликации (*Logical Decoding*). Создается специальный логический слот репликации, который транслирует изменения в поток абстрактных событий. 
* **В чём фишка:** Debezium не выполняет SQL-запросы вида `SELECT` к физическим таблицам 1С. Он выступает в роли стандартного подписчика на системный поток репликации операционной системы СУБД. На уровне СУБД это выглядит как штатная технологическая процедура резервирования/передачи логов, что позволяет пройти аудит безопасности в крупных корпорациях.

---

## 4. Подводные камни реализации Inbox (Потребитель на 1С)

Слой приема данных (Consumer) в Highload-системах должен проектироваться с расчетом на экстремальные всплески трафика (например, лавина чеков в Черную Пятницу или массовая утренняя синхронизация цен со всеми кассами ритейла).

### 4.1. Взрыв HTTP-сервисов 1С при всплеске трафика (Spike)
**Архитектурный провал:** Реализация синхронной бизнес-логики непосредственно внутри метода HTTP-сервиса (например, в обработчике `Пост`). 

#### Анатомия падения rphost:
1. Когда внешний сервис (сайт, маркетплейс, кассовый сервер) присылает HTTP-запрос, платформа 1С выделяет под этот сеанс один рабочий поток процесса `rphost`.
2. Если HTTP-сервис сразу начинает выполнять тяжелые операции (парсить терабайтный JSON, искать номенклатуру по подстроке, проверять остатки и проводить документ), этот поток зависает в СУБД на время от 500 мс до нескольких секунд.
3. При наплыве пользователей (Spike) пул доступных HTTP-потоков в `rphost` моментально исчерпывается. Веб-сервер (IIS/Apache/NGINX) больше не может передать запросы в 1С и начинает массово возвращать клиентам ошибки `HTTP 504 Gateway Timeout` или `HTTP 503 Service Unavailable`. Работа всей компании парализована.

---

### 4.2. Архитектурный эталон: Пайплайн Inbox (Асинхронный прием)

**Главное правило Highload:** HTTP-сервис 1С должен быть максимально легковесным и работать как транзитный шлюз. Его единственная задача — быстро принять пакет, положить его «как есть» в СУБД и освободить сетевой поток. 

#### Схема работы пайплайна:
1. HTTP-сервис выполняет только две операции: проверку авторизации (токен/права) и мгновенную запись сырого тела сообщения (строки JSON) в регистр-очередь входящих событий (`втОчередьInbox`) в статусе `Новый`.
2. Клиенту **сразу же** возвращается ответ `HTTP 202 Accepted` (Запрос принят, но еще не обработан). Время удержания HTTP-потока снижается до 2–5 миллисекунд. Пул сокетов всегда свободен.
3. Вся тяжелая бизнес-логика (парсинг, проведение, расчеты) полностью выносится в асинхронный слой и выполняется параллельными фоновыми заданиями по алгоритму шардирования, который мы описали в Разделе 3.

*✅ Правильный код HTTP-сервиса (Пайплайн мгновенного приема пакета):*
```bsl
Функция OrdersPost(Запрос)
	
	Ответ = Новый HTTPСервисОтвет;
	
	// 1. Быстрая проверка авторизации (до открытия транзакций)
	Если Не ПроверитьАвторизациюТокена(Запрос) Тогда
		Ответ.КодСостояния = 401; 
		Возврат Ответ;
	КонецЕсли;
	
	// 2. Fail-Fast Валидация
	СыройJSON = Запрос.ПолучитьТелоКакСтроку();
	Если ПустаяСтрока(СыройJSON) Тогда
		Ответ.КодСостояния = 400; 
		Возврат Ответ;
	КонецЕсли;
	
	// 3. Выделяем ИдентификаторСообщения для обеспечения идемпотентности
	КлючИдемпотентности = Запрос.Заголовки.Получить("X-Idempotency-Key");
	IDСообщения = ?(ЗначениеЗаполнено(КлючИдемпотентности), 
	                Новый УникальныйИдентификатор(КлючИдемпотентности), 
	                Новый УникальныйИдентификатор);
	
	// 4. ИДЕАЛЬНОЕ РАСПРЕДЕЛЕНИЕ ПО ПОТОКАМ:
	// Берем последний символ GUID (он гарантированно случайный) и преобразуем в число через код символа
	СтрокаGUID = Строка(IDСообщения);
	КодПоследнегоСимвола = КодСимвола(Прав(СтрокаGUID, 1));
	НомерТекущегоПотока = КодПоследнегоСимвола % 10; // Строго равномерное распределение 0..9
	
	МенеджерЗаписи = РегистрыСведений.втОчередьInbox.СоздатьМенеджерЗаписи();
	МенеджерЗаписи.ИмяИнтеграции       = "ПриемЗаказовСМаркетплейса";
	МенеджерЗаписи.НомерПотока          = НомерТекущегоПотока;
	МенеджерЗаписи.ИдентификаторСообщения = IDСообщения;
	МенеджерЗаписи.СтатусОбработки     = Перечисления.СтатусыОчередиИнтеграции.Новый;
	МенеджерЗаписи.ТелоСообщения       = СыройJSON;
	МенеджерЗаписи.КоличествоПопыток   = 0;
	МенеджерЗаписи.ДатаСледующегоСтарта = ТекущаяДатаСеанса();
	
	Попытка
		МенеджерЗаписи.Записать(Истина); // Быстрый INSERT в СУБД
		
		Ответ.КодСостояния = 202; // Accepted
		Ответ.УстановитьТелоИзСтроки("{""status"":""accepted"",""message_id"":""" + Строка(IDСообщения) + """}");
	Исключение
		// ОБРАБОТКА ДУБЛЯ КЛЮЧА ИДЕМПОТЕНТНОСТИ:
		// Проверяем, существует ли уже запись. Если да — возвращаем 202, подтверждая прием.
		МенеджерЗаписиОтката = РегистрыСведений.втОчередьInbox.СоздатьМенеджерЗаписи();
		МенеджерЗаписиОтката.ИмяИнтеграции = "ПриемЗаказовСМаркетплейса";
		МенеджерЗаписиОтката.ИдентификаторСообщения = IDСообщения;
		МенеджерЗаписиОтката.Прочитать();
		
		Если МенеджерЗаписиОтката.Выбран() Тогда
			// Данные уже у нас в СУБД, клиенту волноваться не о чем
			Ответ.КодСостояния = 202; 
			Ответ.УстановитьТелоИзСтроки("{""status"":""accepted"",""message_id"":""" + Строка(IDСообщения) + """,""note"":""already_registered""}");
		Иначе
			// Действительно непредвиденная ошибка СУБД (например, блокировка)
			Ответ.КодСостояния = 409; 
			Ответ.УстановитьТелоИзСтроки("{""error"":""ошибка сохранения данных очереди""}");
		КонецЕсли;
	КонецПопытки;
	
	Возврат Ответ;
	
КонецФункции

```
### 4.2.1. Специфика пайплайна Inbox при различных стратегиях хранения Payload

**Архитектурное правило:** Логика работы входящего HTTP-сервиса (Пайплайна Inbox) должна динамически адаптироваться под выбранный физический тип хранения тела сообщения (см. Раздел 1.4). Попытка применить единый шаблон кода для легких JSON-событий и мегабайтных бинарных файлов гарантированно приведет к исчерпанию пула потоков `rphost`.

Ниже приведены правила модификации Шага 4 HTTP-сервиса в зависимости от веса входящего пакета:

---

#### Сценарий 1: Прямое текстовое хранение (Строка JSON / ХранилищеЗначений)
* **Как меняется код шлюза:** Код остается идентичным базовому шаблону (Раздел 4.2). Метод `Запрос.ПолучитьТелоКакСтроку()` отрабатывает мгновенно. 
* **Нюанс для `ХранилищаЗначений`:** Если принято решение сжимать данные на платформе, то в реквизит регистра пишется `Новый ХранилищеЗначений(СыройJSON, Новый СжатиеДанных(9))`. 
* ⚠️ **Highload-предупреждение:** Операция `Новый СжатиеДанных` выполняется синхронно в потоке HTTP-вызова и нагружает CPU. Применяй максимальную степень сжатия (9) только если RPS невелик. При высоком RPS снижай степень сжатия до 1–3 либо полностью отказывайся от сжатия на приеме, переложив его на внешнюю систему-отправитель.

---

#### Сценарий 2: Вынос в служебный Справочник (Гибридный метод)
* **Как меняется код шлюза:** Чтобы не раздувать и не фрагментировать таблицу регистра входящей очереди, тяжелое тело сообщения изолируется в ссылочном объекте.
* **Реализация пайплайна:** 
  1. Внутри метода HTTP-сервиса создается элемент служебного справочника (например, `втВходящаяОчередьПрисоединенныеФайлы`).
  2. Сырое тело сообщения записывается в реквизит этого справочника. Объект записывается (`Справочник.Записать()`), генерируя легальную `Ссылка`.
  3. В менеджер записи регистра `втОчередьInbox` в поле `ТелоСообщения` записывается **только полученная легкая Ссылка**, а текстовый реквизит регистра остается пустым.
* **Плюсы:** Статусы `Новый` $\rightarrow$ `ВРаботе` в регистре меняются мгновенно без блокировок тяжелых двоичных данных СУБД.

---

#### Сценарий 3: Стриминг во Внешнее Хранилище (Shared NAS / S3) — Эталон для тяжелых пакетов
* **Как меняется код шлюза:** Если внешняя система присылает файл весом от 50 МБ до нескольких гигабайт, метод `Запрос.ПолучитьТелоКакСтроку()` вызовет моментальный `Memory Leak` и падение потока по лимитам ОЗУ. Вызывать его **категорически запрещено**.
* **Реализация пайплайна:**
  1. Код шлюза считывает входящий сетевой поток порциями и перенаправляет его напрямую в файл на сетевом диске (NAS) или стримит в S3-бакет (через Chunked Upload). В оперативной памяти `rphost` в каждый момент времени находится буфер размером всего в несколько килобайт.
  2. В менеджер записи регистра `втОчередьInbox` в поле `ТелоСообщения` записывается **только плоский текстовый путь к файлу** (например, `/mnt/1c_integration_share/inbox/file_GUID.xml`) или URL-адрес S3-объекта.
  3. Асинхронный воркер при обработке этой записи (Раздел 3) будет читать данные также потоком с сетевого диска или S3-ссылки, полностью защищая базу данных и ОЗУ 1С от перегрузки.

## 5. Асинхронная обработка и параллелизм в 1С

Для утилизации всех ядер серверного процессора и минимизации лага очередей асинхронная обработка Inbox/Outbox должна выполняться в несколько параллельных потоков. Однако параллелизм в 1С без жесткого контроля логических границ неизбежно приводит к хаосу в данных.

### 5.1. Многопоточность через Фоновые Задания (Шардирование)
**Правило:** Запуск параллельных потоков должен быть строго изолирован на уровне СУБД. Как было доказано в Разделе 3.1, воркеры разделяют нагрузку по измерению `НомерПотока` (Шардирование/Partitioning). 

При записи сообщения в Inbox/Outbox ему присваивается хэш-номер потока (0..9). Каждый инстанс фонового задания (`ФоновыеЗадания.Выполнить()`) запускается со своим параметром номера потока, монопольно блокирует его через `БлокировкаДанных` и обрабатывает свой пул сообщений, вообще не пересекаясь с другими воркерами в СУБД.

---

### 5.2. Предотвращение Состояния Гонки (Race Condition)
**Анатомия бизнес-коллизии:** Шардирование по `НомерПотока` защищает таблицы очередей от дедлоков, но оно **НЕ защищает** реальные бизнес-объекты от логического искажения.

> 💡 **Пример катастрофы:** Внешняя система прислала два быстрых изменения по одному и тому же Контрагенту (например, Изменение №1: смена телефона, Изменение №2: смена адреса). Так как для сообщений сгенерировались разные GUID, хэш-функция распределения потоков может раскидать их в разные бакеты: Изменение №1 попало в `НомерПотока = 2`, а Изменение №2 — в `НомерПотока = 5`.
> 
> Два параллельных фоновых задания одновременно берут эти потоки в работу. Они одновременно считывают карточку одного и того же Контрагента, модифицируют свойства и делают `.Записать()`. Тот воркер, который вызвал `Записать()` на долю миллисекунды позже, **полностью затрет и уничтожит** изменения, внесенные первым воркером. В базе останутся некорректные данные.

#### Архитектурное решение: Паттерн «Ключ Блокировки Сущности»

Чтобы гарантировать строгую последовательность обработки изменений одной сущности в многопоточной среде, воркер перед обработкой конкретного сообщения обязан наложить **Исключительную управляемую блокировку 1С на бизнес-ключ этой сущности**.

*✅ Правильный алгоритм обработки пакета внутри воркера:*
```bsl
Процедура ОбработатьОдиночноеСообщениеInbox(Знач ТекЗаписьInbox)
	
	ИдентификаторСущности = ТекЗаписьInbox.БизнесКлючОбъекта; 
	СсылкаНаСущность = СправочникСсылкаПоGuid(ИдентификаторСущности); // Предполагаем, что функция безопасна
	
	// Гарантируем сохранение контекста сеанса
	СтарыйТаймаут = ПолучитьВремяОжиданияБлокировкиДанных();
	
	// Переключаем таймаут транзакции на 0 секунд ДО начала транзакции
	УстановитьВремяОжиданияБлокировкиДанных(0);
	
	НачатьТранзакцию();
	Попытка
		
		// ШАГ 1: Взводим управляемые блокировки на ВСЕ модифицируемые ресурсы
		Блокировка = Новый БлокировкаДанных;
		
		// Блокируем контрагента
		ЭлементБлокировкиСущность = Блокировка.Добавить("Справочник.Контрагенты");
		ЭлементБлокировкиСущность.УстановитьЗначение("Ссылка", СсылкаНаСущность);
		ЭлементБлокировкиСущность.Режим = РежимБлокировкиДанных.Исключительный;
		
		// Защита от дедлока на Шаге 3: блокируем запись очереди Inbox СРАЗУ
		ЭлементБлокировкиОчередь = Блокировка.Добавить("РегистрСведений.втОчередьInbox");
		ЭлементБлокировкиОчередь.УстановитьЗначение("ИмяИнтеграции", ТекЗаписьInbox.ИмяИнтеграции);
		ЭлементБлокировкиОчередь.УстановитьЗначение("НомерПотока", ТекЗаписьInbox.НомерПотока);
		ЭлементБлокировкиОчередь.УстановитьЗначение("ИдентификаторСообщения", ТекЗаписьInbox.ИдентификаторСообщения);
		ЭлементБлокировкиОчередь.Режим = РежимБлокировкиДанных.Исключительный;
		
		// Если хоть один ресурс занят — платформа мгновенно выбросит исключение и уйдет в общий блок
		Блокировка.Заблокировать();
		
		// ШАГ 2: Чтение и модификация объекта под защитой лока
		ОбъектКонтрагент = СсылкаНаСущность.ПолучитьОбъект();
		
		Если ОбъектКонтрагент = Неопределено Тогда
			ОбъектКонтрагент = Справочники.Contragents.СоздатьЭлемент(); // Проверь имя справочника в метаданных
			ОбъектКонтрагент.УстановитьСсылкуНового(СсылкаНаСущность);
		КонецЕсли;
		
		ОбновитьРеквизитыИзJson(ОбъектКонтрагент, ТекЗаписьInbox.ТелоСообщения);
		ОбъектКонтрагент.Записать(); 
		
		// ШАГ 3: Фиксация статуса сообщения в Inbox (Ресурс уже безопасно заблокирован на Шаге 1)
		МенеджерОчереди = РегистрыСведений.втОчередьInbox.СоздатьМенеджерЗаписи();
		МенеджерОчереди.ИмяИнтеграции          = ТекЗаписьInbox.ИмяИнтеграции;
		МенеджерОчереди.НомерПотока           = ТекЗаписьInbox.НомерПотока;
		МенеджерОчереди.ИдентификаторСообщения = ТекЗаписьInbox.ИдентификаторСообщения;
		МенеджерОчереди.Прочитать();
		
		Если МенеджерОчереди.Выбран() Тогда
			МенеджерОчереди.СтатусОбработки = Перечисления.СтатусыОчередиИнтеграции.Обработано;
			МенеджерОчереди.Записать(Истина);
		КонецЕсли;
		
		ЗафиксироватьТранзакцию();
		УстановитьВремяОжиданияБлокировкиДанных(СтарыйТаймаут); // Восстановление при успехе
		
	Исключение
		
		// Безопасно закрываем транзакцию без отравления сеанса
		Если ТранзакцияАктивна() Тогда
			ОтменитьТранзакцию();
		КонецЕсли;
		
		УстановитьВремяОжиданияБлокировкиДанных(СтарыйТаймаут); // Восстановление при ошибке
		
		// Анализируем причину падения
		ИнформацияОбОшибке = ИнформацияОбОшибке();
		ОписаниеОшибки = НРег(КраткоеОписаниеОшибки(ИнформацияОбОшибке));
		
		Если Найти(ОписаниеОшибки, "таймаут") > 0 ИЛИ Найти(ОписаниеОшибки, "блокировк") > 0 Тогда
			// Штатный параллельный отскок воркера. Ресурс занят. Уходим без паники и записи логов аварии.
			Возврат;
		Иначе
			// Реальная системная авария (ошибка заполнения, битый json, упала СУБД)
			РегистрироватьОшибкуОчереди(ТекЗаписьInbox, КраткоеОписаниеОшибки(ИнформацияОбОшибке));
		КонецЕсли;
	КонецПопытки;

КонецПроцедуры

```

#### Почему этот паттерн идеален для Highload:
1. Если два воркера схватили два разных изменения по одному контрагенту, второй воркер **не зависнет на 20 секунд** в ожидании таймаута СУБД. Благодаря перехвату исключения блокировки 1С, он мгновенно сделает `ROLLBACK`, отпустит поток и перейдет к обработке следующего (свободного) сообщения из своей пачки.
2. Гарантируется строгая последовательность (`FIFO`) внесения изменений в карточку объекта: сообщения обработаются одно за другим, исключая потерю данных при параллельной записи объекта.

## 6. Идемпотентность и Дедупликация (Idempotency Layer)

В любых распределенных системах высокой нагрузки физически невозможно гарантировать тип доставки «Ровно один раз» (Exactly-Once) без дикой деградации производительности. Стандартом является гарантия **«Как минимум один раз» (At-Least-Once)**. 

Это означает, что из-за моргания сети, рестартов брокеров (RabbitMQ/Kafka) или сетевых ретраев отправителей система 1С гарантированно и регулярно будет получать **совершенно одинаковые дубли сообщений**. Обязанность архитектора — спроектировать надежный слой дедупликации.

### 6.1. Архитектура Регистра Идемпотентности
**Правило:** Для защиты бизнес-логики от повторного выполнения создается независимый регистр сведений `втКлючиИдемпотентности`. 

Перед тем как воркер начнет разбирать и проводить документ из Inbox-очереди, он обязан проверить, не обрабатывался ли этот пакет ранее.

#### Алгоритм сквозной дедупликации:
1. Воркер берет сообщение из Inbox и открывает транзакцию.
2. Выполняется быстрая проверка (Index Seek) по измерению `ИдентификаторСообщения` в регистре `втКлючиИдемпотентности`.
3. **Если ID найден в статусе `Обработано`:** воркер понимает, что это дубль. Он **не запускает** бизнес-логику повторно. Он сразу меняет статус текущего сообщения в Inbox на `Обработано` (или удаляет его), фиксирует транзакцию и молча подтверждает брокеру успешный прием (`Acknowledge`). Клиент спокоен, база данных защищена от дублирования документов.
4. **Если ID не найден:** в регистр пишется новая запись в статусе `ВРаботе`, и запускается реальный бизнес-скрипт проведения.

*✅ Правильный код дедупликации внутри асинхронного воркера:*
```bsl
Функция ПроверитьИОбработатьИдемпотентно(Знач ТекЗаписьInbox)
	
	IDСообщения = ТекЗаписьInbox.ИдентификаторСообщения;
	ИмяПотока   = ТекЗаписьInbox.ИмяИнтеграции;
	
	СтарыйТаймаут = ПолучитьВремяОжиданияБлокировкиДанных();
	УстановитьВремяОжиданияБлокировкиДанных(0); // Highload-отскок
	
	// =========================================================================
	// ТРАНЗАКЦИЯ 1: Быстрое резервирование ключа (Длительность ~2-5 миллисекунд)
	// =========================================================================
	НачатьТранзакцию();
	Попытка
		
		Блокировка = Новый БлокировкаДанных;
		ЭлементБлокировки = Блокировка.Добавить("РегистрСведений.втКлючиИдемпотентности");
		ЭлементБлокировки.УстановитьЗначение("ИдентификаторСообщения", IDСообщения);
		ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
		
		// Если ресурс занят, платформа выбросит исключение и управление СРАЗУ перейдет в общий блок Исключение
		Блокировка.Заблокировать(); 
		
		КлючИдемпотентности = РегистрыСведений.втКлючиИдемпотентности.СоздатьМенеджерЗаписи();
		КлючИдемпотентности.ИдентификаторСообщения = IDСообщения;
		КлючИдемпотентности.Прочитать();
		
		Если КлючИдемпотентности.Выбран() Тогда
			Если КлючИдемпотентности.Статус = Перечисления.СтатусыКлючей.Обработано 
				ИЛИ КлючИдемпотентности.Статус = Перечисления.СтатусыКлючей.ВРаботе Тогда
				
				// Нашли дубль, который уже готов или обрабатывается параллельно
				ОчиститьТекущееСообщениеInbox(ТекЗаписьInbox);
				ЗафиксироватьТранзакцию();
				
				УстановитьВремяОжиданияБлокировкиДанных(СтарыйТаймаут); // Обязательно восстанавливаем
				Возврат Истина; 
			КонецЕсли;
		КонецЕсли;
		
		// Задаем статус "ВРаботе" и МГНОВЕННО фиксируем, отпуская все локи СУБД!
		КлючИдемпотентности.ИдентификаторСообщения = IDСообщения;
		КлючИдемпотентности.Статус                 = Перечисления.СтатусыКлючей.ВРаботе;
		КлючИдемпотентности.ДатаРегистрации        = ТекущаяДатаСеанса();
		КлючИдемпотентности.Записать(Истина);
		
		ЗафиксироватьТранзакцию();
		УстановитьВремяОжиданияБлокировкиДанных(СтарыйТаймаут);
		
	Исключение
		// Сюда мы падаем И в случае таймаута блокировки (Lock timeout), И при любой другой ошибке
		Если ТранзакцияАктивна() Тогда
			ОтменитьТранзакцию();
		КонецЕсли;
		
		УстановитьВремяОжиданияБлокировкиДанных(СтарыйТаймаут); // Гарантированный возврат таймаута
		
		// Отличаем штатный отскок по блокировке от реальной аварии СУБД
		ИнформацияОбОшибке = ИнформацияОбОшибке();
		Если Найти(НРег(КраткоеОписаниеОшибки(ИнформацияОбОшибке)), "таймаут") > 0 
			ИЛИ Найти(НРег(КраткоеОписаниеОшибки(ИнформацияОбОшибке)), "блокировк") > 0 Тогда
			// Это штатный параллельный поток держит ключ, просто уходим без паники в логах
			Возврат Ложь;
		Иначе
			// Реальная авария (нет прав, упал хост, ошибка структуры БД)
			РегистрироватьАвариюОчереди(ТекЗаписьInbox, КраткоеОписаниеОшибки(ИнформацияОбОшибке));
			Возврат Ложь;
		КонецЕсли;
	КонецПопытки;
	
	БизнесОтказ = ВыполнитьТяжелуюБизнесЛогику(ТекЗаписьInbox);
	
	// =========================================================================
	// ТРАНЗАКЦИЯ 2: Фиксация результата (Длительность ~2 миллисекунды)
	// =========================================================================
	НачатьТранзакцию();
	Попытка
		// Защита от дедлоков: блокируем запись перед чтением/модификацией
		Блокировка = Новый БлокировкаДанных;
		ЭлементБлокировки = Блокировка.Добавить("РегистрСведений.втКлючиИдемпотентности");
		ЭлементБлокировки.УстановитьЗначение("ИдентификаторСообщения", IDСообщения);
		ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
		Блокировка.Заблокировать();
		
		КлючИдемпотентностиФинал = РегистрыСведений.втКлючиИдемпотентности.СоздатьМенеджерЗаписи();
		КлючИдемпотентностиФинал.ИдентификаторСообщения = IDСообщения;
		КлючИдемпотентностиФинал.Прочитать();
		
		Если БизнесОтказ Тогда
			// Бизнес-логика не прошла. Фиксируем статус Ошибка в базе!
			КлючИдемпотентностиФинал.Статус = Перечисления.СтатусыКлючей.Ошибка;
			КлючИдемпотентностиФинал.Записать(Истина);
			
			// Фиксируем транзакцию, чтобы статус "Ошибка" долетел до СУБД
			ЗафиксироватьТранзакцию(); 
			Возврат Ложь;
		КонецЕсли;
		
		// Успех. Переводим в финальный статус
		КлючИдемпотентностиФинал.Статус = Перечисления.СтатусыКлючей.Обработано;
		КлючИдемпотентностиФинал.Записать(Истина);
		
		// Удаляем из входящей очереди (Inbox), так как сообщение полностью обработано
		ОчиститьТекущееСообщениеInbox(ТекЗаписьInbox);
		
		ЗафиксироватьТранзакцию();
		Возврат Истина;
		
	Исключение
		// Сюда падаем только если легла сама СУБД при фиксации статуса
		Если ТранзакцияАктивна() Тогда
			ОтменитьТранзакцию();
		КонецЕсли;
		
		РегистрироватьАвариюОчереди(ТекЗаписьInbox, КраткоеОписаниеОшибки(ИнформацияОбОшибке()));
		Возврат Ложь;
	КонецПопытки;
	
КонецФункции
```

### 6.2. TTL (Time-To-Live) для ключей дедупликации
**Проблема раздувания СУБД:** Если бизнес генерирует 100 RPS, то за месяц в регистре `втКлючиИдемпотентности` накопится **~260 миллионов записей**. База начнет дико раздуваться, индексы СУБД станут тяжелыми, а скорость Index Seek начнет деградировать.

#### Стратегия ротации и контроля TTL:
1. Ключи дедупликации не нужно хранить вечно. Практика Highload показывает, что сетевые дубли или повторные батчи из брокера прилетают в 99.9% случаев в диапазоне **от нескольких секунд до 3–7 дней** с момента аварии. Хранить историю дольше 30 дней бессмысленно.
2. В структуру регистра `втКлючиИдемпотентности` вводится обязательный ресурс `ДатаРегистрации` (Тип: `Дата`), по которому строится дополнительный индекс СУБД.
3. Раз в сутки ночное регламентное задание выполняет **порционную очистку старых ключей (дворник)**:
   ```bsl
   // Удаляем записи старше 30 дней мелкими батчами, чтобы исключить Lock Escalation
   ГраницаУдаления = ТекущаяДатаСеанса() - (30 * 24 * 60 * 60);
   ```
4. Если инфраструктура позволяет, для баз на PostgreSQL под регистр идемпотентности настраивается **секционирование (партиционирование) по месяцам**. В конце месяца дежурный скрипт СУБД делает мгновенный `DROP PARTITION` за позапрошлый месяц с нулевой нагрузкой на процессор.

## 7. Стратегии повторов и "Самолечение" (Resiliency)

Если при обработке входящего или исходящего пакета произошел сбой, система не должна останавливать весь поток обмена или бесконечно долбить упавший контур, вызывая само-DDOS инфраструктуры.

### 7.1. Алгоритм Exponential Backoff с Jitter на встроенном языке 1С
При возникновении временных технических ошибок (таймаут сети, блокировка в СУБД) время следующей попытки обработки сообщения должно расти геометрически ($2^{\text{Попытка}}$), размываясь случайным шумом (Jitter). Это предотвращает эффект «грохочущего стада» (Thundering Herd), когда сотни воркеров одновременно штурмуют только что оживший сервис.

* **Логика расчета даты следующего старта:**
  * Базовая задержка: 5 секунд.
  * Формула экспоненты: `Задержка = БазоваяЗадержка * Pow(2, КоличествоПопыток)`.
  * Добавление Jitter: К полученному числу добавляется случайное количество секунд (например, `Рандом от 0 до 15`), чтобы разнести запуск параллельных потоков во времени.
  * `ДатаСледующегоСтарта = ТекущаяДатаСеанса() + Задержка + РандомныеСекунды`.

### 7.2. Разделение ошибок: Dead Letter Queue (DLQ) внутри 1С
В Highload-системах критически важно разделять типы ошибок в блоке `Попытка...Исключение`, чтобы не тратить ресурсы сервера на заведомо провальные ретраи.

* **Инфраструктурные (Временные) ошибки:** Если перехвачена ошибка блокировки данных (`ИнформацияОбОшибке().Описание` содержит маркеры конфликта блокировок СУБД или управляемых локов 1С) или сетевой таймаут (код 504/502).
  * *Поведение:* Статус сообщения меняется на `Ошибка`, счетчик `КоличествоПопыток` инкрементируется (+1), рассчитывается дата следующего старта по Exponential Backoff.
* **Бизнес-ошибки / Битые данные (Токсичные пакеты):** Если перехвачена ошибка валидации данных (например, в JSON не заполнено обязательное поле, прилетел несуществующий ИНН, или возникла ошибка десериализации схем). Код 1С **никогда не исправит это сам**, сколько бы попыток мы ни делали.
  * *Поведение:* Сообщение моментально, минуя ретраи, переводится в статус `Брак` (внутренняя **DLQ / Parking Lot**). Поток освобождается. Пакет изолируется в регистре для последующего ручного разбора аналитиками или разработчиками.

## 8. Эксплуатация и Очистка (Day 2 Operations & Bloat Prevention)

Таблицы очередей Inbox/Outbox подвержены колоссальному объему DML-операций: непрерывный `INSERT` новых строк, постоянный `UPDATE` статусов и массовый `DELETE` отработанных пакетов. Такое поведение ломает стандартную логику хранения данных в СУБД.

### 8.1. Проблема Postgres Bloat и MS SQL Fragmentation
* **PostgreSQL (Проблема Bloat):** Движок Postgres использует модель MVCC. При каждом `UPDATE` статуса (например, `Новый` $\rightarrow$ `ВРаботе` $\rightarrow$ `Обработано`) старая версия строки физически не изменяется, а помечается как мертвая и рядом пишется новая версия строки. То же происходит при `DELETE`. Если очередь молотит с высокой скоростью, таблица регистра и её индексы за часы превращаются в «швейцарский сыр», раздуваясь в размерах в 10–50 раз (**Table/Index Bloat**). Скорость чтения деградирует по экспоненте, а регламентный `VACUUM` начинает тормозить всю СУБД.
* **MS SQL Server (Проблема Фрагментации):** Постоянные хаотичные вставки с GUID-измерениями и удаление строк приводят к дикой фрагментации страниц кластерного индекса (до 95–98%). СУБД тратит огромные ресурсы на чтение разрозненных страниц с диска.

### 8.2. Пакетная очистка (Soft Delete vs Hard Delete)
Чтобы защитить СУБД от деградации, в Highload-очередях категорически **запрещено** физически удалять строчку (`МенеджерЗаписи.Удалить()`) или перезаписывать её пустой структурой сразу в транзакции воркера на каждую операцию.

#### Стратегия выживания СУБД:
1. **Мягкое удаление (Soft Delete):** Воркер в транзакции только быстро меняет статус на `Обработано`. Это дешевый `UPDATE`, который локализован в одной странице данных.
2. **Пакетное жесткое удаление (Hard Delete — Дворник):** Раз в сутки, в период минимальной активности (ночью), запускается специализированный служебный скрипт. Он выбирает успешно обработанные записи (`Статус = Обработано`) старше установленного Retention-периода и удаляет их **строго ограниченными батчами** (например, по 1000 строк за один SQL-запрос через НаборЗаписей). Это исключает Lock Escalation и минимизирует генерацию логов транзакций СУБД.
3. **Обязательное обслуживание индексов:** Сразу после ночной жесткой очистки очередей, на уровне СУБД в обязательном порядке должны выполняться регламентные процедуры обслуживания конкретно этих таблиц очередей:
   * Для PostgreSQL: Направленный **`VACUUM FULL`** или **`REINDEX`** для физического схлопывания таблицы на диске и высвобождения памяти операционной системе.
   * Для MS SQL Server: Операция **`ALTER INDEX ... REBUILD`** для полной пересборки фрагментированных страниц индексов.
