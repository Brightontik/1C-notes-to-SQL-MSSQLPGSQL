# Обработка состояний гонки (Race Condition Handling) в 1С:Предприятие

## 1. Анатомия Race Condition в среде 1С

### Что такое Состояние Гонки (Race Condition)
**Состояние гонки (Race Condition)** — это ситуация в многопоточной среде, когда корректность работы алгоритма зависит от случайного порядка или времени выполнения параллельных потоков кода. 

В 1С это означает, что несколько параллельных сеансов одновременно считывают, рассчитывают и модифицируют одни и те же ячейки данных (регистры, справочники) в СУБД. Без должной синхронизации результат выполнения бизнес-логики становится непредсказуемым.

### Источники многопоточности в 1С:Предприятие 8.3
Платформа 1С является многопоточной на уровне кластера серверов. Конкурентную среду за ресурсы базы данных создают:
* **HTTP- и Web-сервисы:** Каждый входящий запрос обрабатывается сервером приложений в отдельном потоке (сеансе). При высокой интенсивности запросов от внешних систем вероятность коллизий максимальна.
* **Параллельные фоновые задания:** Механизм `ФоновыеЗадания.Выполнить()` распределяет задачи по рабочим процессам (`rphost`), создавая конкуренцию на уровне СУБД.
* **Параллельное проведение документов:** Асинхронные обработчики, очереди или многопоточный подбор партий.

---

### Классический кейс: «Double Spending» (Избыточное списание)

Ситуация возникает, когда бизнес-логика разделяет процесс проверки («Достаточно ли товара?») и процесс фиксации («Списать товар») на две отдельные дисковые операции, между которыми нет блокирующего барьера.

#### Схема развития гонки (Timeline)

Ниже приведена пошаговая визуализация того, как два параллельных HTTP-запроса приводят к отрицательному остатку на складе (исходный остаток = 150 единиц, каждый поток хочет списать по 100 единиц).


| Время | Поток №1 (HTTP-Сервис А) | Поток №2 (HTTP-Сервис Б) | Остаток в БД | Итог операции |
| :--- | :--- | :--- | :---: | :--- |
| **T1** | Инициирует транзакцию | — | **150** | Поток 1 стартовал |
| **T2** | `ВЫБРАТЬ Остаток...` (Видит 150) | Инициирует транзакцию | **150** | Поток 2 стартовал |
| **T3** | Проверка логики: `150 >= 100` (Разрешено) | `ВЫБРАТЬ Остаток...` (Видит 150) | **150** | **Поток 2 прочитал грязные/устаревшие данные!** |
| **T4** | `ЗАПИСАТЬ Остаток = 50` | Проверка логики: `150 >= 100` (Разрешено) | **50** | Поток 1 зафиксировал списание |
| **T5** | Завершает транзакцию (Commit) | — | **50** | Поток 1 отпустил ресурсы |
| **T6** | — | `ЗАПИСАТЬ Остаток = 50 - 100` | **-50** | **Бизнес-логика нарушена (минус на складе)** |
| **T7** | — | Завершает транзакцию (Commit) | **-50** | Система перешла в некорректное состояние |

#### Почему это происходит на уровне СУБД?
По умолчанию в режиме управляемых блокировок 1С транзакции на уровне СУБД (MS SQL, PostgreSQL) выполняются в режиме изоляции **Read Committed** (Чтение завершенных черновиков). 
* При обычном запросе `ВЫБРАТЬ` СУБД накладывает кратковременную разделяемую блокировку (Shared Lock), которая снимается *сразу после чтения строки*, не дожидаясь окончания всей транзакции.
* Поток №2 беспрепятственно читает те же самые данные, пока Поток №1 еще выполняет расчеты в памяти сервера приложений.

### Низкоуровневая схема взаимодействия потоков

```text
Поток №1 (Сеанс HTTP)             База Данных (СУБД)             Поток №2 (Сеанс HTTP)
        │                                 │                                 │
(T1) Start Trans ───────────────────────> │                                 │
(T2) SELECT Остаток ────────────────────> │                                 │
        │ <────────────────────────────── │ [150]                           │
        │                                 │ <─────────────────────── Start Trans (T2)
        │                                 │ <─────────────────────── SELECT Остаток (T3)
        │                                 │ ──────────────────────> [150] (Устарело!)
(T4) Проверка: ОК                         │                                 │
(T4) UPDATE Остаток = 50 ───────────────> │                                 │
(T5) COMMIT ────────────────────────────> │                                 │
        │                                 │ ──────────────────────> Проверка: ОК (T4)
        │                                 │ <────────────────────── UPDATE Остаток = -50 (T6)
        │                                 │ <────────────────────── COMMIT (T7)
        ▼                                 ▼                                 ▼
```

**Критический вывод для разработчика:** Обычная проверка остатков через запрос `ВЫБРАТЬ` (без явного указания намерений блокировки) внутри транзакции **не защищает** от параллельного изменения этих данных другими потоками.

## 2. Виды гонок при интеграциях и параллельной обработке

При построении распределенных систем и интеграционных контуров с 1С разработчик сталкивается с тремя основными паттернами состояний гонки. Каждая из них требует своего изолированного подхода к решению.

### A. Гонка «Чтение-Изменение-Запись» (Read-Modify-Write)
Классический паттерн конкуренции за существующий ресурс. Возникает при параллельном обновлении числовых или статусных полей одного и того же объекта (баланс контрагента, статус заказа, накопительные итоги) разными потоками интеграции.

* **Механика проблемы:** Поток №1 и Поток №2 одновременно считывают текущее состояние объекта (например, СтатусЗаказа = `Согласован`). Оба потока на основании этой информации принимают решение о переводе статуса дальше. Поток №1 записывает статус `ВДоставке`. Поток №2, не зная об изменениях, перезаписывает данные статусом `Отменен`.
* **Последствия:** Потеря обновлений (Lost Update), нарушение консистентности и логической цепочки бизнес-процесса.

### B. Гонка «Проверка-Действие» (Check-Then-Act)
Паттерн гонки, завязанный на проверку факта существования данных. Чаще всего проявляется при реализации механизмов дедупликации, регистрации уникальных внешних ИД или обработке входящих очередей (Inbox-таблиц).

* **Механика проблемы:** Поток №1 получает внешнее событие с `UUID_123`. Он проверяет по регистру сведений: *"Есть ли уже запись с таким UUID?"*. Регистр пуст. Поток №1 начинает длительный процесс создания документа. В этот же миг Поток №2 получает дубль этого сообщения, проверяет тот же регистр, видит, что записи все еще нет (так как Поток №1 еще не зафиксировал транзакцию), и тоже начинает создавать документ.
* **Последствия:** Дублирование документов в базе, нарушение уникальности данных, падение транзакций по ошибке СУБД `Duplicate key value violates unique constraint`.

```text
Поток №1 (Пакет А)                 Регистр Inbox (СУБД)               Поток №2 (Пакет Б)
        │                                   │                                  │
(T1) Проверка UUID_123 ───────────────────> │                                  │
        │ <────────────────────────── [Нет записи]                             │
        │                                   │ <─────────────────── Проверка UUID_123 (T2)
        │                                   │ <────────────────────────── [Нет записи]
(T3) Создание Документа №1                  │                                  │
(T4) Запись в Inbox (ОК) ─────────────────> │                                  │
        │                                   │ <─────────────────── Создание Документа №2 (T3)
        │                                   │ <─────────────────── Запись в Inbox (T5)
        ▼                                   ▼                                  ▼
                              [ОШИБКА: Дубликат ключа!]
```

### C. Гонка за рамками 1С (В брокерах сообщений / Out-of-order delivery)
Архитектурная гонка, возникающая на стыке транспортного уровня (Kafka, RabbitMQ) и принимающей стороны 1С из-за нарушения очередности доставки сообщений (Out-of-order delivery).

* **Механика проблемы:** В шине данных генерируются два последовательных события по одному объекту: Событие №1 (`Документ Создан`) и Событие №2 (`Документ Изменен`). Из-за сетевых задержек, сбоя сети или конфигурации брокера (например, обработка сообщений из одной группы разными воркерами/партициями Kafka) Событие №2 прилетает в HTTP-сервис 1С раньше, чем Событие №1.
* **Последствия:** 1С пытается изменить или обновить документ, которого физически еще не существует в базе данных, что приводит к падению интеграционного потока или созданию "оторванных" от логики данных.

### Сводная таблица видов гонок и зон их возникновения


| Тип гонки | Где возникает (Локация) | Главный маркер проблемы | Базовый способ защиты |
| :--- | :--- | :--- | :--- |
| **Read-Modify-Write** | Внутри транзакции 1С | Изменение существующей записи | Управляемая блокировка (Исключительная) или `ДЛЯ ИЗМЕНЕНИЯ` |
| **Check-Then-Act** | Внутри транзакции 1С | Проверка на существование / поиск дублей | Блокировка "намерения создания" по уникальному ключу |
| **Out-of-order delivery** | Транспортный уровень (Шина / 1С) | Нарушение хронологии событий | Версионирование сообщений + Идемпотентность приемника |

## 3. Инструменты СУБД и Платформы для борьбы с гонками

Для эффективного подавления Race Condition на низком уровне необходимо понимать, как взаимодействуют механизмы транзакций СУБД и собственный менеджер блокировок 1С.

### A. Транзакции и уровни изоляции: миф о защите СУБД
Большинство СУБД (PostgreSQL, MS SQL Server) по умолчанию работают на уровне изоляции **Read Committed (Чтение зафиксированных данных)**. Этот режим спроектирован для высокой параллельности работы пользователей, но он абсолютно **не защищает от состояния гонки**.

* **От чего защищает Read Committed:** Исключительно от *грязного чтения* (Dirty Read). Поток №2 никогда не увидит промежуточные данные Потока №1, пока Поток №1 не вызовет операцию `COMMIT`.
* **Почему возникает гонка:** На уровне Read Committed блокировки чтения (`SELECT`) являются кратковременными. СУБД блокирует строку ровно на тот микромомент, пока считывает её с диска в память. Как только чтение строки завершено, блокировка снимается. 
* **Результат:** СУБД позволяет двум параллельным транзакциям беспрепятственно прочитать одно и то же значение подряд. С точки зрения СУБД, обе транзакции ведут себя законно, но на уровне прикладной бизнес-логики 1С происходит наложение операций.

---

### B. Управляемые блокировки 1С (`БлокировкаДанных`)
В современном управляемом режиме 1С (который обязателен для всех типовых конфигураций БСП) платформа делегирует СУБД только базовые операции, а контроль логической целостности берет на себя. 

**`БлокировкаДанных`** — это единственный штатный и архитектурно правильный инструмент синхронизации параллельных потоков кода в 1С. Он работает на уровне кластера серверов 1С (в памяти менеджера блокировок) и оперирует не физическими строками таблиц SQL, а метаданными (пространствами блокировок).

```text
               [ Кластер серверов 1С:Предприятие ]
              /                                   \
      [ Поток №1 ]                             [ Поток №2 ]
          │                                         │
    Пытается установить                       Пытается установить
    Исключительную блокировку                 Исключительную блокировку
    по Складу "Основной"                      по Складу "Основной"
          │                                         │
          ▼                                         ▼
┌────────────────────────────────────────────────────────────────────────┐
│                      Менеджер управляемых блокировок 1С                │
├────────────────────────────────────────────────────────────────────────┤
│ [Активна: Поток №1] ──► Очередь ожидания: [Поток №2]                   │
└────────────────────────────────────────────────────────────────────────┘
          │                                         │
     РАЗРЕШЕНО (T1)                           ЖДЕТ (До T5)
          │                                         │
          ▼                                         ▼
┌────────────────────────────────────────────────────────────────────────┐
│                      Уровень СУБД (Read Committed)                     │
└────────────────────────────────────────────────────────────────────────┘
```

### C. Режимы управляемых блокировок: Разделяемый vs Исключительный

При использовании объекта `БлокировкаДанных` разработчик обязан явно указать логический режим взаимодействия с квантом данных.

#### 1. Разделяемый режим (`РежимБлокировкиДанных.Разделяемый`)
* **Суть:** "Я читаю эти данные и хочу быть уверен, что никто не изменит их, пока моя транзакция не завершится. Но другие потоки тоже могут параллельно читать их вместе со мной".
* **Матрица совместимости:** Совместим с другими *Разделяемыми* блокировками. Несовместим с *Исключительными*.
* **Критический маркер (Антипаттерн):** Разделяемый режим **категорически запрещено использовать** в транзакциях типа «Чтение-Изменение-Запись». Если Поток №1 и Поток №2 установят разделяемую блокировку на один и тот же остаток товара, они оба успешно прочитают его. Но когда Поток №1 попытается записать изменения, ему потребуется исключительный замок, и система уйдет в состояние взаимной блокировки (**Deadlock**).

#### 2. Исключительный режим (`РежимБлокировкиДанных.Исключительный`)
* **Суть:** "Я планирую анализировать и изменять эти данные. Пока моя транзакция не завершится, ни один другой параллельный поток не должен даже заходить на чтение этой области".
* **Матрица совместимости:** Полностью **несовместим** ни с какими другими блокировками (ни с Разделяемыми, ни с Исключительными).
* **Результат применения:** Гарантирует, что только один поток в единицу времени работает с выделенным квантом данных. Все конкурентные потоки выстраиваются сервером приложений 1С в очередь ожидания.

#### Таблица совместимости управляемых блокировок 1С



| Текущий замок \ Новый запрос | Разделяемый (Shared) | Исключительный (Exclusive) |
| :--- | :---: | :---: |
| **Разделяемый (Shared)** | **Совместимы** (OK) | Конфликт (Ожидание / Таймаут) |
| **Исключительный (Exclusive)** | Конфликт (Ожидание / Таймаут) | Конфликт (Ожидание / Таймаут) |

**Инженерное правило памятки:** Для борьбы с Race Condition (как в кейсах *Read-Modify-Write*, так и в *Check-Then-Act*) перед выполнением запроса проверки данных всегда должна взводиться управляемая блокировка в **Исключительном** режиме по целевому аналитическому разрезу (например, Номенклатура + Склад).

## 4. Практические паттерны защиты от Race Condition в 1С

Для устранения состояний гонки на практике применяются три основных инженерных подхода в зависимости от типа данных и архитектурных требований приложения.

### A. Паттерн «Пессимистическая блокировка по Ключу»
Применяется для защиты бизнес-логики вида **Check-Then-Act** (например, регистрация уникальных ИД интеграции, создание строго одного документа под событие, распределение уникального промокода).

#### Суть паттерна
Выделяется атомарный строковый или ссылочный бизнес-ключ (`UUID контрагента`, `Артикул`, `НомерЗаказа`). Поток взводит **управляемую исключительную блокировку** на этот ключ *до* выполнения проверок на существование данных или чтения остатков. Параллельные потоки с таким же ключом засыпают на входе в транзакцию.

#### Реализация через служебный регистр сведений
Если объект блокирования — виртуальная сущность (например, «Уникальный Внешний ИД пакета»), у которой нет жестких измерений в регистрах, под нее создается служебный регистр сведений: `БлокировкаБизнесПроцессов` (Измерение: `КлючБлокировки` [Строка]). Данные в него физически писать не нужно — он используется только как *пространство имен* для менеджера блокировок.

```bsl
// Пример кода в обработчике HTTP-сервиса (Прием пакета интеграции)
НачатьТранзакцию();

// Шаг 1: Формируем уникальный бизнес-ключ
БизнесКлюч = "ИНТЕГРАЦИЯ_ЗАКАЗ_" + ВходящийIDЗаказа;

// Шаг 2: Взводим пессимистический барьер по ключу
Блокировка = Новый БлокировкаДанных;
ЭлементБлокировки = Блокировка.Добавить("РегистрСведений.БлокировкаБизнесПроцессов");
ЭлементБлокировки.УстановитьЗначение("КлючБлокировки", БизнесКлюч);
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
Блокировка.Заблокировать(); // Конкурентные потоки зависают здесь

// Шаг 3: Безопасное выполнение Check-Then-Act
Запрос = Новый Запрос;
Запрос.Текст = 
    "ВЫБРАТЬ 1 

    |ИЗ РегистрСведений.InboxЛогДанных 
    |ГДЕ ВнешнийID = &ВходящийIDЗаказа";
Запрос.УстановитьПараметр("ВходящийIDЗаказа", ВходящийIDЗаказа);

Если Запрос.Выполнить().Пустой() Тогда
    // Действие (Act): данных нет, безопасно создаем объект без риска дублирования
    СоздатьНовыйДокументВБазе(ВходящийIDЗаказа, ДанныеПакета);
КонецЕсли;

ЗафиксироватьТранзакцию();
```

### B. Паттерн «Оптимистическая блокировка (Версионирование)»
Используется в высоконагруженных системах для паттернов **Read-Modify-Write**, где физическое ожидание потоков (очереди блокировок) неприемлемо, а конфликты одновременного изменения происходят относительно редко.

#### Суть паттерна
В таблицу данных (например, в свойства объекта или ресурсы независимого регистра сведений) добавляется реквизит/ресурс `ВерсияДанных` типа `Число` (счетчик) или `УникальныйИдентификатор` / `Timestamp`. 
Каждый поток считывает данные вместе с текущей версией. При записи проверяется условие соответствия версии. На уровне СУБД это превращается в условный SQL-запрос:
```sql
UPDATE Table_Name SET Balance = New_Balance, Version = New_Version WHERE ID = Target_ID AND Version = Old_Version;
```

#### Механика работы в 1С
Для ссылочных объектов (Документы, Справочники) платформа 1С выполняет этот механизм **автоматически** на уровне объектной записи с помощью скрытого реквизита `ВерсияДанных`. 
Если вам нужно реализовать это вручную для плоских таблиц (например, независимого регистра сведений «БалансКонтрагентов»):

```text
Поток №1 (Прочитал: Версия=1) ──────► Проверка OK ──────► Запись (БД Версия=1) ──► Спешно UPDATE (Успех, Версия=2)


Поток №2 (Прочитал: Версия=1) ──────► Проверка OK ───────────────────────────────► Попытка UPDATE ГДЕ Версия=1 
                                                                                   [Отказ СУБД: 0 строк обновлено]
                                                                                   [Логика 1С: Rollback + Повтор]
```

При падении оптимистической блокировки транзакция полностью откатывается, поток выполняет `Паузу()` на случайную долю секунды, считывает актуальные данные (уже измененные Потоком №1), забирает новую `ВерсиюДанных` и повторяет попытку расчета.

### C. Специфика работы с агрегатными данными (Управление итогами)
При параллельной записи движений документов в регистры накопления (например, `ОстаткиТоваров`) классические блокировки по измерениям `Склад + Номенклатура` наглухо парализуют работу склада при проведении многострочных накладных. Для решения этой проблемы 1С использует встроенный механизм **разделения итогов**.

#### Разделение итогов
Это платформенный способ борьбы с Race Condition за ресурсы агрегатных таблиц. В СУБД к физической таблице итогов регистра добавляется скрытое поле `_Splitter` (Разделитель). 
* **При записи движений:** Вместо обновления одной строки остатка товара (что вызвало бы блокировку строки для всех остальных потоков), платформа 1С генерирует *новую* независимую строку итогов с уникальным значением разделителя. Разные потоки пишут изменения одного и того же товара параллельно, не блокируя друг друга.
* **При чтении остатков:** Механизм автоматически выполняет агрегацию `СУММА(...)` по всем разделителям для конкретной номенклатуры, отдавая точный консолидированный остаток.

#### Критическая особенность для разработчика
Если внутри транзакции проведения вам необходимо гарантировать, что товар не уйдет в минус, и вы используете метод `ЗаблокироватьДляИзменения()` в запросе или свойство `БлокироватьДляИзменения = Истина` у регистра:
* Платформа 1С для этой выборки **намеренно отключает разделитель итогов** и переводит транзакцию в режим жесткой пессимистической управляемой блокировки. 
* Это необходимо, чтобы зафиксировать точное состояние регистра на момент проверки и не допустить овердрафта со стороны параллельного потока.

```text
                 ┌────────────────────────────────────────────────────────┐
                 │    РегистрНакопления: Режим "Разделение Итогов"        │
                 └────────────────────────────────────────────────────────┘
                               /                              \
       При обычной ЗАПИСИ движений                      При чтении с флагом "ДЛЯ ИЗМЕНЕНИЯ"
       ───────────────────────────                      ───────────────────────────────────
       Разделитель ИТОГОВ: ВКЛЮЧЕН                      Разделитель ИТОГОВ: ОТКЛЮЧЕН
       Потоки пишут параллельно в разные строки.         Включается жесткий пессимистический барьер.
       Блокировок нет. Высокий TPS.                     Конкуренты ждут завершения транзакции.
```

### 4.1 Гонка на уровне интеграций и Кэша (Distributed & Cache Race Conditions)

Когда 1С работает в связке с внешним высокопроизводительным слоем кэширования (например, **Redis**) или обрабатывает асинхронные ответы от внешних API (интернет-магазины, CRM, платежные шлюзы), классические управляемые блокировки 1С перестают защищать данные, так как процессы физически происходят вне кластера 1С.

Здесь выделяют два критических сценария гонки и паттерны борьбы с ними.

---

#### Паттерн №1: «Гонка обновления кэша и БД» (Cache Invalidation Race)

Возникает в гибридных архитектурах (например, *Сайт ──► Redis ──► 1С*), когда один поток читает/обновляет кэш, а в это же время 1С записывает изменения в основную БД.

* **Механика проблемы:** 
  1. На сайте происходит Cache Miss (данные по остаткам товара в Redis истекли).
  2. Поток Сайта идет в БД 1С, считывает актуальный остаток (**100 шт.**).
  3. В этот же миг менеджер в 1С списывает этот товар. Транзакция в 1С успешно завершается и обновляет БД (**0 шт.**), попутно отправляя команду очистки/инвалидации кэша в Redis.
  4. Поток Сайта (из п.2), задержавшись из-за сетевого лага, добегает до Redis и записывает туда прочитанное ранее *старое* значение (**100 шт.**).
* **Результат:** База данных 1С содержит актуальный `0`, но во внешнем кэше Redis намертво застревает грязное устаревшее значение `100`. Сайт продает несуществующий товар.

```text
Поток Сайта (Read)                Слой Кэша (Redis)                 Поток 1С (Write)
        │                                 │                                 │
(T1) Cache Miss ────────────────────────► │                                 │
(T2) SELECT 1С (Видит 100) ───────────────┼─────────────────────────────────┼────────┐
        │                                 │                                 │        │ (1С меняет в БД)
        │                                 │ <────────────────────── UPDATE 1С (Стало 0) ─────┘
        │                                 │ <────────────────────── Del Key (Инвалидация)
        │                                 │   [Кэш пуст]                    │
(T3) SET Key = 100 ─────────────────────> │                                 │
        ▼                                 ▼                                 ▼
                              [Гонка выиграна Сайтом:                       
                               В кэше застряло "100", а в БД "0"]
```

##### Архитектурное решение: Запись «Дата Изменения + Версия Данных»

Для борьбы с этим в модель данных (и в Redis, и в 1С) внедряется строгий контроль версий и временных меток. Вместо простой перезаписи ключа `SET key value` операция кэширования превращается в условную атомарную сверку.

1. **Комбинированный токен версии:** При изменении объекта в 1С, в шину/кэш транслируется не просто новое состояние, а пакет: `{"Value": Данные, "Updated_At": "2026-05-24T21:00:00.123", "Version": 1042}`.
2. **Атомарные Lua-скрипты в Redis:** Потоки чтения (Сайт) и потоки записи (1С) обновляют Redis не прямой командой, а через выполнение легковесного Lua-скрипта внутри Redis. Поскольку Redis однопоточен, Lua-скрипт выполняется абсолютно атомарно:

```lua
-- Пример логики Lua-скрипта для Redis
local current_version = redis.call('HGET', KEYS[1], 'version')
local new_version = tonumber(ARGV[1])

if not current_version or tonumber(current_version) < new_version then
    redis.call('HSET', KEYS[1], 'value', ARGV[2], 'version', new_version, 'timestamp', ARGV[3])
    return 1 -- Кэш обновлен, так как прилетели более свежие данные
else
    return 0 -- Отклонено: попытка записать устаревшие данные из прошлого
end
```

---

#### Паттерн №2: «Лавина кэша» (Cache Stampede / Thundering Herd)

Специфическая гонка "наперегонки к базе данных 1С", которая может полностью обрушить сервер приложений.

* **Механика проблемы:** Высоконагруженный сайт запрашивает у Redis цены номенклатуры (частота — 500 запросов в секунду). В момент `T` у ключа истекает время жизни (TTL). В эту же миллисекунду все 500 параллельных потоков веб-сервера получают `Cache Miss`. Не найдя значения в кэше, **все 500 потоков одновременно** отправляют тяжелые SQL-запросы в СУБД 1С.
* **Результат:** Процессы `rphost` захлебываются, загрузка процессора на сервере СУБД улетает в 100%, транзакции 1С начинают падать по таймауту блокировок.

##### Инженерное решение: Распределенный замок (Distributed Lock / Redlock)

Чтобы избежать лавины, первый поток, поймавший `Cache Miss`, пытается захватить распределенный Mutex-флаг в Redis с коротким таймаутом (инструмент синхронизации вне 1С). Все остальные потоки, не сумев взять замок, засыпают на стороне веб-сервера на несколько миллисекунд и пробуют повторно прочитать кэш, пока первый поток спокойно вычисляет данные в 1С и обновляет Redis.

```text
Веб-Поток №1                      Слой Кэша (Redis)                  База Данных 1С
     │                                    │                                 │
Cache Miss ─────────────────────────────> │                                 │
SETNX lock_key (Успех) ─────────────────> │                                 │
     │                                    │ ─── Heavy Query ──────────────> │
Веб-Поток №2                              │                                 │
     │                                    │                                 │
Cache Miss ─────────────────────────────> │                                 │
SETNX lock_key (Отказ, занято) ─────────> │                                 │
Спит 50мс...                              │                                 │
     │                                    │ <─── Возврат Данных ────────────│
     │                                    │                                 │
Записывает данные в кэш ────────────────> │                                 │
Удаляет lock_key ───────────────────────> │                                 │
     │                                    │                                 │
[Поток №2 проснулся]                      │                                 │
GET key (Успех, взял из кэша) ──────────> │                                 │
     ▼                                    ▼                                 ▼
                         (Вместо 2 тяжелых запросов в 1С ушел только 1)
```

Реализация захвата такого распределенного замка перед вызовом HTTP-сервисов 1С на стороне вызывающей системы (Node.js/Python/PHP) минимизирует паразитное конкурентное чтение:
```sql
-- Команда Redis для безопасного взятия блокировки (атомарный замок на 5 секунд)
SET lock_order_123 "UUID_потока" NX PX 5000
```

## 5. Архитектурные методы предотвращения гонок на уровне интеграции

Когда нагрузка и объемы данных растут, решать проблему Race Condition исключительно блокировками внутри 1С становится экономически невыгодно (падает общая пропускная способность — TPS). На стыке 1С и внешних систем применяются фундаментальные архитектурные паттерны синхронизации.

### A. Строгое партиционирование (Sharding по бизнес-ключу)
Если обрабатывать входящие сообщения в случайных параллельных потоках, гонка «Check-Then-Act» неизбежна. Архитектурное решение — гарантировать, что все связанные сообщения обрабатываются строго последовательно, а независимые — параллельно.

#### Механика паттерна в брокерах (Kafka / RabbitMQ Sharding)
При отправке события в шину данных внешняя система обязана рассчитать **Ключ Партиционирования (Partition Key)**. Например, `ИД_Склада`, `GUID_Контрагента` или `ИД_Заказа`.

* **Как это работает:** Брокер пропускает этот ключ через хэш-функцию: \(Partition = Hash(Key) \pmod{N}\), где \(N\) — количество партиций (очередей) в топике. 
* **Результат:** Сообщения с одинаковым ключом (например, все операции по Складу №5) **жестко привязываются к одной конкретной партиции** и выстраиваются в очередь FIFO (First-In, First-Out).
* **На стороне 1С:** За каждой партицией закрепляется ровно один воркер (сеанс HTTP-сервиса или фонового задания 1С). Так как один воркер обрабатывает события по Складу №5 строго друг за другом, возникновение состояния гонки физически невозможно на уровне дизайна системы. При этом Склад №2 и Склад №3 обрабатываются другими воркерами параллельно в соседних потоках.

```text
  [ Входящие события ] ──► [ Хэш-функция Брокера ] ──► [ Партиции Kafka ] ──► [ Обработчики 1С ]
  
  Заказ_123 (Склад_A) ───────────┐
                                 ▼
  Заказ_789 (Склад_B) ──► [ Hash(Склад) ] ──► [ Партиция 0 ] ──► [ Воркер 1С №1 ] (Строго по очереди)
                                 ▲            (Только Склад_A)
  Заказ_456 (Склад_A) ───────────┘
                                 
                                 └──────────► [ Партиция 1 ] ──► [ Воркер 1С №2 ] (Строго по очереди)
                                              (Только Склад_B)
```

---

### B. Идемпотентный Upsert на уровне СУБД
При высокой интенсивности мелких транзакций синхронизации данных (например, Inbox/Outbox таблицы логов интеграции, регистрация статусов пакетов) классический цикл 1С `НачатьТранзакцию() -> ПроверитьЗапрос() -> СоздатьМенеджерЗаписи() -> Записать() -> ЗафиксироватьТранзакцию()` создает колоссальный оверхед и блокировки.

#### Суть паттерна
Использование атомарных SQL-операций, объединяющих проверку и запись на уровне ядра СУБД за один сингл-квери такт (Single-turn operation). В PostgreSQL для этого применяется синтаксис **`INSERT ... ON CONFLICT DO UPDATE`** (или `MERGE` в MS SQL).

#### Реализация в контексте 1С
Поскольку стандартный ORM-движок 1С не умеет генерировать конструкции `ON CONFLICT`, для критически нагруженных интеграционных Inbox-таблиц разработчики используют механизмы **прямых запросов** к СУБД через ADO или Native-драйверы (в режиме безопасного взаимодействия) либо оптимизируют прикладную запись.

Если Поток №1 и Поток №2 одновременно попытаются выполнить атомарный SQL-запрос `UPSERT` для одной строки, ядро PostgreSQL разрешит конфликт на уровне уникального индекса (Unique Constraint):

```sql
-- Пример прямого SQL-запроса, отправляемого из 1С в PostgreSQL для Inbox лога
INSERT INTO _InfoRg12345 (_Fld991, _Fld992, _Fld993) -- Измерения: ИД_Пакета, Статус, Версия
VALUES ('packet_uuid_abc', 'IN_PROGRESS', 1)
ON CONFLICT (_Fld991) -- Если пакет с таким UUID уже существует в базе
DO UPDATE SET 
    _Fld992 = EXCLUDED._Fld992,
    _Fld993 = _InfoRg12345._Fld993 + 1 -- Атомарно инкрементируем внутреннюю версию
WHERE _InfoRg12345._Fld993 < EXCLUDED._Fld993; -- Защита от Out-of-order delivery
```

##### Результат для архитектуры:
1. **Атомарность:** Проверка уникальности ключа и обновление записи происходят внутри СУБД в рамках единой низкоуровневой операции над страницей данных.
2. **Идемпотентность:** Повторный прилет того же пакета из-за сбоя сети не создаст дубль в 1С и не вызовет ошибку транзакции, а безопасно обновит существующую запись или проигнорирует её (если `DO NOTHING`), предотвратив Race Condition.

## 6. Подводные камни и цена борьбы с Race Condition

Внедрение механизмов защиты от состояний гонки имеет свою инженерную цену. Избыточная или неграмотная блокировка ресурсов может привести к критическому падению производительности системы или полной остановке параллельной обработки.

### A. Проблема взаимных блокировок (Deadlocks)

**Взаимная блокировка (Deadlock)** — это тупиковая ситуация, когда два или более потока бесконечно ждут освобождения ресурсов, заблокированных друг другом. Это прямой результат избыточной или нескоординированной защиты от Race Condition.

#### Механика взаимоблокировки
* **Поток №1** обрабатывает Накладную №1: успешно взводит исключительную блокировку на **Товар А** и собирается заблокировать **Товар Б**.
* В эту же миллисекунду **Поток №2** обрабатывает Накладную №2: успешно взводит исключительную блокировку на **Товар Б** и собирается заблокировать **Товар А**.

```text
  Поток №1 ──► [ Блокирует Товар А ] ──► Ждет освобождения ──► [ Товар Б ]
                                                                   ▲
                                                                   │
  Поток №2 ──► [ Блокирует Товар Б ] ──► Ждет освобождения ──► [ Товар А ]
```
Оба потока засыпают в вечном ожидании. Через несколько секунд СУБД или менеджер блокировок 1С принудительно аварийно завершает одну из транзакций по ошибке `TDEADLOCK`, откатывая все её изменения.

#### Золотое правило предотвращения Deadlocks: Фиксированный порядок
Единственный надежный способ гарантировать отсутствие взаимных блокировок — **строго упорядочивать объекты блокирования (ресурсы) перед их захватом**.

1. Конкурентные потоки должны блокировать ресурсы в абсолютно одинаковой последовательности.
2. Перед вызовом `Блокировка.Заблокировать()` массив блокируемых элементов (например, список Номенклатуры или Складов) должен быть **отсортирован по возрастанию** (по GUID, Коду или Артикулу).

Если оба потока сначала отсортируют список, то и Поток №1, и Поток №2 попытаются сначала заблокировать **Товар А**, а уже затем **Товар Б**. Поток №2 честно уснет еще на первом шаге, не создавая тупиковой ситуации.

---

### B. Падение пропускной способности (Throughput Serialization)

Абсолютная безопасность данных через жесткие пессимистические блокировки неизбежно приводит к **сериализации (выстраиванию в одну очередь)** параллельной системы. Если перегнуть палку, многопоточное приложение (HTTP-сервисы, фоновые задания) превращается в однопоточное.

* **Проблема:** Если вы заблокируете весь регистр или крупный узел данных (например, весь Склад вместо конкретной позиции Номенклатуры), параллельные HTTP-запросы выстроятся в длинную цепочку ожидания. Вместо целевых 500 RPS (Requests Per Second) система начнет выдавать максимум 10-15 RPS.
* **Поиск баланса:** Блокировка должна быть максимально **гранулярной (узконаправленной)**. Блокировать нужно минимально возможный квант данных (например, сочетание `Номенклатура + Склад + Характеристика`).

---

### C. Таймауты блокировок: Стратегия обработки сбоев

Когда поток выстраивается в очередь за заблокированным ресурсом, он не может ждать вечно. В 1С время ожидания управляемой блокировки ограничено параметром `ВремяОжиданияБлокировки` (по умолчанию 20 секунд). По истечении этого времени поток падает с исключением: **`Ошибка блокировки данных. Слишком длинное ожидание...`**

Для интеграционных шлюзов и API просто упасть с ошибкой — антипаттерн. Необходимо закладывать стратегии восстановления:

#### 1. Уровень HTTP/Web-сервисов (Паттерн Retry-After)
Если HTTP-поток не смог захватить блокировку по ключу за короткое время (для API время ожидания лучше снизить до 3-5 секунд), транзакция откатывается, а внешнему клиенту возвращается статус **`HTTP 503 Service Unavailable`** или **`HTTP 423 Locked`** с заголовком `Retry-After: 2`. Это сообщает внешней системе, что ресурс временно занят и запрос нужно повторить через 2 секунды.

#### 2. Уровень фоновых заданий (Паттерн Exponential Backoff)
При обработке очередей внутри 1С упавшее из-за блокировки фоновое задание не должно маркироваться как аварийное. Исключение перехватывается, и задание планируется заново с экспоненциальной задержкой:
```bsl
// Шаблон обработки таймаута в очереди фоновых заданий
Попытка
    НачатьТранзакцию();
    // ... Блокировка и расчет бизнес-логики ...
    ЗафиксироватьТранзакцию();
Исключение
    Информация = ИнформацияОбОшибке();
    Если Найти(Информация.Описание, "Ошибка блокировки данных") > 0 Тогда
        ОтменитьТранзакцию();
        // Перезапускаем задачу через фоновое задание с отложенным стартом (Backoff)
        ТекущаяЗадержка = РассчитатьЭкспоненциальнуюЗадержку(КоличествоПопыток);
        ПараметрыЗапуска = Новый Структура("IDЗадачи, Попытка", IDЗадачи, КоличествоПопыток + 1);
        
        ФоновыеЗадания.Выполнить("ИнтеграцияСлужебный.ВыполнитьЗадачуОтложено", 
            Новый Массив(ПараметрыЗапуска),, ТекущаяЗадержка);
    Иначе
        ВызватьИсключение; // Другие критические ошибки прокидываем дальше
    КонецЕсли;
КонецПопытки;
```
