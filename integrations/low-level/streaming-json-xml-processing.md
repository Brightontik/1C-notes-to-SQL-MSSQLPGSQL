# Потоковая обработка JSON и XML в Highload (Стандарты и Базис)

**Область применения:** HTTP/REST-сервисы, фоновые задания обмена, обработка сверхтяжелых пакетов данных (от 50 МБ до нескольких гигабайт).
**Цель правила:** Исключить лавинообразный рост потребления оперативной памяти процессом `rphost`, предотвратить падение кластера 1С по Out of Memory и обеспечить стабильное время обработки пакетов.

---

## 1. Теоретический базис (Data Representation Limits)

### 1.1. JSON vs XML в Highload
Выбор формата передачи данных напрямую влияет на пропускную способность сети и утилизацию процессора.

* **XML (eXtensible Markup Language):**
  * **Минусы:** Обладает огромным избыточным весом (оверхед) из-за обязательного дублирования имен полей в закрывающих тегах (например, `<PartnerID>GUID</PartnerID>`). На больших массивах данных этот избыточный вес может составлять до 30–50% от общего объема сетевого трафика.
  * **Плюсы:** Идеален для строгого контроля типов и валидации данных на сетевом шлюзе с помощью XSD-схем. Позволяет жестко валидировать структуру до того, как данные начнут обрабатываться кодом прикладного решения.
* **JSON (JavaScript Object Notation):**
  * **Плюсы:** Является стандартом де-факто для современных REST API. Минималистичен, не содержит закрывающих тегов, легко читается и имеет меньший размер по сравнению с аналогичным XML.
  * **Минусы:** Слабая типизация. В JSON отсутствуют нативные типы для Даты (передается как строка) и Бинарных данных (требует упаковки в Base64, что увеличивает размер бинарного файла на 33%).

### 1.2. Анатомия утечки памяти (DOM vs Pull Parsing)
Главная причина падения процессов `rphost` при обработке файлов — неправильный выбор модели парсинга.

* **DOM-модель (Document Object Model):**
  * **Принцип работы:** Парсер читает текстовый файл целиком, анализирует его структуру и строит в оперативной памяти полное дерево объектов, полностью повторяющее иерархию файла.
  * **Проблема в 1С:** Встроенный язык 1С работает поверх тяжелых C++ оберток. При использовании DOM-модели возникает жесткая линейная зависимость: **пакет данных размером 200 МБ в текстовом виде на диске при полной загрузке в память превращается в 1.5–2 ГБ занятой оперативной памяти процесса `rphost`**. Если параллельно запущено несколько таких потоков, сервер падает по Out of Memory.
  * **Запрещенные методы для Highload:** 
    * `ФабрикаXDTO.ПрочитатьXML()` — при передаче файла целиком.
    * `Новый ЧтениеXML; ЧтениеXML.ОткрытьФайл(); ДокументDOM = Новый ДокументDOM;`
    * `Функция ПрочитатьJSON()` — при передаче всего потока без использования функций фильтрации.

* **Потоковая модель (Stream / Pull Parsing):**
  * **Принцип работы:** Парсер работает как однонаправленный курсор. Он открывает файл/сетевой поток и читает его последовательно, выдавая программисту ровно один логический элемент за раз (начало элемента, имя свойства, значение, конец элемента).
  * **Поведение памяти:** В оперативной памяти в каждый момент времени находится ровно одна обрабатываемая строка или один узел. После перехода курсора на следующий элемент, память из-под предыдущего мгновенно освобождается. Потребление ОЗУ процессом `rphost` остается **константным** (обычно в пределах нескольких мегабайт) независимо от того, весит файл 10 МБ или 10 ГБ.

### 1.3. Концепция потока (Stream)
Поток — это абстракция для порционной работы с данными с использованием фиксированного размера буфера ввода-вывода (I/O Buffer).
* Вместо загрузки файла в память, код настраивает «трубопровод». 
* Платформа считывает с диска или сети небольшую порцию байт (например, 4 КБ или 8 КБ) в буфер, обрабатывает её на встроенном языке и запрашивает следующую порцию.
* **Правило архитектора:** Весь жизненный цикл тяжелого пакета от чтения из сети до записи в базу данных должен быть организован исключительно через потоковые объекты.

## 2. Реализация потоковой сериализации в 1С (Streaming Writers)

При формировании тяжелых пакетов (выгрузка миллионов строк остатков, цен или чеков) данные должны записываться напрямую в целевой поток (файл или сокет) порциями, без промежуточного накопления объектов в оперативной памяти `rphost`.

### 2.1. Механика потоковой записи JSON (ЗаписьJSON)
**Правило:** Для выгрузки больших массивов данных запрещено использовать связку высокоуровневого метода `ЗаписатьJSON(Запись, Структура/Массив)` со структурами, заполненными миллионами элементов. Вместо этого необходимо применять исключительно низкоуровневые токенные методы управления потоком.

*❌ Неправильно (Вызовет лавинообразный рост ОЗУ и падение rphost):*
```bsl
// Память забьется еще на этапе формирования массива в 1С
МассивОстатков = Новый Массив;
Пока Выборка.Следующий() Цикл
	МассивОстатков.Добавить(Новый Структура("Номенклатура,Остаток", Выборка.Код, Выборка.Остаток));
КонецЦикла;

ЗаписьJSON = Новый ЗаписьJSON;
ЗаписьJSON.ОткрытьФайл("C:\temp\heavy_export.json");
ЗаписатьJSON(ЗаписьJSON, МассивОстатков); // Избыточный расход памяти
ЗаписьJSON.Закрыть();
```

*✅ Правильно (Потребление памяти стабильно ~0 МБ, данные сразу улетают в файл/сеть):*
```bsl
ЗаписьJSON = Новый ЗаписьJSON;
ЗаписьJSON.ОткрытьФайл("C:\temp\heavy_export.json");

// Пишем заголовки пакета вручную через токены
ЗаписьJSON.ЗаписатьНачалоОбъекта();
ЗаписьJSON.ЗаписатьИмяСвойства("timestamp");
ЗаписьJSON.ЗаписатьЗначение(ТекущаяДатаСеанса());
ЗаписьJSON.ЗаписатьИмяСвойства("stocks");
ЗаписьJSON.ЗаписатьНачалоМассива();

Пока Выборка.Следующий() Цикл
	// Каждый элемент массива пишется точечно, память из-под строки сразу очищается
	ЗаписьJSON.ЗаписатьНачалоОбъекта();
	
	ЗаписьJSON.ЗаписатьИмяСвойства("sku");
	ЗаписьJSON.ЗаписатьЗначение(Выборка.Код);
	
	ЗаписьJSON.ЗаписатьИмяСвойства("qty");
	ЗаписьJSON.ЗаписатьЗначение(Выборка.Остаток);
	
	ЗаписьJSON.ЗаписатьКонецОбъекта();
КонецЦикла;

ЗаписьJSON.ЗаписатьКонецМассива();
ЗаписьJSON.ЗаписатьКонецОбъекта();
ЗаписьJSON.Закрыть();
```

### 2.2. Механика потоковой записи XML (ЗаписьXML)
**Правило:** Запрещено формировать гигантские XML-реестры через генерацию тяжелых XDTO-пакетов фабрики (`ФабрикаXDTO.Записать()`), если структура документа линейна или известна заранее. Формирование должно идти через токенные методы `ЗаписатьНачалоЭлемента()` и `ЗаписатьТекст()`.

*✅ Правильно (Формирование XML-пакета без накладных расходов XDTO):*
```bsl
ЗаписьXML = Новый ЗаписьXML;
ЗаписьXML.ОткрытьФайл("C:\temp\heavy_export.xml", "UTF-8");
ЗаписьXML.ЗаписатьОбъявлениеXML();

ЗаписьXML.ЗаписатьНачалоЭлемента("Root");

Пока Выборка.Следующий() Цикл
	ЗаписьXML.ЗаписатьНачалоЭлемента("Row");
	
	ЗаписьXML.ЗаписатьНачалоЭлемента("SKU");
	ЗаписьXML.ЗаписатьТекст(Выборка.Код);
	ЗаписьXML.ЗаписатьКонецЭлемента();
	
	ЗаписьXML.ЗаписатьНачалоЭлемента("QTY");
	ЗаписьXML.ЗаписатьТекст(Строка(Выборка.Остаток));
	ЗаписьXML.ЗаписатьКонецЭлемента();
	
	ЗаписьXML.ЗаписатьКонецЭлемента(); // Row
КонецЦикла;

ЗаписьXML.ЗаписатьКонецЭлемента(); // Root
ЗаписьXML.Закрыть();
```

### 2.3. Контроль структуры и экономия тактов CPU
Платформа 1С предоставляет механизм автоматического контроля валидности генерируемого потока (например, чтобы программист случайно не забыл закрыть объект или массив).

* **Свойство `ПроверятьСтруктуру`:**
  * **На этапе разработки и тестирования:** Необходимо устанавливать `ЗаписьJSON.ПроверятьСтруктуру = Истина;`. Это заставит платформу выбрасывать исключение, если нарушена вложенность скобок, что уберет генерацию битых пакетов.
  * **В продакшене:** Настройка **обязательно** отключается (`ЗаписьJSON.ПроверятьСтруктуру = Ложь;`). Проверка структуры на лету сжирает значительное количество процессорных тактов (CPU) на валидацию стэка токенов. Если код отлажен, эта проверка на проде становится избыточной тратой ресурсов сервера.

### 2.4. Вывод напрямую в сокет или файл (Исключение промежуточных строк)
**Правило:** Категорически запрещено собирать тело тяжелого ответа в промежуточную переменную типа `Строка` (например, через `ЗаписьJSON.УстановитьСтроку()`), чтобы затем передать ее в HTTP-запрос. Данные должны стримиться напрямую на диск или в сетевой сокет.

*❌ Неправильно (Строка в памяти rphost продублирует вес файла и вызовет Memory Leak):*
```bsl
ЗаписьJSON.УстановитьСтроку();
// ... логика циклической записи ...
ГигантскийТекст = ЗаписьJSON.Закрыть(); // Ошибка! Строка заняла 1 ГБ ОЗУ

Ответ = Новый HTTPСервисОтвет(200);
Ответ.УстановитьТелоИзСтроки(ГигантскийТекст);
```

*✅ Правильно (Использование временного файла-буфера):*
```bsl
ИмяВременногоФайла = ПолучитьИмяВременногоФайла("json");

ЗаписьJSON = Новый ЗаписьJSON;
ЗаписьJSON.ОткрытьФайл(ИмяВременногоФайла);
// ... логика циклической записи ...
ЗаписьJSON.Закрыть();

Ответ = Новый HTTPСервисОтвет(200);
// Платформа сама свяжет файл с сокетом и передаст его потоком по сети
Ответ.УстановитьТелоИзФайла(ИмяВременногоФайла); 
```

### 2.5. Специфика работы с файлами в многонодовом (многосерверном) кластере 1С

**Проблема изоляции локальных дисков:** 
Метод `ПолучитьИмяВременногоФайла()` генерирует путь к локальному Temp-каталогу операционной системы конкретного рабочего сервера (ноды кластера), на котором в данную миллисекунду выполняется рабочий процесс `rphost`. 

Если кластер 1С состоит из двух и более серверов (например, `App-Server-01` и `App-Server-02`), то файл, записанный потоком на `App-Server-01`, будет физически **недоступен** для процессов, выполняющихся на `App-Server-02`. Это приводит к критическим сбоям при передаче файлов, асинхронном выполнении фоновых заданий или повторных попытках отправки (Retry Policy).

Для организации сквозного доступа к потоковым файлам без утилизации ОЗУ `rphost` применяются следующие архитектурные стратегии:

---

#### Вариант А: Использование Временного Хранилища 1С (Платформенный метод)
Передача файла между нодами кластера средствами самой платформы.

* **Физика процесса:** 
  Код формирует временный файл локально, закрывает поток записи, а затем вызывает метод `ПоместитьВоВременноеХранилище(Новый ДвоичныеДанные(ПутьКФайлу))`. Платформа сериализует этот файл и уносит его внутрь базы данных (в служебную таблицу СУБД `Config` / `Files`). Любая другая нода кластера сможет прочитать этот файл по полученному адресу `e1cib/tempstorage/...`.
* **Плюсы:** Работает «из коробки» на любой инфраструктуре, автоматическая очистка сеансовых данных платформой.
* **Критический минус для Highload:** Полностью нарушает концепцию потоковой обработки! Конструктор `Новый ДвоичныеДанные(ПутьКФайлу)` принудительно **загружает весь файл целиком в оперативную память** `rphost`. Если файл весит 1 ГБ, этот метод мгновенно вызовет утечку RAM, сводя на нет всю эффективность потоковых писателей (`Streaming Writers`).

---

#### Вариант Б: Сетевое Общее Хранилище (Shared Network Storage — NAS / NFS / SMB)
Вынос каталога временных файлов за пределы локальных серверов приложений на выделенный отказоустойчивый файловый массив.

* **Физика процесса:** 
  На выделенном сетевом хранилище разворачивается общая папка (NFS для Linux, SMB/UNC-путь для Windows). На всех серверах кластера 1С этот сетевой ресурс монтируется в одну и ту же точку файловой системы (например, как папка `/mnt/1c_integration_share/` на всех Linux-нодах).
* **Реализация в коде:**
  Вместо `ПолучитьИмяВременногоФайла()` путь формируется явно в выделенную общую папку с генерацией уникального GUID в имени:
  ```bsl
  ИмяФайлаЭкспорта = "/mnt/1c_integration_share/" + Строка(Новый УникальныйИдентификатор) + ".json";
  ЗаписьJSON.ОткрытьФайл(ИмяФайлаЭкспорта);
  ```
* **Плюсы:** Сохраняется 100% потоковая запись/чтение. Файл пишется напрямую в сеть порциями, ОЗУ серверов 1С не нагружается. Любой сервер кластера имеет моментальный доступ к файлу.
* **Минусы:** Появляется жесткая зависимость от сетевой доступности файлового хранилища. Требуется ручной контроль очистки старых файлов кодом 1С (через фоновые задания), так как автоматический Temp-очиститель ОС сюда не дотянется.

---

#### Вариант В: Объектное Хранилище (S3-Compatible Storage) — Эталон для Highload
Вынос хранения файлов из контура 1С во внешнюю специализированную инфраструктуру (MinIO, Яндекс Облако Object Storage, AWS S3).

* **Физика процесса:** 
  Сервер 1С формирует файл локально на текущей ноде. Сразу после закрытия потока, файл асинхронно или синхронно выталкивается во внешнее S3-хранилище по HTTP-протоколу порциями (Chunked Upload). Вместо пути к файлу между системами и нодами кластера передается только плоская строка — текстовый URL-адрес объекта в S3 хранилище.
* **Плюсы:** 
  * Безграничное масштабирование, диски серверов 1С никогда не переполнятся.
  * Автоматическое сжатие, версионирование данных и ротация (Retention Policy на уровне бакетов S3 удалит мусор сама через 3 дня).
  * Идеально для интеграций: внешняя система (приемник) забирает тяжелый файл напрямую из S3 по ссылке, вообще снимая с 1С нагрузку по отдаче контента.
* **Минусы:** Требуется наличие S3-инфраструктуры в компании и написание/внедрение обертки для работы с S3 API (на встроенном языке 1С или через внешние компоненты).

## 3. Реализация потоковой десериализации в 1С (Streaming Readers)

При приеме тяжелых пакетов (например, миллионных накладных, реестров чеков или полных прайс-листов от поставщиков) загрузка файла целиком в память через объектные модели (`ДокументDOM` или `ФабрикаXDTO`) запрещена. Чтение должно выполняться строго в потоковом (итераторном) режиме.

### 3.1. Итераторный подход через ЧтениеJSON
**Правило:** Разбор файла должен строиться вокруг цикла `Пока ЧтениеJSON.Прочитать() Цикл`. Платформа выступает в роли токенизатора, возвращая на каждом шаге тип текущего элемента через свойство **`ТипТекущегоЗначения`** и системное перечисление **`ТипЗначенияJSON`**.

*✅ Правильно (Низкоуровневый разбор токенов с каноническим синтаксисом 1С):*
```bsl
ЧтениеJSON = Новый ЧтениеJSON;
ЧтениеJSON.ОткрытьФайл("/mnt/1c_integration_share/heavy_import.json");

Пока ЧтениеJSON.Прочитать() Цикл
	ТекущийТокен = ЧтениеJSON.ТипТекущегоЗначения;
	
	Если ТекущийТокен = ТипЗначенияJSON.ИмяСвойства Тогда
		ТекущееИмяСвойства = ЧтениеJSON.ТекущееЗначение;
		
	ИначеЕсли ТекущийТокен = ТипЗначенияJSON.Число 
		ИЛИ ТекущийТокен = ТипЗначенияJSON.Строка 
		ИЛИ ТекущийТокен = ТипЗначенияJSON.Булево Тогда
		
		// Логика обработки конкретного примитивного значения
		Если ТекущееИмяСвойства = "total_amount" Тогда
			ИтоговаяСумма = ЧтениеJSON.ТекущееЗначение;
		КонецЕсли;
	КонецЕсли;
КонецЦикла;

ЧтениеJSON.Закрыть();
```

### 3.2. Итераторный подход через ЧтениеXML
**Правило:** Последовательный обход дерева XML выполняется с помощью метода `ЧтениеXML.Прочитать()`. Платформа не строит иерархию объектов в памяти, а лишь перемещает указатель по текстовому файлу, считывая типы узлов (`ТипУзлаXML.НачалоЭлемента`, `ТипУзлаXML.Текст`, `ТипУзлаXML.КонецЭлемента`).

*✅ Правильно (Потоковый парсинг линейных XML-данных):*
```bsl
ЧтениеXML = Новый ЧтениеXML;
ЧтениеXML.ОткрытьФайл("/mnt/1c_integration_share/heavy_import.xml");

Пока ЧтениеXML.Прочитать() Цикл
	Если ЧтениеXML.TipUzla = ТипУзлаXML.НачалоЭлемента Тогда
		ИмяЭлемента = ЧтениеXML.Имя;
		
	ИначеЕсли ЧтениеXML.ТипУзла = ТипУзлаXML.Текст Тогда
		Если ИмяЭлемента = "Barcode" Тогда
			ТекущийШтрихкод = ЧтениеXML.Значение;
			// Сразу выполняем поиск или регистрацию в базе
		КонецЕсли;
	КонецЕсли;
КонецЦикла;

ЧтениеXML.Закрыть();
```

### 3.3. Паттерн «Гибридный парсинг» (Windowed Deserialization)
**Архитектурная проблема чисто токенного чтения:** Описывать разбор сложных, глубоко вложенных JSON объектов исключительно через `Если...ИначеЕсли` по токенам — это ад для поддержки. Код превращается в нечитаемую «лапшу» из сотен строк, где легко запутаться в открывающихся и закрывающихся скобках.

**Решение — Гибридный парсинг:** Платформа 1С позволяет комбинировать подходы. Мы читаем файл потоком до тех пор, пока не дойдем до ключевого узла (начало массива элементов). Как только курсор встает на начало конкретного объекта (карточки товара), мы «выкусываем» из общего потока ровно этот один объект в компактную структуру 1С с помощью высокоуровневой функции `ПрочитатьJSON(Чтение)`. 

* **Физика процесса:** В памяти одновременно находится только один текущий объект. Мы обрабатываем его, записываем в БД, структура автоматически очищается сборщиком мусора, а поток переходит к следующему элементу массива. Механизм защищен от сдвигов и корректно читает первый элемент массива, независимо от пробелов в файле.

*✅ Правильно (Реализация гибридного парсинга на встроенном языке 1С):*
```bsl
ЧтениеJSON = Новый ЧтениеJSON;
ЧтениеJSON.ОткрытьФайл("/mnt/1c_integration_share/millions_orders.json");

// Шаг 1: Доходим потоком строго до токена НАЧАЛО МАССИВА, который идет после "orders_list"
МассивНайден = Ложь;
Пока ЧтениеJSON.Прочитать() Цикл
	Если ЧтениеJSON.ТипТекущегоЗначения = ТипЗначенияJSON.ИмяСвойства 
		И ЧтениеJSON.ТекущееЗначение = "orders_list" Тогда
		
		// Нам мало найти имя свойства. Нам нужно убедиться, что за ним идет массив.
		// Делаем контролируемый шаг вперед внутри этого же блока.
		Если ЧтениеJSON.Прочитать() И ЧтениеJSON.ТипТекущегоЗначения = ТипЗначенияJSON.НачалоМассива Тогда
			МассивНайден = Истина;
			Прервать;
		КонецЕсли;
	КонецЕсли;
КонецЦикла;

Если Не МассивНайден Тогда
	ВызватьИсключение "Критическая ошибка интеграции: Секция orders_list не найдена или повреждена";
КонецЕсли;

// Текущее положение указателя: Курсор стоит строго НА токене "[" (НачалоМассива).

// Шаг 2: Итерируемся по потоку дальше. 
// Первый же вызов Прочитать() сдвинет курсор с "[" на "{" первого объекта (или на "]" если массив пустой).
Пока ЧтениеJSON.Прочитать() Цикл
	
	// Если встретили КонецМассива (]) - пакет полностью обработан, выходим
	Если ЧтениеJSON.ТипТекущегоЗначения = ТипЗначенияJSON.КонецМассива Тогда
		Прервать;
	КонецЕсли;
	
	// Если встретили НачалоОбъекта ({) - это старт ОДНОГО конкретного заказа
	Если ЧтениеJSON.ТипТекущегоЗначения = ТипЗначенияJSON.НачалоОбъекта Тогда
		
		// Шаг 3: Выкусываем ровно один заказ. 
		// Метод ПрочитатьJSON() забирает весь объект целиком и останавливает поток НА токене "}" (КонецОбъекта).
		ТекущийЗаказ = ПрочитатьJSON(ЧтениеJSON); 
		
		// Шаг 4: Обрабатываем одиночный объект
		СоздатьДокументЗаказПокупателя(ТекущийЗаказ);
		
		// Финал итерации: На следующей итерации проверка условия Пока ЧтениеJSON.Прочитать() 
		// сдвинет поток с положения "}" (КонецОбъекта) на следующий токен (либо "{" следующего заказа, либо "]").
	КонецЕсли;
	
КонецЦикла;

ЧтениеJSON.Закрыть();
```

### 3.4. Архитектурное примечание: Поведение указателя на скрытых токенах (Разделителях)

При ручной итерации по потоку JSON разработчик должен четко понимать разницу между обработкой стандартного JSON-массива и потоковых форматов типа JSON Lines (NDJSON).

#### Сценарий А: Формат JSON Lines / NDJSON (Каждый объект на новой строке без запятых)
Если входящий файл сформирован без использования запятых-разделителей между объектами, то при вызове `ЧтениеJSON.Прочитать()` указатель парсера перемещается напрямую с токена `КонецОбъекта` (`}`) предыдущего элемента на токен `НачалоОбъекта` (`{`) следующего. Никаких промежуточных тактов чтения не происходит.

#### Сценарий Б: Стандартный JSON-массив (Объекты разделены запятыми)
В валидном JSON-массиве между объектами гарантированно присутствуют запятые (`}, {`). 
* **Физика платформы 1С:** В перечислении `ТипЗначенияJSON` отсутствует элемент для представления запятой. Когда метод `Прочитать()` наталкивается на запятую, свойство `ТипТекущегоЗначения` продолжает возвращать **предыдущий успешный статус чтения** (или служебный статус), фактически выполняя автопромотку (пропуск) разделителя.
* **Влияние на код:** Метод высокоуровневого выкусывания `ПрочитатьJSON(ЧтениеJSON)` останавливает разбор строго на токене `КонецОбъекта` (`}`). На следующей итерации цикла условие `Пока ЧтениеJSON.Прочитать()` берет на себя физический сдвиг указателя. Платформа самостоятельно съедает запятую-разделитель и переводит курсор на следующий валидный токен (либо на `{` нового объекта, либо на `]` конца массива).

**Вывод для разработчика:** Написание дополнительных холостых вызовов `ЧтениеJSON.Прочитать()` для «пропуска запятой» категорически запрещено — это гарантированно приведет к сдвигу указателя и потере первого токена внутри следующей бизнес-сущности. Платформа полностью контролирует разделители сама.

## 4. Сжатие «на лету» (On-the-fly Compression)

При передаче терабайтных пакетов данных между системами сетевой канал становится узким местом. Однако классические подходы к архивации в 1С создают избыточную нагрузку на диски, полностью нивелируя выигрыш от сжатия.

### 4.1. Антипаттерн временных файлов (Дисковый I/O Bottleneck)
**Проблема классического подхода:** Традиционный алгоритм 1С-разработчиков выглядит так: 
1. Выгрузить огромную выборку (например, 1 ГБ JSON) во временный файл на SSD.
2. Открыть этот файл объектом `ЗаписьZipФайла` и упаковать его в новый архив `.zip`.
3. Считать получившийся ZIP-архив с диска методом `Новый ДвоичныеДанные()`.
4. Отправить байты по сети и удалить оба файла.

* **В чём катастрофа для Highload:** Этот подход порождает **Дисковый I/O Bottleneck** (бутылочное горлышко ввода-вывода). Сервер тратит колоссальное количество времени и тактов процессора на физическую запись и чтение гигабайтных массивов с диска. Нагрузка на дисковую подсистему взлетает до 100%, замедляя работу СУБД и соседних сеансов `rphost`.

### 4.2. Реализация сжатия в 1С без дискового оверхеда

**Ограничение платформы 1С:** Встроенный язык «1С:Предприятие» не поддерживает концепцию сквозных Writable-потоков для архивации. Объект `ЗаписьZipФайла` не умеет генерировать исходящий поток компрессии, в который можно было бы направить вывод `ЗаписьJSON`. Архиватор умеет работать только как конечный потребитель готового потока.

Для реализации сжатия без создания промежуточного сырого текстового файла на диске применяется двухэтапная потоковая модель с явным управлением буфером.

#### Вариант А: Использование Потока в памяти (Для средних пакетов до 50-100 МБ)
Данные генерируются напрямую в RAM-буфер, а затем сжимаются архиватором при записи.

*✅ Правильно и синтаксически легально:*
```bsl
// Шаг 1: Создаем сырой поток в оперативной памяти сервера
ПотокДанных = Новый ПотокВПамяти;

// Шаг 2: Открываем ЗаписьJSON прямо в этот поток
ЗаписьJSON = Новый ЗаписьJSON;
ЗаписьJSON.ОткрытьПоток(ПотокДанных); 

// Шаг 3: Потоковая токенная запись данных
ЗаписьJSON.ЗаписатьНачалоОбъекта();
ЗаписьJSON.ЗаписатьИмяСвойства("items");
ЗаписьJSON.ЗаписатьНачалоМассива();

Пока Выборка.Следующий() Цикл
	ЗаписьJSON.ЗаписатьНачалоОбъекта();
	ЗаписьJSON.ЗаписатьИмяСвойства("id");
	ЗаписьJSON.ЗаписатьЗначение(Выборка.Код);
	ЗаписьJSON.ЗаписатьКонецОбъекта();
КонецЦикла;

ЗаписьJSON.ЗаписатьКонецМассива();
ЗаписьJSON.ЗаписатьКонецОбъекта();

// Важно: закрываем JSON, чтобы платформа принудительно сбросила остатки буферов в поток
ЗаписьJSON.Закрыть(); 

// Шаг 4: Перематываем поток в начало, чтобы архиватор смог начать чтение байт
ПотокДанных.Перейти(0, ПозицияВПотоке.Начало); 

// Шаг 5: Инициализируем архив на диске
ЗаписьАрхива = Новый ЗаписьZipФайла(ИмяФайлаАрхива);

// Шаг 6: Передаем поток в архиватор (сжатие произойдет на лету при вызове Записать)
ЗаписьАрхива.Добавить(ПотокДанных, "data.json"); 
ЗаписьАрхива.Записать();

// Очищаем ресурсы
ПотокДанных.Закрыть();
```
* ⚠️ **Highload-предупреждение:** Этот метод полностью убирает нагрузку на дисковый I/O при создании JSON, но **утилизирует оперативную память сервера**. Весь объем сгенерированного текста будет полностью занимать RAM процесса `rphost` до момента завершения архивации. При экстремальных объемах данных (сотни мегабайт/гигабайты) метод вызовет Out of Memory.

#### Вариант Б: Использование Файлового потока (Для экстремальных Highload-выгрузок от 500 МБ)
Если утилизация RAM недопустима, `ПотокВПамяти` заменяется на `ФайловыйПоток`.

* **Физика процесса:** Код остается идентичным Варианту А, но на Шаге 1 инициализируется файловый стрим: `ПотокДанных = Новый ФайловыйПоток(ПутьКПромежуточномуФайлу, РежимОткрытияФайла.Создать)`.
* **Результат:** `ЗаписьJSON` сбрасывает данные на диск мелкими квантами по мере заполнения внутреннего буфера. RAM сервера свободен (~0 МБ). Затем архиватор считывает этот файл также порциями и жмет в ZIP. Да, здесь появляется дисковый оверхед, но система гарантированно стабильна и защищена от падения по памяти при любых объемах данных.


### 4.3. Использование внешних компонент (Native API — C++ / Rust)
Штатные механизмы ZIP в 1С работают синхронно и ограничены алгоритмами платформы. Для экстремального Highload (например, передача сотен тысяч пакетов в секунду по HTTP REST) стандартного ZIP недостаточно из-за высоких накладных расходов процессора.

* **Архитектурный подход:** Применение внешних компонент на C++ или Rust (через Native API), инкапсулирующих сверхбыстрые библиотеки сжатия **`zlib`**, **`lz4`** или **`zstd`**.
* **Механика работы:** 
  1. Код 1С формирует текстовые или бинарные пакеты небольшими квантами данных (чанками) в оперативной памяти `rphost`.
  2. Чанк памяти передается во внешнюю компоненту.
  3. Компонента асинхронно, используя отдельные потоки процессора (Multithreading), сжимает этот чанк по алгоритму `lz4` (самый быстрый алгоритм сжатия в мире с минимальной нагрузкой на CPU) и возвращает упакованный массив байт обратно для мгновенной отправки в сеть.
  4. Это исключает блокировки основного рабочего процесса `rphost` и позволяет утилизировать все ядра серверного процессора.

## 5. Highload-стратегии мирового уровня (Beyond Standard 1С)

Когда интенсивность обменов превышает тысячи RPS, текстовая сериализация (даже потоковая) становится главным потребителем серверных мощностей CPU и сетевой полосы. Архитектор обязан выходить за рамки стандартного стека 1С.

### 5.1. Смена текстовых форматов на бинарные (Protobuf / MessagePack)
**Суть стратегии:** Текстовые форматы (JSON, XML) неэффективны по своей природе — они заставляют СУБД и rphost парсить строки, тратя миллиарды тактов процессора на конвертацию текста в типы данных (числа, даты) и обратно. В критических и сверхвысоконагруженных стыках (например, 1С $\leftrightarrow$ внешний микросервис программы лояльности или процессинга на Go/Rust) текстовые форматы заменяются на бинарные.

* **Protocol Buffers (Protobuf) от Google:** Схема данных компилируется в бинарный код. Вместо имен полей (вроде `"partner_guid"`) передаются жестко типизированные числовые теги (индексы).
* **MessagePack:** По сути, тот же JSON, но упакованный в компактный бинарный формат «на лету».
* **Результат для Highload:** Экономия до 50–70% объема сетевого трафика. Скорость десериализации на стороне внешних сервисов возрастает в 5–8 раз, полностью разгружая CPU серверов приложений.
* **Реализация в 1С:** Поскольку встроенный язык 1С не поддерживает бинарные протоколы обмена «из коробки», для работы с Protobuf/MessagePack пишется внешняя компонента (Native API) на C++ или Rust, которая принимает от 1С структуру/массив, пакует её в бинарный поток и сама выстреливает в сеть через gRPC/TCP.

### 5.2. NDJSON (Newline Delimited JSON / JSON Lines)
**Суть стратегии:** Стандартный JSON-файл представляет собой единый монолитный документ, обернутый в корневые квадратные или фигурные скобки `[ ... ]`. Чтобы прочитать его конец, парсер обязан отслеживать состояние всего документа. Формат **NDJSON (или JSONL)** убирает корневые скобки. В нем каждая отдельная строка файла — это абсолютно независимый, валидный и изолированный JSON-объект, отделенный символом переноса строки (`\n`).

* **Почему это эталон для логов и очередей:** 
  1. Файл можно читать самым примитивным и быстрым текстовым стримом (`ТекстовыйДокумент` или `ЧтениеТекста`) построчно.
  2. Строку можно мгновенно скормить в метод `ПрочитатьJSON()` без прокрутки токенов и поиска начала массива.
  3. Файл устойчив к сбоям: если при записи терабайтного лога посередине моргнет питание, NDJSON-файл не повредится целиком (в отличие от обычного JSON, где незакрытая скобка `]` в конце ломает весь файл). Все строки до момента сбоя останутся валидными и читаемыми.

### 5.3. Zero-Copy Архитектура
**Суть стратегии:** В классической архитектуре обмена данные постоянно копируются в памяти: rphost выгрузил данные из СУБД в ТаблицуЗначений $\rightarrow$ скопировал их в Структуру $\rightarrow$ скопировал в буфер сериализатора $\rightarrow$ скопировал в сетевой буфер ОС. Каждое копирование гигабайтного массива в памяти нагружает шину данных и процессор. Паттерн **Zero-Copy** исключает дублирование данных в оперативной памяти.

* **Физика процесса:** Вместо физического перемещения и копирования массивов байт между процессами, система передает лишь низкоуровневый указатель (Pointer) на конкретный адрес в оперативной памяти, где этот массив уже лежит.
* **Пример в экосистеме 1С:** Взаимодействие 1С с драйверами торгового оборудования или высокоскоростными очередями сообщений через внешние компоненты Native API. Вместо передачи тяжелых строк или двоичных данных через интерфейс платформы (что вызывает внутреннее копирование в памяти rphost), 1С выделяет область памяти, а внешняя компонента работает с ней напрямую через С-указатели. Это позволяет гонять терабайты данных между 1С и сетевой картой с нулевым оверхедом на копирование.
