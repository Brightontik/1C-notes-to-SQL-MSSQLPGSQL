# Этап 1: Философия БСП и архитектурный минимум

Библиотека Стандартных Подсистем (БСП) — это сервисно-ориентированный каркас (Framework), навязывающий жесткие стандарты производительности. Главная цель БСП в высоконагруженных системах — свести архитектурную сложность управления платформой к $O(1)$ для прикладного разработчика.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП


| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Вычисление прав доступа** | Проверка «в лоб» через циклы или частые запросы к СУБД (O(N)). | Кеширование в параметрах сеанса при старте системы (O(1)). |
| **Инициализация при старте** | Каждый модуль конфигурации дергает сервер БД своими запросами. | Единая транзакция инициализации через `ПараметрыРаботыПрограммы`. |
| **Обновление настроек кластера** | Требуется перезапуск сеансов пользователей для сброса кеша. | Инвалидация кеша на лету через регистрацию изменений параметров. |
| **Сложность поддержки кода** | Растет экспоненциально (O(2^N)) с ростом количества подсистем. | Линейная зависимость (O(1)) за счет слабой связанности (Low Coupling). |

---

## 🗺️ Схема взаимодействия подсистем: Слабая связанность

В БСП подсистемы никогда не вызывают друг друга напрямую "жестким кодом". Связь идет через переопределяемые модули и механизм событий. Это защищает систему от каскадных падений.

### Архитектурная реализация слабой связанности

Чтобы подсистемы не зависели друг от друга напрямую, БСП использует строгое разделение логики на три уровня взаимодействия.


| Компонент БСП | Роль в архитектуре | Влияние на производительность и отказоустойчивость |
| :--- | :--- | :--- |
| **Основной API** *(ОбщегоНазначения)* | Точка входа для прикладного кода. Логика инкапсулирована внутри общего модуля. | **Высокое.** Предотвращает дублирование кода. Оптимизирует вызовы функций за счет централизованных алгоритмов. |
| **Переопределяемые модули** | Интерфейс расширения (точки интеграции). Разработчик пишет здесь свой код, не меняя ядро БСП. | **Безопасное.** Исключает каскадные ошибки при обновлении: если одна подсистема падает, ядро продолжает стабильно работать. |
| **Регистрация в кеше БД** | Фиксация конфигурации подсистем в базе данных при первом запуске (обновлении ИБ). | **Оптимальное.** Сложность проверки доступности фичи снижается до **O(1)** благодаря чтению из параметров работы программы в RAM. |

## ⚡ Оптимизация через Модули с Повторным Использованием (ПВЗ)

Один из главных механизмов БСП для снижения сложности с $O(N)$ (где $N$ — количество вызовов) до $O(1)$ (однократное вычисление) — это повторное использование возвращаемых значений.

### 1. Наивный подход (Постоянный спам СУБД)
Разработчик вызывает функцию чтения настроек внутри цикла обработки строк документа.

```bsl
// Сложность: O(N) по вызовам к СУБД. На 10 000 строк документа будет 10 000 SELECT-запросов.
Для Каждого Строка Из ТЧДанных Цикл
    Если ЗначениеВыбраннойОпции("ИспользоватьИнтеграцию") Тогда // ❌ Каждый раз летит запрос в БД
        ОбработатьСтроку(Строка);
    КонецЕсли;
КонецЦикла;
```

### 2. Подход БСП (Кеширование в RAM)
БСП выносит такие проверки в специальные модули. Сервер 1С вычисляет значение один раз, сохраняет в RAM, а при повторных вызовах мгновенно отдает из памяти.

```bsl
// Сложность: O(1). Первый вызов — SELECT к БД, остальные 9 999 вызовов — чтение из RAM сервера.
Для Каждого Строка Из ТЧДанных Цикл
    // БСП использует вызов модуля ПовтИсп
    Если СтандартныеПодсистемыПовтИсп.ПараметрыРаботыПрограммы().ИспользоватьИнтеграцию Тогда 
        ОбработатьСтроку(Строка);
    КонецЕсли;
КонецЦикла;
```

---

## 📌 Памятка архитектора: Три правила БСП

1. **Никаких прямых запросов к системным константам в циклах.** Всегда ищи аналог в модулях `...ПовтИсп`.
2. **Интеграция только через Переопределяемые модули.** Не изменяй типовые модули менеджеров, используй предназначенные для этого точки расширения БСП.
3. **Экономия памяти (RAM).** Модули с повторным использованием на время сеанса не должны хранить тяжелые таблицы значений. Только примитивные типы, структуры или ссылки, иначе память сервера будет сожрана ($O(M)$ по памяти, где $M$ — размер данных сеанса).

# Этап 2: Асинхронность и подсистема «Длительные операции»

Основная задача подсистемы — исключить монопольное удержание серверного потока клиентским сеансом при выполнении тяжелых задач. БСП переводит синхронное ожидание выполнения в асинхронный мониторинг, снижая нагрузку на интерфейс до **O(1)**.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП

Когда разработчик пытается реализовать длительный расчет «на коленке», система сталкивается с неконтролируемым ростом сеансов и утечками памяти.


| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Интерфейс пользователя** | Блокируется наглухо до конца расчета. Пользователь думает, что программа «зависла». | Свободен. Отображается немодальное окно прогресса. Можно продолжать работу. |
| **Сложность по памяти** | Передача больших массивов через параметры формы дублирует данные в RAM (**O(N)**). | Передача через Временное Хранилище с изоляцией контекста (**O(1)** для сеанса). |
| **Контроль таймаутов** | Отсутствует. Задание может висеть вечно, блокируя таблицы БД. | Жесткие лимиты времени. Автоматическое завершение «зависших» потоков. |
| **Опрос статуса** | Постоянный спам сервера короткими вызовами. Деградация производительности. | Оптимальный интервал опроса с адаптивным увеличением шага. |

---

## 🗺️ Схема асинхронного выполнения: Разделение потоков

БСП полностью изолирует тяжелый расчет от основного сеанса пользователя. Взаимодействие происходит через менеджер фоновых заданий платформы и Временное Хранилище.

```text
+-----------------------+                    +-------------------------+

|   Клиентский поток    |--1. Запуск за O(1)->|    Фоновое задание      |
| (Интерфейс свободен)  |                    |  (Изолированный поток)  |
+-----------------------+                    +-------------------------+

            |                                             |
   3. Опрос / Чтение                               2. Помещение итога

            |                                             |
            v                                             v
+----------------------------------------------------------------------+

|                         Временное Хранилище                          |
|                           (Обмен данными)                            |
+----------------------------------------------------------------------+
```

---

## ⚡ Оптимизация передачи данных через Временное Хранилище

Если нужно передать на расчет 100 000 строк данных (например, массив структур), передача их "в лоб" как параметр процедуры приведет к сериализации и раздуванию памяти сервера приложений. Правильный путь — использование Временного Хранилища.

### 1. Наивный подход (Перегрузка RAM сеанса)
Передача тяжелой коллекции напрямую в параметры серверного вызова.

```bsl
// Сложность по памяти: O(N). Коллекция копируется в памяти между контекстами.
&НаСервере
Процедура ВыполнитьТяжелыйРасчетНапрямую(МассивДанных)
    // Сервер выделит память под копию массива для этого сеанса
    Для Каждого Элемент Из МассивДанных Цикл
        // Обработка...
    КонецЦикла;
КонецПроцедуры
```

### 2. Подход БСП (Изоляция через Хранилище)
Данные упаковываются во Временное Хранилище, фоновое задание считывает их, обрабатывает и уничтожает за собой контекст.

```bsl
// Сложность по памяти для основного сеанса: O(1). Память освобождается мгновенно.
&НаСервере
Функция ЗапуститьАсинхронныйРасчет(МассивДанных)
    
    // Помещаем данные в хранилище, привязывая к УИД формы
    АдресПараметров = ПоместитьВоВременноеХранилище(МассивДанных, УникальныйИдентификатор);
    
    ПараметрыПроцедуры = Новый Структура;
    ПараметрыПроцедуры.Вставить("АдресПараметров", АдресПараметров);
    
    ПараметрыВыполнения = ДлительныеОперации.ПараметрыВыполненияВФоне(УникальныйИдентификатор);
    ПараметрыВыполнения.НаименованиеЗадачи = "Расчет тяжелых метрик";
    
    // Запуск фонового задания средствами БСП
    Возврат ДлительныеОперации.ЗапуститьВыполнениеВФоне(
        "МойЭкспортныйМодуль.ВыполнитьРасчетВФоне", 
        ПараметрыПроцедуры, 
        ПараметрыВыполнения
    );
КонецФункции

// Код в экспортном серверном модуле
Процедура ВыполнитьРасчетВФоне(ПараметрыПроцедуры, АдресРезультата) Экспорт
    // Получаем данные внутри изолированного фонового задания
    ИсходныеДанные = ПолучитьИзВременногоХранилища(ПараметрыПроцедуры.АдресПараметров);
    
    РезультатРасчета = Новый Массив; // Сюда собираем итог
    
    // ... логика расчета ...
    
    // Возвращаем результат через специализированный адрес БСП
    ПоместитьВоВременноеХранилище(РезультатРасчета, AddressРезультата);
КонецПроцедуры
```

---

## 📌 Памятка архитектора по Длительным Операциям

1. **Экспортные процедуры.** Процедура, передаваемая в `ЗапуститьВыполнениеВФоне`, должна находиться только в общем модуле с флагом "Сервер". Она обязательно должна быть `Экспорт`.
2. **Никакого интерактивного кода.** Внутри фонового задания вызов `Предупреждение()`, `Вопрос()` или обращение к параметрам сеанса, завязанным на клиента, приведет к падению потока.
3. **Очистка памяти.** После завершения расчета и прочтения данных из `АдресРезультата`, принудительно очищай временное хранилище, если расчет вызывался вне контекста формы, чтобы не держать RAM до закрытия сеанса.


# Этап 3: Пакетная обработка данных и Очередь заданий

При обработке больших массивов данных (миллионы записей) последовательный перебор приводит к росту транзакций во времени, вызывая каскадные таймауты и блокировки таблиц СУБД. БСП позволяет распараллелить эту задачу, снижая временную сложность с **O(N)** до **O(N/Потоки)**.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП



| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Алгоритм деления** | Перебор всей таблицы в одной транзакции или выборка по `Выбрать(Верхние)`. | Пакетное разделение по границам уникальных идентификаторов или кодов. |
| **Параллелизм** | Запуск бесконтрольных фоновых заданий, забивающих rphost до отказа. | Ограничение пула потоков под доступные ресурсы процессора. |
| **Сбой потока** | При падении одного потока вся пакетная обработка прерывается или зависает. | Автоматический перезапуск упавшего задания из очереди (до N попыток). |
| **Нагрузка на СУБД** | Монопольные блокировки, приводящие к каскадному коллапсу очереди пользователей. | Изолированные транзакции порций данных с паузами между пакетами. |

---

## 🗺️ Схема многопоточной пакетной обработки

БСП управляет распределением задач через Очередь заданий, координируя пул параллельных рабочих потоков.

```text
       +---------------------------------------------+

       |   Главный поток (Диспетчер БСП)             |
       |   - Разбивает данные на пакеты по 1000 шт.  |
       +---------------------------------------------+

                              |
                              v
       +---------------------------------------------+
       |   Подсистема БСП "Очередь заданий"          |
       |   - Нарезает задачи и квотирует потоки      |
       +---------------------------------------------+

               |              |              |
         +-----+              +-----+        +-----+

         |                          |              |
         v                          v              v
+-----------------+        +-----------------+        +-----------------+

| Рабочий Поток 1 |        | Рабочий Поток 2 |        | Рабочий Поток N |
| Транзакция O(K) |        | Транзакция O(K) |        | Транзакция O(K) |
+-----------------+        +-----------------+        +-----------------+
```

---

## ⚡ Оптимизация: Переход к параллельным пакетам O(N/Потоки)

Представим задачу: нужно актуализировать регистр сведений на 1 000 000 записей. 

### 1. Наивный подход (Монопольный захват таблицы)
Разработчик запускает один тяжелый цикл, который держит транзакцию открытой минуты или часы.

```bsl
// Сложность: O(N) в одной транзакции. Гарантированный Deadlock для других пользователей.
&НаСервере
Процедура АктуализироватьДанныеЗапросом()
    Выборка = ПоследниеДанныеДляМодификации(); // Возвращает 1 000 000 строк
    
    НачатьТранзакцию();
    Пока Выборка.Следующий() Цикл
        // Модификация данных в базе
        ЗаписатьИзмененияВРегистр(Выборка.Ссылка); 
    КонецЦикла;
    ЗафиксироватьТранзакцию(); // Блокировка держится до самого конца
КонецПроцедуры
```

### 2. Подход БСП (Пакетный параллелизм через Очередь)
Разработчик делит массив на порции (размер пакета `K = 1000`), и каждая порция обрабатывается изолированно в Очереди заданий.

```bsl
// Сложность: O(N/Потоки). Время удержания блокировки СУБД снижено до микросекунд (размер пакета K).
&НаСервере
Процедура ЗапуститьПакетнуюОбработкуБСП(МассивИдентификаторов)
    
    РазмерПакета = 1000;
    Индекс = 0;
    
    Пока Индекс < МассивИдентификаторов.Количество() Цикл
        
        // Формируем порцию данных для одного задания
        Пакет = Новый Массив;
        Для Шаг = 1 По РазмерПакета Цикл
            Если Индекс >= МассивИдентификаторов.Количество() Тогда Прервать; КонецЕсли;
            Пакет.Добавить(МассивИдентификаторов[Индекс]);
            Индекс = Индекс + 1;
        КонецЦикла;
        
        // Помещаем пакет во Временное Хранилище
        АдресПакета = ПоместитьВоВременноеХранилище(Пакет);
        
        // Регистрируем задачу в Очереди заданий БСП
        ПараметрыЗадания = Новый Структура;
        ПараметрыЗадания.Вставить("АдресПакета", АдресПакета);
        
        ОчередьЗаданий.ДобавитьЗадание(
            "МойЭкспортныйМодуль.ОбработатьПакетДанных", 
            ПараметрыЗадания, 
            3, // Количество попыток при сбое
            "Пакетная обработка данных регистра"
        );
        
    КонецЦикла;
КонецПроцедуры

// Процедура-исполнитель в общем модуле
Процедура ОбработатьПакетДанных(ПараметрыЗадания) Экспорт
    ПорцияДанных = ПолучитьИзВременногоХранилища(ПараметрыЗадания.АдресПакета);
    
    // Каждая порция выполняется в своей короткой транзакции
    НачатьТранзакцию();
    Для Каждого Ссылка Из ПорцияДанных Цикл
        ЗаписатьИзмененияВРегистр(Ссылка);
    КонецЦикла;
    ЗафиксироватьТранзакцию();
КонецПроцедуры
```

---

## 📌 Памятка архитектора по Очереди заданий

1. **Размер пакета (K).** Оптимальный размер пакета для транзакции в высоконагруженных системах — от 1000 до 5000 записей. Меньше — вырастут накладные расходы на запуск потоков, больше — начнутся блокировки.
2. **Управление количеством потоков.** Не создавай заданий больше, чем ядер процессора на сервере приложений, минус резерв на работу пользователей. БСП позволяет регулировать лимиты через настройки очередей.
3. **Идемпотентность кода.** Процедура обработчика пакета должна быть написана так, чтобы при повторном запуске (после аварийного падения потока) она не дублировала данные и не ломала логику.
