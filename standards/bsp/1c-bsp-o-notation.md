# Этап 1: Философия БСП и архитектурный минимум

Библиотека Стандартных Подсистем (БСП) — это сервисно-ориентированный каркас (Framework), навязывающий жесткие стандарты производительности. Главная цель БСП в высоконагруженных системах — свести архитектурную сложность управления платформой к $O(1)$ для прикладного разработчика.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП


| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Вычисление прав доступа** | Проверка «в лоб» через циклы или частые запросы к СУБД (O(N)). | Кеширование в параметрах сеанса при старте системы (O(1)). |
| **Инициализация при старте** | Каждый модуль конфигурации дергает сервер БД своими запросами. | Единая транзакция инициализации через `ПараметрыРаботыПрограммы`. |
| **Обновление настроек кластера** | Требуется перезапуск сеансов пользователей для сброса кеша. | Инвалидация кеша на лету через регистрацию изменений параметров. |
| **Сложность поддержки кода** | Растет экспоненциально (O(2^N)) с ростом количества подсистем. | Линейная зависимость (O(1)) за счет слабой связанности (Low Coupling). |

---

## 🗺️ Схема взаимодействия подсистем: Слабая связанность

В БСП подсистемы никогда не вызывают друг друга напрямую "жестким кодом". Связь идет через переопределяемые модули и механизм событий. Это защищает систему от каскадных падений.

### Архитектурная реализация слабой связанности

Чтобы подсистемы не зависели друг от друга напрямую, БСП использует строгое разделение логики на три уровня взаимодействия.


| Компонент БСП | Роль в архитектуре | Влияние на производительность и отказоустойчивость |
| :--- | :--- | :--- |
| **Основной API** *(ОбщегоНазначения)* | Точка входа для прикладного кода. Логика инкапсулирована внутри общего модуля. | **Высокое.** Предотвращает дублирование кода. Оптимизирует вызовы функций за счет централизованных алгоритмов. |
| **Переопределяемые модули** | Интерфейс расширения (точки интеграции). Разработчик пишет здесь свой код, не меняя ядро БСП. | **Безопасное.** Исключает каскадные ошибки при обновлении: если одна подсистема падает, ядро продолжает стабильно работать. |
| **Регистрация в кеше БД** | Фиксация конфигурации подсистем в базе данных при первом запуске (обновлении ИБ). | **Оптимальное.** Сложность проверки доступности фичи снижается до **O(1)** благодаря чтению из параметров работы программы в RAM. |

## ⚡ Оптимизация через Модули с Повторным Использованием (ПВЗ)

Один из главных механизмов БСП для снижения сложности с $O(N)$ (где $N$ — количество вызовов) до $O(1)$ (однократное вычисление) — это повторное использование возвращаемых значений.

### 1. Наивный подход (Постоянный спам СУБД)
Разработчик вызывает функцию чтения настроек внутри цикла обработки строк документа.

```bsl
// Сложность: O(N) по вызовам к СУБД. На 10 000 строк документа будет 10 000 SELECT-запросов.
Для Каждого Строка Из ТЧДанных Цикл
    Если ЗначениеВыбраннойОпции("ИспользоватьИнтеграцию") Тогда // ❌ Каждый раз летит запрос в БД
        ОбработатьСтроку(Строка);
    КонецЕсли;
КонецЦикла;
```

### 2. Подход БСП (Кеширование в RAM)
БСП выносит такие проверки в специальные модули. Сервер 1С вычисляет значение один раз, сохраняет в RAM, а при повторных вызовах мгновенно отдает из памяти.

```bsl
// Сложность: O(1). Первый вызов — SELECT к БД, остальные 9 999 вызовов — чтение из RAM сервера.
Для Каждого Строка Из ТЧДанных Цикл
    // БСП использует вызов модуля ПовтИсп
    Если СтандартныеПодсистемыПовтИсп.ПараметрыРаботыПрограммы().ИспользоватьИнтеграцию Тогда 
        ОбработатьСтроку(Строка);
    КонецЕсли;
КонецЦикла;
```

---

## 📌 Памятка архитектора: Три правила БСП

1. **Никаких прямых запросов к системным константам в циклах.** Всегда ищи аналог в модулях `...ПовтИсп`.
2. **Интеграция только через Переопределяемые модули.** Не изменяй типовые модули менеджеров, используй предназначенные для этого точки расширения БСП.
3. **Экономия памяти (RAM).** Модули с повторным использованием на время сеанса не должны хранить тяжелые таблицы значений. Только примитивные типы, структуры или ссылки, иначе память сервера будет сожрана ($O(M)$ по памяти, где $M$ — размер данных сеанса).

# Этап 2: Асинхронность и подсистема «Длительные операции»

Основная задача подсистемы — исключить монопольное удержание серверного потока клиентским сеансом при выполнении тяжелых задач. БСП переводит синхронное ожидание выполнения в асинхронный мониторинг, снижая нагрузку на интерфейс до **O(1)**.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП

Когда разработчик пытается реализовать длительный расчет «на коленке», система сталкивается с неконтролируемым ростом сеансов и утечками памяти.


| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Интерфейс пользователя** | Блокируется наглухо до конца расчета. Пользователь думает, что программа «зависла». | Свободен. Отображается немодальное окно прогресса. Можно продолжать работу. |
| **Сложность по памяти** | Передача больших массивов через параметры формы дублирует данные в RAM (**O(N)**). | Передача через Временное Хранилище с изоляцией контекста (**O(1)** для сеанса). |
| **Контроль таймаутов** | Отсутствует. Задание может висеть вечно, блокируя таблицы БД. | Жесткие лимиты времени. Автоматическое завершение «зависших» потоков. |
| **Опрос статуса** | Постоянный спам сервера короткими вызовами. Деградация производительности. | Оптимальный интервал опроса с адаптивным увеличением шага. |

---

## 🗺️ Схема асинхронного выполнения: Разделение потоков

БСП полностью изолирует тяжелый расчет от основного сеанса пользователя. Взаимодействие происходит через менеджер фоновых заданий платформы и Временное Хранилище.

```text
+-----------------------+                    +-------------------------+

|   Клиентский поток    |--1. Запуск за O(1)->|    Фоновое задание      |
| (Интерфейс свободен)  |                    |  (Изолированный поток)  |
+-----------------------+                    +-------------------------+

            |                                             |
   3. Опрос / Чтение                               2. Помещение итога

            |                                             |
            v                                             v
+----------------------------------------------------------------------+

|                         Временное Хранилище                          |
|                           (Обмен данными)                            |
+----------------------------------------------------------------------+
```

---

## ⚡ Оптимизация передачи данных через Временное Хранилище

Если нужно передать на расчет 100 000 строк данных (например, массив структур), передача их "в лоб" как параметр процедуры приведет к сериализации и раздуванию памяти сервера приложений. Правильный путь — использование Временного Хранилища.

### 1. Наивный подход (Перегрузка RAM сеанса)
Передача тяжелой коллекции напрямую в параметры серверного вызова.

```bsl
// Сложность по памяти: O(N). Коллекция копируется в памяти между контекстами.
&НаСервере
Процедура ВыполнитьТяжелыйРасчетНапрямую(МассивДанных)
    // Сервер выделит память под копию массива для этого сеанса
    Для Каждого Элемент Из МассивДанных Цикл
        // Обработка...
    КонецЦикла;
КонецПроцедуры
```

### 2. Подход БСП (Изоляция через Хранилище)
Данные упаковываются во Временное Хранилище, фоновое задание считывает их, обрабатывает и уничтожает за собой контекст.

```bsl
// Сложность по памяти для основного сеанса: O(1). Память освобождается мгновенно.
&НаСервере
Функция ЗапуститьАсинхронныйРасчет(МассивДанных)
    
    // Помещаем данные в хранилище, привязывая к УИД формы
    АдресПараметров = ПоместитьВоВременноеХранилище(МассивДанных, УникальныйИдентификатор);
    
    ПараметрыПроцедуры = Новый Структура;
    ПараметрыПроцедуры.Вставить("АдресПараметров", АдресПараметров);
    
    ПараметрыВыполнения = ДлительныеОперации.ПараметрыВыполненияВФоне(УникальныйИдентификатор);
    ПараметрыВыполнения.НаименованиеЗадачи = "Расчет тяжелых метрик";
    
    // Запуск фонового задания средствами БСП
    Возврат ДлительныеОперации.ЗапуститьВыполнениеВФоне(
        "МойЭкспортныйМодуль.ВыполнитьРасчетВФоне", 
        ПараметрыПроцедуры, 
        ПараметрыВыполнения
    );
КонецФункции

// Код в экспортном серверном модуле
Процедура ВыполнитьРасчетВФоне(ПараметрыПроцедуры, АдресРезультата) Экспорт
    // Получаем данные внутри изолированного фонового задания
    ИсходныеДанные = ПолучитьИзВременногоХранилища(ПараметрыПроцедуры.АдресПараметров);
    
    РезультатРасчета = Новый Массив; // Сюда собираем итог
    
    // ... логика расчета ...
    
    // Возвращаем результат через специализированный адрес БСП
    ПоместитьВоВременноеХранилище(РезультатРасчета, AddressРезультата);
КонецПроцедуры
```

---

## 📌 Памятка архитектора по Длительным Операциям

1. **Экспортные процедуры.** Процедура, передаваемая в `ЗапуститьВыполнениеВФоне`, должна находиться только в общем модуле с флагом "Сервер". Она обязательно должна быть `Экспорт`.
2. **Никакого интерактивного кода.** Внутри фонового задания вызов `Предупреждение()`, `Вопрос()` или обращение к параметрам сеанса, завязанным на клиента, приведет к падению потока.
3. **Очистка памяти.** После завершения расчета и прочтения данных из `АдресРезультата`, принудительно очищай временное хранилище, если расчет вызывался вне контекста формы, чтобы не держать RAM до закрытия сеанса.


# Этап 3: Пакетная обработка данных и Очередь заданий

При обработке больших массивов данных (миллионы записей) последовательный перебор приводит к росту транзакций во времени, вызывая каскадные таймауты и блокировки таблиц СУБД. БСП позволяет распараллелить эту задачу, снижая временную сложность с **O(N)** до **O(N/Потоки)**.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП



| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Алгоритм деления** | Перебор всей таблицы в одной транзакции или выборка по `Выбрать(Верхние)`. | Пакетное разделение по границам уникальных идентификаторов или кодов. |
| **Параллелизм** | Запуск бесконтрольных фоновых заданий, забивающих rphost до отказа. | Ограничение пула потоков под доступные ресурсы процессора. |
| **Сбой потока** | При падении одного потока вся пакетная обработка прерывается или зависает. | Автоматический перезапуск упавшего задания из очереди (до N попыток). |
| **Нагрузка на СУБД** | Монопольные блокировки, приводящие к каскадному коллапсу очереди пользователей. | Изолированные транзакции порций данных с паузами между пакетами. |

---

## 🗺️ Схема многопоточной пакетной обработки

БСП управляет распределением задач через Очередь заданий, координируя пул параллельных рабочих потоков.

```text
       +---------------------------------------------+

       |   Главный поток (Диспетчер БСП)             |
       |   - Разбивает данные на пакеты по 1000 шт.  |
       +---------------------------------------------+

                              |
                              v
       +---------------------------------------------+
       |   Подсистема БСП "Очередь заданий"          |
       |   - Нарезает задачи и квотирует потоки      |
       +---------------------------------------------+

               |              |              |
         +-----+              +-----+        +-----+

         |                          |              |
         v                          v              v
+-----------------+        +-----------------+        +-----------------+

| Рабочий Поток 1 |        | Рабочий Поток 2 |        | Рабочий Поток N |
| Транзакция O(K) |        | Транзакция O(K) |        | Транзакция O(K) |
+-----------------+        +-----------------+        +-----------------+
```

---

## ⚡ Оптимизация: Переход к параллельным пакетам O(N/Потоки)

Представим задачу: нужно актуализировать регистр сведений на 1 000 000 записей. 

### 1. Наивный подход (Монопольный захват таблицы)
Разработчик запускает один тяжелый цикл, который держит транзакцию открытой минуты или часы.

```bsl
// Сложность: O(N) в одной транзакции. Гарантированный Deadlock для других пользователей.
&НаСервере
Процедура АктуализироватьДанныеЗапросом()
    Выборка = ПоследниеДанныеДляМодификации(); // Возвращает 1 000 000 строк
    
    НачатьТранзакцию();
    Пока Выборка.Следующий() Цикл
        // Модификация данных в базе
        ЗаписатьИзмененияВРегистр(Выборка.Ссылка); 
    КонецЦикла;
    ЗафиксироватьТранзакцию(); // Блокировка держится до самого конца
КонецПроцедуры
```

### 2. Подход БСП (Пакетный параллелизм через Очередь)
Разработчик делит массив на порции (размер пакета `K = 1000`), и каждая порция обрабатывается изолированно в Очереди заданий.

```bsl
// Сложность: O(N/Потоки). Время удержания блокировки СУБД снижено до микросекунд (размер пакета K).
&НаСервере
Процедура ЗапуститьПакетнуюОбработкуБСП(МассивИдентификаторов)
    
    РазмерПакета = 1000;
    Индекс = 0;
    
    Пока Индекс < МассивИдентификаторов.Количество() Цикл
        
        // Формируем порцию данных для одного задания
        Пакет = Новый Массив;
        Для Шаг = 1 По РазмерПакета Цикл
            Если Индекс >= МассивИдентификаторов.Количество() Тогда Прервать; КонецЕсли;
            Пакет.Добавить(МассивИдентификаторов[Индекс]);
            Индекс = Индекс + 1;
        КонецЦикла;
        
        // Помещаем пакет во Временное Хранилище
        АдресПакета = ПоместитьВоВременноеХранилище(Пакет);
        
        // Регистрируем задачу в Очереди заданий БСП
        ПараметрыЗадания = Новый Структура;
        ПараметрыЗадания.Вставить("АдресПакета", АдресПакета);
        
        ОчередьЗаданий.ДобавитьЗадание(
            "МойЭкспортныйМодуль.ОбработатьПакетДанных", 
            ПараметрыЗадания, 
            3, // Количество попыток при сбое
            "Пакетная обработка данных регистра"
        );
        
    КонецЦикла;
КонецПроцедуры

// Процедура-исполнитель в общем модуле
Процедура ОбработатьПакетДанных(ПараметрыЗадания) Экспорт
    ПорцияДанных = ПолучитьИзВременногоХранилища(ПараметрыЗадания.АдресПакета);
    
    // Каждая порция выполняется в своей короткой транзакции
    НачатьТранзакцию();
    Для Каждого Ссылка Из ПорцияДанных Цикл
        ЗаписатьИзмененияВРегистр(Ссылка);
    КонецЦикла;
    ЗафиксироватьТранзакцию();
КонецПроцедуры
```

---

## 📌 Памятка архитектора по Очереди заданий

1. **Размер пакета (K).** Оптимальный размер пакета для транзакции в высоконагруженных системах — от 1000 до 5000 записей. Меньше — вырастут накладные расходы на запуск потоков, больше — начнутся блокировки.
2. **Управление количеством потоков.** Не создавай заданий больше, чем ядер процессора на сервере приложений, минус резерв на работу пользователей. БСП позволяет регулировать лимиты через настройки очередей.
3. **Идемпотентность кода.** Процедура обработчика пакета должна быть написана так, чтобы при повторном запуске (после аварийного падения потока) она не дублировала данные и не ломала логику.


# Этап 4: Кеширование данных и Параметры работы программы

При росте числа одновременных пользователей (от 500+ сеансов) постоянное чтение структуры метаданных, функциональных опций и глобальных настроек напрямую из СУБД создает критическую нагрузку на дисковую подсистему. БСП решает эту проблему через агрегированное кеширование данных в RAM сервера приложений.

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП



| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Локация хранения** | Постоянные повторные обращения к СУБД или константам (**O(N)**). | Оперативная память сервера приложений (**O(1)**). |
| **Синхронизация в кластере** | Настройки, измененные в одном сеансе, не видны в других без перезапуска. | Автоматическая инвалидация кеша во всех рабочих процессах (rphost) кластера. |
| **Объем структуры** | Чтение настроек по одной переменной, размазанное по коду. | Единый монолитный кеш параметров, собираемый за одну транзакцию. |
| **Надежность при старте** | Каждый сеанс нагружает СУБД инициализацией своих настроек. | Сеансы берут уже готовый, прогретый кеш из памяти сервера. |

---

## 🗺️ Схема распределенного кеширования в кластере

БСП использует СУБД только как хранилище эталона настроек. Рабочие процессы сервера 1С обращаются к базе только при обновлении версии кеша.

```text
+-------------------------------------------------------------+

|                         База Данных                         |
|  - Хранит актуальную структуру и версию кеша (ВерсияКэша)  |
+-------------------------------------------------------------+
                               ^

                               | (Только при изменении версии)
                               v
+-------------------------------------------------------------+
|                      Кластер Серверов 1С                    |
+-------------------------------------------------------------+

        |                                             |
        v (Чтение из RAM за O(1))                     v (Чтение из RAM за O(1))
+-------------------------------+             +-------------------------------+

|     Рабочий процесс rphost 1  |             |     Рабочий процесс rphost 2  |
|  - Локальный кеш настроек     |             |  - Локальный кеш настроек     |
+-------------------------------+             +-------------------------------+

        |                                             |
   Сеансы 1..50                                  Сеансы 51..100
```

## ⚡ Оптимизация: Снижение нагрузки на СУБД с O(N) до O(1)

Рассмотрим сценарий: в системе работает регламентное задание по обработке заказов. На каждую итерацию требуется проверять, включена ли интеграция с внешней логистической системой и каковы адреса шлюзов.

### 1. Наивный подход (Деградация диска СУБД)
Разработчик запрашивает константы или параметры из БД внутри каждой итерации или при каждом создании формы.

```bsl
// Сложность: O(N) запросов к БД. На 50 000 обработанных заказов — 50 000 SELECT-запросов к таблице констант.
Для Каждого Заказ Из МассивЗаказов Цикл
    // ❌ Прямое обращение к БД на каждой итерации
    Если Константы.ИспользоватьВнешнююЛогистику.Получить() Тогда 
        АдресШлюза = Константы.АдресЛогистическогоШлюза.Получить(); 
        ОтправитьЗаказНаШлюз(Заказ, АдресШлюза);
    КонецЕсли;
КонецЦикла;
```

### 2. Подход БСП (Использование параметров работы программы)
БСП считывает эти данные один раз при старте или первом обращении, сохраняя структуру в памяти сервера приложений для всех последующих вызовов.

```bsl
// Сложность: O(1). Данные мгновенно извлекаются из RAM текущего рабочего процесса rphost.
Для Каждого Заказ Из МассивЗаказов Цикл
    // БСП обращается к глобальному кешу параметров работы программы
    ПараметрыРаботы = СтандартныеПодсистемыПовтИсп.ПараметрыРаботыПрограммы();
    
    // Все настройки уже лежат в одной структуре в оперативной памяти
    Если ПараметрыРаботы.ДоступнаВнешняяЛогистика Тогда 
        АдресШлюза = ПараметрыРаботы.АдресЛогистическогоШлюза;
        ОтправитьЗаказНаШлюз(Заказ, АдресШлюза);
    КонецЕсли;
КонецЦикла;
```

## 📌 Памятка архитектора по работе с кешем

1. **Правило обновления настроек.** Если ты программно меняешь настройки, которые закешированы в БСП, обязательно вызывай `ПараметрыРаботыПрограммы.АктуализироватьПараметры()`. Иначе другие сеансы увидят изменения только через системный таймаут (обычно 20 минут).
2. **Идентификация версии.** БСП отслеживает актуальность кеша через специальный служебный регистр сведений. Изменение версии кеша заставляет все процессы rphost перечитать данные при следующем обращении пользователей.
3. **Объем кешируемых данных.** Никогда не заталкивай в переопределяемые модули формирования кеша (`ПриОпределенииПараметровРаботыПрограммы`) тяжелые таблицы или динамические данные (курсы валют, остатки). Этот кеш должен содержать только условно-постоянные признаки и метаструктуры, иначе rphost быстро исчерпает доступную RAM.

# Этап 5: Безопасное обновление и отложенные обработчики

При обновлении крупных конфигураций монопольное выполнение тяжелых обработчиков данных приводило бы к простою бизнеса часами или сутками. БСП предоставляет технологию **отложенного обновления**, которая разделяет процесс на критический (блокирующий, сложность **O(1)** по времени простоя) и фоновый (порционный, выполняющийся параллельно с работой пользователей).

---

## ⚖️ Архитектурный баттл: Велосипед vs БСП




| Критерий сравнения | Самописный "Велосипед" ❌ | Стандарт БСП ✅ |
| :--- | :--- | :--- |
| **Доступность базы** | База полностью заблокирована для пользователей до окончания всех расчетов. | База открывается сразу после обновления метаструктуры. Данные обновляются на лету. |
| **Блокировки таблиц** | Обработка всей таблицы в одной транзакции вызывает эскалацию блокировок СУБД. | Порционная обработка данных малыми квантами времени с паузами между транзакциями. |
| **Отказоустойчивость** | Если сервер упал на 99% обновления, после перезапуска всё начинается сначала. | Точечная фиксация прогресса. После сбоя БСП продолжает ровно с того места, где остановилась. |
| **Контроль монопольности** | Разработчик "надеется", что никто не зайдет в базу во время выполнения кода. | Жесткое программное разделение на монопольный режим и оперативный фоновый режим. |

---

## 🗺️ Схема конвейера обновления данных БСП

БСП делит жизненный цикл обновления на фазы. Основная масса данных уходит в самый безопасный — третий этап.

```text
+-----------------------------------------------------------------------+

| 1. Монопольный режим (Оперативная фаза)                               |
| - Выполняется ДО допуска пользователей. Только критичные метаданные.  |
| - Длительность: секунды (Сложность O(1) для простоя бизнеса).          |
+-----------------------------------------------------------------------+

                                   |
                                   v
+-----------------------------------------------------------------------+
| 2. Открытие информационной базы для пользователей                     |
| - Пользователи заходят и начинают полноценно работать в системе.      |
+-----------------------------------------------------------------------+

                                   |
                                   v
+-----------------------------------------------------------------------+
| 3. Отложенное обновление (Фоновый режим)                              |
| - Запускаются фоновые потоки БСП. Считывают данные порциями.          |
| - Плавная конвертация терабайтных таблиц за O(N/Порции) без простоев. |
+-----------------------------------------------------------------------+
```

---

## ⚡ Оптимизация: Написание отложенного обработчика данных

Задача: Перенести данные из старого текстового реквизита `Описание` в новую табличную часть документа `ДополнительныеСведения` для 10 000 000 документов.

### 1. Наивный подход (Катастрофа в продакшене)
Разработчик пишет обработчик в модуле обновления, который блокирует базу при запуске.

```bsl
// Сложность: O(N) в монопольном режиме. База "лежит" 5 часов. Бизнес несет убытки.
Процедура ПриОбновленииИБ_ПлохойПример() Экспорт
    Выборка = Документы.РеализацияТоваровУслуг.Выбрать();
    Пока Выборка.Следующий() Цикл
        // Миллионы документов перезаписываются в одном потоке, пока пользователи ждут открытия базы
        ДокОбъект = Выборка.ПолучитьОбъект();
        МигрироватьДанныеВТЧ(ДокОбъект);
        ДокОбъект.Записать(РежимЗаписиДокумента.Запись);
    КонецЦикла;
КонецПроцедуры
```

### 2. Подход БСП (Регистрация отложенного обработчика)
Мы регистрируем обработчик как отложенный. База открывается мгновенно, а БСП сама порциями разгребает документы в фоне.

```bsl
// Код в модуле ОбновлениеИнформационнойБазыБСП. 
// База открывается мгновенно. Сложность ожидания для бизнеса: O(1).
Процедура ПриДобавленииОбработчиковОбновления(Обработчики) Экспорт
    
    Обработчик = Обработчики.Добавить();
    Обработчик.Версия = "2.4.1.50";
    Обработчик.Процедура = "МойМодульОбновления.КонвертироватьОписанияДокументов";
    Обработчик.РежимВыполнения = "Отложенно"; // Ключевой параметр БСП
    Обработчик.ВызыватьПриИзмененииДанных = Истина;
    
КонецПроцедуры

// Код обработчика в экспортном модуле. Вызывается БСП автоматически.
Процедура КонвертироватьОписанияДокументов(ПараметрыОбновления) Экспорт
    
    // БСП сама передает параметры порции и лимиты, чтобы не перегружать СУБД
    РазмерПорции = 500; 
    
    // Выбираем только необработанные документы за эту итерацию (O(K), где K = 500)
    Запрос = Новый Запрос;
    Запрос.Текст = "ВЫБРАТЬ ПЕРВЫЕ " + Строка(РазмерПорции) + " Ссылка ИЗ Документ.РеализацияТоваровУслуг...";
    
    Выборка = Запрос.Выполнить().Выбрать();
    
    Если Выборка.Количество() = 0 Тогда
        ПараметрыОбновления.ОбработкаЗавершена = Истина; // Сигнализируем БСП, что задача закрыта
        Возврат;
    КонецЕсли;
    
    Пока Выборка.Следующий() Цикл
        // Обработка строго одной порции в изолированной короткой транзакции
        ОбработатьОдинДокумент(Выборка.Ссылка);
    КонецЦикла;
    
КонецПроцедуры
```

## 📌 Памятка архитектора по обновлению ИБ

1. **Режим Замещения.** При написании отложенных обработчиков никогда не используй жесткую логику, которая может заблокировать проведение новых документов пользователями. Документы должны обновляться незаметно для текущих бизнес-процессов.
2. **Использование ОбменДанными.Загрузка.** При записи объектов внутри обработчиков обновления всегда устанавливай `ДокОбъект.ОбменДанными.Загрузка = Истина`. Это отключит прикладные проверки, подписки на события и тяжелые контроли остатков, снижая нагрузку на сервер до исторического минимума.
3. **Контроль незавершенных обработчиков.** Если в системе остались незавершенные отложенные обработчики, БСП заблокирует повторное обновление конфигурации. Архитектор должен мониторить состояние обновления через встроенную обработку `РезультатыОбновленияПрограммы`.

# Хардкор-раздел: Протухание кэша БСП vs Parameter Sniffing в СУБД

Механизм кэширования БСП (`ПараметрыРаботыПрограммы`) на первый взгляд кажется идеальным спасением с вычислительной сложностью **O(1)**. Однако в высоконагруженных распределенных системах архитектор сталкивается с эффектом **гниения кэша** (Cache Rot), который по своей разрушительной силе идентичен явлению *Parameter Sniffing* в СУБД.

---

## 💥 Анатомия проблемы: Параллели с СУБД

*   **В СУБД (Parameter Sniffing):** Оптимизатор кэширует план выполнения под «первую попавшуюся» выборку. Если следующие параметры требуют кардинально другого плана (сканирование вместо поиска по индексу), СУБД уходит в деградацию по CPU и дисковому чтению.
*   **В БСП (Смерть кэша в кластере):** При изменении функциональной опции или структуры метаданных на одном из серверов кластера, остальные рабочие процессы (`rphost`) продолжают использовать старый слепок памяти. Это приводит к каскадным ошибкам бизнес-логики и лавинообразным повторным инициализациям кэша, забивающим СУБД монопольными запросами.

---

## 🛠️ Механизм БСП по борьбе с гниением кэша

БСП не изобретает велосипедов, а использует паттерн **двухуровневой валидации с отслеживанием версии** (Optimistic Version Control).

### Как это устроено под капотом (ASCII-Логика)

```text
+-----------------------------------------------------------------+

|                       Регистр Сведений                          |
|             "ПараметрыРаботаПрограммы" (В СУБД)                 |
|   - Хранит уникальную строку: ИмяПараметра = "ВерсияКэша"       |
|   - Значение = ГУИД (Глобальный маркер версии)                  |
+-----------------------------------------------------------------+
          ^                                               ^

          |                                               |
  (А) Сравнение версии за O(1)                    (Б) Сравнение версии за O(1)

          |                                               |
+-----------------------------------+           +-----------------------------------+

|      Рабочий процесс rphost 1     |           |      Рабочий процесс rphost 2     |
| - Хранит в RAM тяжелый монолит    |           | - Хранит в RAM тяжелый монолит    |
| - ЛокальныйМаркер = ГУИД_А        |           | - ЛокальныйМаркер = ГУИД_А        |
+-----------------------------------+           +-----------------------------------+
```

### Алгоритм защиты:
1. При изменении любой системной настройки (например, через форму) БСП генерирует новый `НовыйГУИД` и записывает его в СУБД в качестве глобальной **Версии кэша**.
2. Каждый рабочий процесс `rphost` при обращении к `СтандартныеПодсистемыПовтИсп.ПараметрыРаботыПрограммы()` делает сверхлегкий запрос к СУБД: `SELECT ВерсияКэша` (индексный поиск, гарантированное чтение из кэша СУБД за **O(1)**).
3. Если локальный маркер процесса совпадает с серверным (`ГУИД_А == ГУИД_А`), процесс отдает тяжелые настройки из RAM, вообще не трогая таблицы СУБД.
4. Если данные были изменены на `rphost 2`, то в СУБД запишется `ГУИД_Б`. При следующем вызове `rphost 1` увидит, что `ГУИД_А != ГУИД_Б`. Локальный кэш этого процесса объявляется **протухшим** (Invalidated) и сбрасывается.

---

## ⚡ Опасный антипаттерн разработчика (Вызов "Грозы кластера")

Самая жесткая ошибка — это попытка динамически влиять на состав параметров кэша БСП в зависимости от контекста сеанса (аналог плохой параметризации в СУБД).

### ❌ Как делать НЕЛЬЗЯ (Динамический кэш, убивающий rphost)
Разработчик решает засунуть в кэш БСП настройки, привязав их к текущему пользователю или времени суток.

```bsl
// Код внутри переопределяемого модуля БСП, формирующего кэш. Смерть производительности!
Процедура ПриОпределенииПараметровРаботыПрограммы(Параметры) Экспорт
    
    // Внимание: Состав кэша начинает зависеть от контекста сеанса!
    Если Пользователи.ТекущийПользователь() = Справочники.Пользователи.НайтиПоРеквизиту("Код", "Архитектор") Тогда
        // Намертво ломает логику монолитности кэша для всего кластера
        Параметры.Вставить("РежимСуперДоступа", Истина); 
    КонецЕсли;
    
    // Каждое изменение версии кэша вызовет полную пересборку монолита на всех rphost
    ОбновлениеИнформационнойБазыБСП.ЗафиксироватьИзменениеВерсииКэша(); 
КонецПроцедуры
```
*Что произойдет:* Кэш станет нестабильным. Один процесс закэширует параметры под "Архитектора", второй — под рядового менеджера. Начнется постоянная инвалидация кэша во всем кластере серверов 1С. Нагрузка на CPU подскочит до 100%, пользователи начнут получать таймауты.

### ✅ Как делать ПРАВИЛЬНО (Статический монолит + динамический расчет)
Кэш должен быть абсолютно статичен по своей структуре для всех процессов кластера. Динамика проверяется по месту требования на основе статичных флагов кэша.

```bsl
// Кэш формируется один раз и одинаков для всех rphost в кластере
Процедура ПриОпределенииПараметровРаботыПрограммы(Параметры) Экспорт
    // Храним только глобальное описание логики прав, без привязки к конкретному сеансу
    Параметры.Вставить("ПрофилиССуперДоступом", ПолучитьСписокСлужебныхПрофилей()); 
КонецПроцедуры

// Проверка по месту требования (Сложность извлечения метаструктуры из кэша: O(1))
Функция РазрешенЛиСуперДоступ() Экспорт
    КэшПараметров = СтандартныеПодсистемыПовтИсп.ПараметрыРаботыПрограммы();
    // Логика сопоставления контекста сеанса и статичного кэша
    Возврат КэшПараметров.ПрофилиССуперДоступом.Найти(ПараметрыСеанса.ТекущийПрофиль) <> Неопределено;
КонецФункции
```

---

## 📌 Чек-лист архитектора: Предотвращение "гниения" кэша БСП

1. **Никакой динамики внутри ПВЗ-модулей БСП.** Процедуры инициализации параметров программы должны выдавать строго идентичный результат независимо от того, какой пользователь, под каким сеансом и на каком сервере кластера их вызвал.
2. **Контроль вызовов `ЗафиксироватьИзменениеВерсииКэша()`.** Данный метод должен вызываться исключительно при реальном изменении глобальной бизнес-логики (например, администратор программно поменял константу ведения учета). Вызывать его в цикле или при рутинных операциях пользователей — это гарантированный расстрел производительности кластера.
3. **Изоляция тяжелых структур.** Если данные меняются чаще, чем раз в день (курсы валют, оперативные статусы, остатки), им категорически запрещено находиться в `ПараметрахРаботыПрограммы`. Используй для них классический механизм `ПараметрыСеанса` или оптимизированные индексом СУБД запросы.
