# Архитектура, топология и физика кластера серверов 1С

Этот документ является глубоким инфраструктурным руководством по устройству кластера серверов 1С, механизмам клиент-серверного взаимодействия, балансировке ресурсов и управлению очередями фоновых заданий под высокой нагрузкой.

---

## Концепт 1: Архитектура и физика клиент-серверного взаимодействия

Понимание клиент-серверной архитектуры 1С на уровне сетевых пакетов, портов и сериализации данных — это базис, без которого невозможно диагностировать сетевые задержки, деградацию интерфейса и просадки производительности на промышленном контуре.

---

### 🗺️ Сетевая анатомия раундтрипа (Client-Server Trip)

Когда пользователь производит любое действие в интерфейсе тонкого клиента (нажимает кнопку, открывает форму, проводит документ), запускается сложный процесс сетевого обмена. Вся цепочка взаимодействия жестко завязана на передачу данных по протоколу TCP/IP.

```text
+-----------------------+              +-----------------------+              +-----------------------+

|     Тонкий клиент     |              |    Сервер приложений  |              |      Сервер СУБД      |
|  (User Interface RAM) |              |      (rphost RAM)     |              | (MS SQL / PostgreSQL) |
+-----------------------+              +-----------------------+              +-----------------------+
            │                                      │                                      │
            │─── 1. Сериализация контекста ───────►│                                      │
            │    (Пакет по TCP/IP, сокет)          │                                      │
            │                                      │─── 2. Генерация SQL-запроса ────────►│
            │                                      │    (Native Client / libpq)           │
            │                                      │                                      │
            │                                      │◄── 3. Получение сырых данных ────────│
            │                                      │    (Набор строк из СУБД)             │
            │◄── 4. Десериализация и отрисовка ────│                                      │
            │    (Обновление структуры формы)      │                                      │
```

#### Низкоуровневые этапы раундтрипа:
1. **Сериализация на клиенте:** Тонкий клиент собирает измененные данные формы, упаковывает их во внутренний бинарный или текстовый формат 1С (контекст) и через сетевой сокет ОС отправляет TCP-пакет на сервер приложений.
2. **Обработка на сервере:** Процесс `rphost` принимает пакет, десериализует его в объекты встроенного языка, выполняет серверный код и, если нужно, формирует низкоуровневый SQL-запрос к СУБД.
3. **Обмен с СУБД:** Сервер приложений отправляет запрос через драйвер СУБД (`SQL Server Native Client` для MS SQL или `libpq` для PostgreSQL), забирает сырой ответ и преобразует его в типы данных 1С (Выборка, Таблица Значений).
4. **Возврат на клиент:** Итоговые данные снова сериализуются платформой и летят обратно на тонкий клиент, где процессор локальной машины пользователя тратит такты на финальную отрисовку интерфейса.

### ⚡ Системные порты и сетевые дескрипторы кластера

Кластер 1С использует строго фиксированную сетку сетевых портов. Понимание их ролей критично для правильной настройки сетевых экранов (Firewalls) и локализации сетевых сбоев.

```text
                    [ Входящий запрос от Тонкого клиента ]
                                      │
                                      ▼
+---------------------------------------------------------------------------+

| Агент сервера (ragent) ── Порт 1540                                       |
| - Принимает первичное подключение, выдает список доступных кластеров      |
+---------------------------------------------------------------------------+
                                      │
                                      ▼
+---------------------------------------------------------------------------+

| Менеджер кластера (rmngr) ── Порт 1541                                    |
| - Авторизует сеанс, определяет рабочий процесс для пользователя           |
+---------------------------------------------------------------------------+
                                      │
                                      ▼
+---------------------------------------------------------------------------+

| Рабочие процессы (rphost) ── Диапазон портов 1560-1591                    |
| - Выполняют прикладной код, держат постоянные TCP-сессии с клиентами      |
+---------------------------------------------------------------------------+
```

*   **Порт 1540 (ragent):** Главный системный демон (служба ОС). Он слушает порт 1540. Его единственная задача — запустить инфраструктуру кластера, управлять процессами и ответить клиенту на самый первый пинг: «Да, я тут, вот список менеджеров кластера».
*   **Порт 1541 (rmngr):** Сердце управления кластером. Слушает порт 1541. Он координирует работу всех сервисов (блокировки, сеансы, регламентные задания). Именно он решает, в какой конкретно рабочий процесс отправить сессию пользователя.
*   **Диапазон портов 1560-1591 (rphost):** Рабочая лошадь кластера. Каждый запущенный процесс `rphost` динамически захватывает один свободный порт из этого диапазона. Именно по этим портам идет основной, самый тяжелый трафик обмена данными формы и СУБД.

### 🧠 Транспортный уровень СУБД и сетевые задержки (Network Latency)

Одна из самых частых и скрытых причин жестких тормозов на промышленных highload-контурах — это **сетевое расстояние между сервером приложений (1С) и сервером СУБД**.

#### Почему это критично на низком уровне:
Когда 1С выполняет сложный запрос со множеством мелких подзапросов или итеративно считывает данные, между `rphost` и СУБД происходит огромное количество микро-транзакций. Этот эффект называется **Chatty Query** (болтливый запрос).

*   Если Сервер 1С и Сервер СУБД разнесены на разные физические сервера (железки) через обычный 1-гигабитный коммутатор, задержка сети (`Network Latency`) на один пакет составляет около **1-2 миллисекунд**. 
*   Если запрос делает 10 000 последовательных сетевых обращений к СУБД, то чистые потери времени *только на сетевой транспорт* составят:
    $$\text{Потери} = 10000 \times 1.5 \text{ мс} = 15 \text{ секунд!}$$
    При этом процессоры на сервере 1С и сервере СУБД будут простаивать, ожидая окончания передачи байтов по медному кабелю.

#### Стратегии выживания на проде (Highload-вывод):
1. **Размещение на одной железке (Shared Memory):** Если объем базы данных позволяет, для баз с высокой интенсивностью коротких запросов сервер 1С и сервер СУБД устанавливаются на один физический сервер. В этом случае сетевой стек TCP/IP вообще не используется — обмен данными идет напрямую через оперативную память сервера по сверхбыстрому протоколу `Shared Memory` (Общая память). Задержка падает практически до нуля.
2. **Выделенная 10-Гигабитная сеть:** Если базы огромные и разделение серверов неизбежно, сервер 1С и сервер СУБД должны быть соединены **напрямую** (минуя общекорпоративные свитчи) через выделенные сетевые карты 10 Gbps (или 25 Gbps) с минимальным пингом.
3. **Размер сетевого пакета (Network Packet Size):** В настройках экземпляра MS SQL Server или PostgreSQL для highload-контуров 1С размер сетевого пакета увеличивают со стандартных 4 Кб до 8 Кб или 16 Кб, чтобы тяжелые выборки таблиц 1С не дробились на сотни мелких сетевых кадров.

## Концепт 2: Анатомия процессов кластера и физика балансировки сеансов

Просто запустить сервер 1С «из коробки» на крупном проекте — это гарантированный способ получить каскадное падение всей системы в пиковые часы нагрузки. По умолчанию платформа стремится упаковать максимум данных в минимальное количество процессов rphost, что под нагрузкой превращает рабочую базу в пороховую бочку.

---

### ⚖️ Архитектурный разбор компонентов кластера

Разберем системные роли процессов 1С на уровне операционной системы (Windows Service / Linux Daemon).



| Процесс ОС | Имя процесса | Главная роль в архитектуре | Последствия падения на проде |
| :--- | :--- | :--- | :--- |
| **Агент сервера** | `ragent` | Запуск, мониторинг и поддержка жизнедеятельности процессов `rmngr` и `rphost`. | Новые подключения невозможны. Существующие пользователи работают, пока живы их `rphost`. |
| **Менеджер кластера** | `rmngr` | Хостинг критических сервисов (Блокировки, Сеансы, Очередь регламентных заданий). | Полный паралич системы. Сеансы зависают, так как `rphost` и СУБД теряют координатора. |
| **Рабочий процесс** | `rphost` | Исполнение прикладного кода, вычисление запросов, работа с RAM сеансов. | Вылет всех пользователей, чьи сеансы физически находились внутри данного процесса `rphost`. |

---

### 🗺️ Карта дорог балансировщика: Физика распределения сеансов

Балансировка сеансов в кластере 1С — это не классический Round-Robin. Платформа распределяет новые подключения на основе сложного расчета «Доступной производительности» каждого процесса `rphost`.

Ниже представлена точная логика ветвления, по которой менеджер кластера `rmngr` ищет целевой процесс для нового пользователя.

```text
               [ Новое подключение пользователя (Тонкий клиент) ]
                                       │
                                       ▼
               [ Менеджер кластера опрашивает пул рабочих процессов ]
                                       │
                     ┌─────────────────┴─────────────────┐
                     ▼                                   ▼
         [ Дерево №1: Свободный rphost ]     [ Дерево №2: Перегруженный rphost ]
         - Занятая память: 4 Гб              - Занятая память: 28 Гб (Лимит!)
         - Количество сеансов: 15            - Количество сеансов: 120
                     │                                   │
                     ▼                                   ▼
           Идеальный Seek баланса!             Катастрофа! Эффект коммуналки.
                     │                                   │
                     ▼                                   ▼
          [ СИТУАЦИЯ 1: Успешный вход ]       [ СИТУАЦИЯ 2: Блокировка / Отказ ]
          Сеанс прицепляется за O(1).         Новый сеанс зависает.
          Нагрузка распределена в RAM.        Соседи вылетают по Out Of Memory.
```

---

### 🧠 Физика «Эффекта коммуналки» в рабочих процессах

Если в настройках кластера параметры распределения оставлены по умолчанию (значение `0`), платформа будет запихивать все новые сеансы в один и тот же рабочий процесс, пока он физически не упрется в лимиты разрядности ОС или железа. Это порождает опасный **эффект коммунальной квартиры**.

#### Что происходит под нагрузкой на самом низком уровне:
1. Внутри одного процесса `rphost` находятся 150 сеансов обычных пользователей (менеджеры, операторы, розница). Они делают короткие, точечные запросы, потребляя по 100-200 Мб RAM на сессию.
2. Сеанс №151 (например, аналитик) запускает неоптимальный отчет «Остатки и обороты» за три года без отборов по индексам.
3. Этот один сеанс заставляет интерпретатор 1С внутри `rphost` выделить гигантский квант памяти (20-30 Гб) под построение тяжелого дерева значений. Память процесса начинает лавинообразно расти.
4. **Проблема утилизации CPU и RAM:** Так как операционная система выделяет ресурсы и кванты процессорного времени на весь процесс `rphost` целиком (как на единую неделимую сущность), планировщик ОС начинает тормозить выполнение потоков *всех* 150 пользователей внутри этого процесса. Интерфейс зависает у всех одновременно.
5. **Финал (Убийство процесса):** Как только `rphost` превышает критический порог доступной физической памяти сервера, операционная система принудительно уничтожает его (механизм `OOM Killer` в Linux или диспетчер памяти Windows). 
*Итог:* Один человек написал плохой код, а система аварийно оборвала сессии 150 сотрудников, уничтожив их транзакции и несохраненные данные.

### ⚡ Практика Highload: Жесткое квотирование через консоль кластера

Чтобы защитить систему от эффекта коммуналки, администраторы крупных кластеров полностью ломают дефолтное поведение платформы, принудительно заставляя `rmngr` изолировать сеансы.

#### Топовые параметры настройки rphost (Жизненно важно для прода):

1. **Количество сеансов на один рабочий процесс = от 15 до 25 (в зависимости от тяжести базы).**
   * *Как это работает:* По умолчанию стоит `0` (бесконечно). Установка лимита в `20` означает, что как только в первый `rphost` зайдет 20 человек, балансировщик кластера заблокирует его для новых подключений и принудительно стартует на сервере операционной системы совершенно новый, изолированный процесс `rphost` для следующих пользователей.
   * *Профит:* Если теперь аналитик запустит свой смертоносный отчет, он раздует и обрушит только свой изолированный процесс, где сидит максимум 19 его соседей. Остальные 90% пользователей базы, работающие в параллельных процессах, даже не заметят проблемы.

2. **Количество рабочих процессов на один процессор = 1.**
   * *Как это работает:* Регулирует плотность распределения процессов по ядрам CPU. Топовая практика — держать количество процессов равным числу ядер (или потоков) для равномерного распределения нагрузки на уровне диспетчера задач ОС.

3. **Допустимый объем памяти = Расчетная величина (например, 6 000 000 Кб на один rphost).**
   * *Как это работает:* Жесткий лимит безопасности. Если один процесс уходит в неконтролируемое раздувание и пересекает эту отметку, кластер переводит данный `rphost` в статус **«Недоступен для новых подключений»** и запускает таймер «Интервала превышения». Сеансы пользователей начинают постепенно и незаметно перетекать в новые чистые процессы при их очередных серверных вызовах.

## Концепт 2.5: Межсерверное взаимодействие и физика системных сбоев (Heartbeat и каскадный коллапс)

В распределенном кластере (когда база обслуживается несколькими физическими серверами) процессы должны постоянно координировать свои действия. Балансировка сеансов, синхронизация транзакционных блокировок 1С и распределение регламентных заданий завязаны на постоянный сетевой пинг между нодами. Если один сервер «задыхается», в критическую зону попадает весь кластер.

---

### ⚖️ Физика сетевого консенсуса: Механизм Heartbeat

Менеджер кластера `rmngr` (на центральном сервере) и процессы `rphost` на рабочих серверах общаются по внутреннему протоколу через фиксированные интервалы времени. Этот процесс называется **Heartbeat (Пульс)**.

```text
+------------------------------+                +------------------------------+
|  Центральный сервер (Нода 1) |                |    Рабочий сервер (Нода 2)   |
|   - Процесс [rmngr]          |                |    - Процесс [rphost]        |
+------------------------------+                +------------------------------+
               │                                                │
               │─────── 1. Ping: "Ты живой?" (Каждые 1-2 сек) ─►│ (RAM в норме, CPU 15%)
               │                                                │
               │◄────── 2. Pong: "Да, нагрузка в норме" ────────│ (Ответ за милисекунды)
               │                                                │
               │   [ СИТУАЦИЯ: На Ноду 2 прилетает тяжелый запрос / Out Of Memory ]
               │                                                │
               │─────── 3. Ping: "Ты живой?" ──────────────────►│ 💥 (CPU 100%, RAM в полке)
               │                                                │    Потоки ОС заморожены.
               │                                                │    Ответа НЕТ.
               │                                                │
               ▼                                                ▼
  [ ТАЙМАУТ ПРЕВЫШЕН (Обычно 15-30с) ]             [ ОС убивает процесс через OOM ]
  - rmngr объявляет Ноду 2 мертвой.               - Сеансы Ноды 2 аварийно рвутся.
  - Начинается каскадный перенос сессий.          - Данные транзакций отменяются.
```

1. **Нормальное состояние:** Каждые несколько секунд (в зависимости от настроек платформы) `rmngr` отправляет пакеты проверки состояния на все рабочие процессы `rphost`. В ответ процессы возвращают метрики: текущий счетчик сеансов, объем утилизированной RAM и загрузку ядер CPU.
2. **Аварийное состояние (Зависание/Раздувание):** Когда на Рабочий сервер (Нода 2) прилетает некорректный запрос, который выжирает всю RAM или намертво грузит CPU, операционная система начинает вытеснять системные потоки `rphost` в файл подкачки (Swap) или просто замораживает планировщик потоков. `rphost` физически теряет способность отвечать на сетевые сокеты.
3. **Фиксация смерти:** `rmngr` не получает ответный `Pong` в течение установленного таймаута (внутренний Ping Timeout платформы). Для менеджера кластера этот сервер становится «черным ящиком». Платформа объявляет рабочий сервер аварийным.

---

### ⚡ Каскадный коллапс: Что происходит на проде при падении одной ноды?

Когда `rmngr` фиксирует смерть одного из серверов кластера, запускается **каскадная реакция**, которая при неправильной настройке может мгновенно «сложить» оставшиеся живые серверы.

#### Хронология катастрофы под нагрузкой:

1. **Обрыв транзакций:** Все пользователи, чьи сеансы находились на упавшем сервере, мгновенно получают ошибку `Соединение с сервером приложений утеряно`. СУБД откатывает (Rollback) все их незавершенные транзакции, чтобы защитить базу от битых данных.
2. **Наплыв выживших (Storm Effect):** Менеджер кластера `rmngr` исключает упавшую ноду из списка доступных для балансировки. Тысячи пользователей, у которых только что вылетела 1С, начинают одновременно нажимать кнопку «Войти в базу».
3. **Эффект домино (Cascading Failure):** Весь этот дикий поток новых подключений `rmngr` начинает перенаправлять на оставшиеся живые серверы (например, на Ноду 3). 
   * Если Нода 3 до этого работала на пределе возможностей (например, CPU был загружен на 70%), то внезапный наплыв 500 новых пользователей мгновенно утилизирует оставшиеся 30% ресурсов.
   * Нода 3 точно так же уходит в полку по CPU/RAM, перестает отвечать на Heartbeat-запросы менеджера, и `rmngr` объявляет мертвой и её. **Кластер полностью уничтожен за 2-3 минуты из-за каскадного переноса веса.**

---

### 🛠️ Инженерная защита: Паттерны выживания кластера

Чтобы распределенный кластер не складывался как карточный домик при отказе одного из узлов, на уровне инфраструктуры настраиваются жесткие барьеры.

1. **Запас по мощности (N+1 / N+2):** Суровое правило корпоративной архитектуры: суммарная мощность серверов кластера должна проектироваться так, чтобы при падении *любого одного физического сервера* оставшиеся ноды могли без деградации принять на себя 100% его пользователей. Если для работы базы нужно 2 сервера — на проде должно стоять 3.
2. **Ограничение объема памяти фоновых заданий в кластере:** Тяжелые фоновые и регламентные задания — главные спонсоры падений по Out Of Memory. В настройках кластера устанавливается лимит: «Максимальный объем памяти фоновых заданий». Если фоновые задачи начинают сжирать память ноды, кластер не даст им запуститься, сохранив RAM под интерактивные сессии живых людей.
3. **Выделенная сеть синхронизации (Cluster Interconnect):** Обмен Heartbeat-пакетами и данными сервиса транзакционных блокировок между серверами кластера должен идти по выделенной изолированной физической сети (отдельные сетевые порты на серверах). Если пользовательский сегмент сети будет забит тяжелой выгрузкой файлов, `rmngr` все равно гарантированно получит `Pong` от рабочего сервера по изолированному каналу, и ложной инвалидации ноды не произойдет.

---

## Концепт 3: Квотирование ресурсов и стратегии выживания при утечках памяти (Resource Profiles)

Процесс `rphost` — это многопоточный исполнитель кода. Главная уязвимость платформы заключается в том, что операционная система (Windows/Linux) выделяет RAM и кванты процессорного времени на весь процесс целиком, как на единую сущность. Если не квотировать ресурсы процессов на уровне менеджера кластера, один маргинальный сеанс способен вызвать лавинообразное раздувание всей ноды.

---

### 🧠 Механика фрагментации памяти кучи (Heap Fragmentation)

Когда разработчик очищает тяжелую коллекцию данных (например, пишет `ТЗ = Неопределено;`), виртуальная машина 1С возвращает эту память во внутренний пул аллокатора платформы. 

Однако на уровне операционной системы процесс `rphost` **не отдает** эту память обратно в ОС мгновенно. ОС видит, что процесс по-прежнему удерживает виртуальные адреса (эффект High Watermark). Память остается зарезервированной за процессом, куча фрагментируется, и в диспетчере задач `rphost` продолжает «весить» десятки гигабайт. Единственный физический способ очистить эту память на проде — полностью перезапустить процесс.

---

### 🗺️ Карта дорог вытеснения процессов (Graceful Recycle)

Чтобы очистка памяти происходила незаметно для пользователей, кластер использует механизм мягкого перезапуска отработанных процессов.

```text
       [ rphost утилизирует память при обработке данных сеансов ]
                                   │
                                   ▼
 [ Достигнут лимит: Память процесса > Допустимый объем памяти (V_warn) ]
                                   │
                                   ▼
     [ Кластер переводит rphost в статус: "Отработанный" ]
     - Процесс полностью блокируется для приема НОВЫХ сеансов.
     - Новые пользователи заходят в автоматически созданный чистый rphost.
     - Запускается внутренний таймер: "Интервал превышения" (T_limit).
                                   │
                ┌──────────────────┴──────────────────┐
                ▼                                     ▼
   [ Пользователи совершают клики ]      [ Таймер T_limit истек ]
   [     (Серверные вызовы)       ]      [ (Например, 300 секунд) ]
                │                                     │
                ▼                                     ▼
    В моменты пауз между вызовами          Менеджер кластера посылает
    платформа бесшовно переносит           сигнал ОС на принудительное
    сеанс в новый чистый rphost.           уничтожение процесса (Kill -9).
                │                                     │
                ▼                                     ▼
  [ СИТУАЦИЯ 1: Бесшовный Recycle ]      [ СИТУАЦИЯ 2: Жесткий сброс ]
  Сеансы мигрировали без потерь.         Обрыв не успевших уйти сессий.
  Старый rphost умер за O(1) памяти.     Пользователи видят ошибку связи.
```

### ⚡ Формулы и регламент настройки лимитов памяти на Highload-проде

Никогда не выставляй параметры квотирования наугад. Для крупных конфигураций (ERP, КА, УХ) на промышленных серверах Эксперты используют строго фиксированные формулы расчета.

#### Исходные данные для расчета:
*   $RAM_{total}$ — Общий объем физической оперативной памяти на сервере.
*   $RAM_{os}$ — Резерв под нужды ОС и СУБД (если она на той же железке). Минимум 4–8 Гб.
*   $N_{cores}$ — Количество физических ядер процессора.

---

#### Параметр 1. Допустимый объем памяти (V_warn)
Это порог, при достижении которого процесс признается «стареющим» и подлежит замене.

$$\text{Допустимый объем памяти} = \frac{RAM_{total} - RAM_{os}}{\text{Количество процессов rphost}}$$

*   *Пример на проде:* Сервер имеет 64 Гб RAM. Мы выделили 8 рабочих процессов `rphost`. Резерв под ОС — 4 Гб.
    $$V_{warn} = \frac{64 - 4}{8} = 7.5 \text{ Гб} \rightarrow 7\ 500\ 000 \text{ Кб}$$
*   *Важно:* Если оставить этот параметр равным `0` (по умолчанию), кластер никогда не запустит процедуру автоматической очистки памяти, и сервер гарантированно упадет по Out Of Memory.

---

#### Параметр 2. Интервал превышения допустимого объема памяти (T_limit)
Время, в течение которого «отработанный» процесс удерживается в памяти, позволяя сеансам завершить текущие транзакции и мигрировать на новые процессы.

*   *Highload-стандарт:* **300 секунд** (5 минут).
*   *Физика процесса:* Если поставить слишком мало (например, 30 секунд), пользователи с долгими операциями не успеют перетечь и вылетят с ошибкой соединения. Если поставить слишком много (например, 3600 секунд), раздутый процесс будет целый час отжирать дефицитную RAM, провоцируя падение соседних процессов.

---

#### Параметр 3. Выключенные процессы останавливать через (T_force)
Жесткий дедлайн для ОС. Время, через которое менеджер кластера принудительно вызовет системный `Kill` для процесса, если он сам не закрылся.

*   *Highload-стандарт:* **60 секунд**. Этого времени гарантированно хватает, чтобы операционная система корректно выгрузила дескрипторы и освободила страницы памяти в системный Buffer Pool.

---

#### Параметр 4. Критический объем памяти (V_max)
Абсолютный барьер безопасности. Значение, при превышении которого процесс ликвидируется немедленно, без ожидания таймеров миграции сеансов.

$$\text{Критический объем памяти} = V_{warn} \times 1.5 \quad (\text{или } 2.0)$$

*   *Пример на проде:* Если допустимый объем равен 7.5 Гб, то критический выставляется в районе **12 000 000 Кб** (около 11.4 Гб). Это защищает сервер от «взрывного» раздувания, когда один некорректный запрос начинает выжирать по 2 Гб памяти в секунду.

---

### 📌 Контрольный чек-лист Эксперта по диагностике памяти

1. **Флаг "Принудительно завершать проблемные процессы".** Должен быть строго **Включен** в свойствах кластера. Если его выключить, то процессы, зависшие в дедлоках или бесконечных циклах, будут висеть в памяти вечно, игнорируя любые лимиты.
2. **Контроль объема памяти фоновых заданий.** Фоновые задачи потребляют в 2–3 раза больше памяти, чем интерактивные сессии. В свойствах рабочих серверов жестко ограничивай параметр *«Объем памяти рабочих процессов, до которого сервер считается производительным»*. Экспертный лимит — **80%** от общего объема RAM, после чего кластер перестает отдавать тяжелые фоновые задачи на эту ноду.

## Концепт 4: Требования функциональности (Правила распределения) и очереди фоновых задач

В высоконагруженных системах интерактивные пользователи (люди, которые бьют заказы, чеки и накладные в реальном времени) и регламентные/фоновые задания (тяжёлые расчёты себестоимости, закрытия месяца, обмены с маркетплейсами) категорически **не должны находиться в одной лодке**. Если не изолировать их потоки, ночной расчёт себестоимости, затянувшийся до утра, парализует работу всей розничной сети или склада.

---

### 🧠 Механика планировщика заданий и приоритизация

За запуск и координацию фоновых и регламентных задач отвечает **Сервис фоновых заданий**, который физически крутится внутри менеджера кластера `rmngr`. 

#### Как работает очередь под нагрузкой:
1. При наступлении расписания или программном вызове `ФоновыеЗадания.Выполнить()` задача встаёт в общую системную очередь `rmngr`.
2. Менеджер кластера оценивает текущую «Доступную производительность» рабочих серверов и процессов.
3. По умолчанию `rmngr` отправляет фоновое задание в тот же самый пул `rphost`, где сидят интерактивные пользователи. Фоновое задание создаёт внутри процесса параллельный поток (`Thread`), который начинает утилизировать общие ядра CPU и RAM, отбирая такты у менеджеров.

---

### 🗺️ Карта дорог изоляции потоков (Требования функциональности)

Чтобы защитить живых людей от тяжёлых фоновых расчётов, Highload-архитекторы используют механизм **Требований функциональности** кластера серверов 1С. Мы принудительно разделяем физические серверы или конкретные процессы на «клиентские» и «фоновые».

```text
               [ Запуск фонового или регламентного задания ]
                                     │
                                     ▼
                [ Менеджер кластера rmngr проверяет правила ]
                                     │
                    ┌────────────────┴────────────────┐
                    ▼                                 ▼
      [ Правило: Клиентский сервер ]     [ Правило: Выделенный сервер ]
      (Нода №1: Для живых людей)         (Нода №2: Для фоновых задач)
      - РК: "Сервис фоновых заданий"     - РК: "Сервис фоновых заданий"
        Режим: "НЕ НАЗНАЧАТЬ"              Режим: "НАЗНАЧАТЬ"
                    │                                 │
                    ▼                                 ▼
         Поток защищен от спама!            Идеальный Seek планировщика!
                    │                                 │
                    ▼                                 ▼
     [ СИТУАЦИЯ 1: Высокий APDEX ]      [ СИТУАЦИЯ 2: Изолированный Heavy ]
     Интерфейс менеджеров летает,       Тяжёлый расчёт утилизирует CPU
     CPU Ноды 1 занят на 15%.           Ноды 2 на 100%, не мешая людям.
```

### ⚡ Практика настройки Требований функциональности (РК) на проде

Настройка правил выполняется в консоли администрирования кластера в ветке **«Требования функциональности»** для каждого конкретного рабочего сервера. Параметры настраиваются через жесткие Требования (Регламентные Категории — РК).

#### Топовый шаблон изоляции для Highload-контура (2 сервера):

##### 🖥️ Сервер 1 (Центральный — App Server для пользователей)
Сюда пускаем только людей. Любые тяжёлые фоновые сервисы отсюда выжигаются.
1.  **Объект требования:** `Сервис фоновых заданий` | **Тип требования:** `Не назначать`.
    *   *Физика:* Кластеру запрещено запускать любые регламентные и фоновые задачи на этой машине.
2.  **Объект требования:** `Клиентское соединение` | **Тип требования:** `Назначать`.
    *   *Физика:* Сервер принимает интерактивные сессии пользователей, гарантируя им максимальный приоритет процессора.

##### ⚙️ Сервер 2 (Фоновый — Выделенный калькулятор)
Этот сервер берёт на себя всю грязную и тяжёлую работу. Пользователи сюда физически зайти не могут.
1.  **Объект требования:** `Клиентское соединение` | **Тип требования:** `Не назначать`.
    *   *Физика:* Обычным пользователям вход сюда закрыт. Их сессии не будут жрать память метаданных.
2.  **Объект требования:** `Сервис фоновых заданий` | **Тип требования:** `Назначать`.
    *   *Физика:* Все регламентные задачи со всех баз кластера принудительно стягиваются на эту железку.

---

### 🧠 Квотирование очередей: Параметр "Максимальный объем памяти фоновых заданий"

Если у тебя на фоновом сервере запущено 50 разных регламентных заданий, и каждое решит сожрать по 2 Гб, фоновый сервер точно так же умрёт по Out Of Memory. Чтобы этого не произошло, Эксперты настраивают квоты на уровне параметров самого Рабочего сервера:

*   **Максимальный объем памяти фоновых заданий:** Задаёт жесткую планку для Сервиса заданий (например, **16 000 000 Кб**).
    *   *Как это работает:* Если текущие запущенные фоновые процессы суммарно уже сожрали 16 Гб RAM, а планировщик по расписанию пытается запустить следующее задание обмена, `rmngr` блокирует старт. Новая задача не падает с ошибкой, а остаётся в очереди в статусе `Ожидание выполнения` до тех пор, пока предыдущие задачи не завершатся и не освободят память.
*   **Количество заданий на один рабочий процесс = 1.**
    *   *Зачем это нужно:* Для фоновых процессов `rphost` устанавливается правило: одна задача — один процесс. Это гарантирует стопроцентную изоляцию тяжёлых расчётов друг от друга на уровне ОС. Если упадёт по памяти расчёт себестоимости, он не утянет за собой обмен с сайтом, который крутится в соседнем процессе `rphost`.

## Концепт 5: Физика очередей, приоритетов и блокировок фоновых заданий

Платформа 1С не имеет встроенного в конфигуратор ползунка «Выставить приоритет High/Low» для фонового задания. Все программно вызванные методы `ФоновыеЗадания.Выполнить()` попадают в планировщик `rmngr` на равных правах. Управление очередностью выполнения, выстраивание зависимостей задач и блокировка тяжелых отчетов до завершения критических процессов (например, обновления ИБ) ложится на плечи архитектора через механизмы Ключей и подсистемы БСП.

---

### ⚖️ Механика квотирования очередей: Как rmngr распределяет задачи

Когда в планировщик `rmngr` одновременно залетает 100 фоновых заданий, менеджер кластера не запускает их все одновременно. Он использует алгоритм **Квотирования рабочих потоков**.

*   **Лимиты кластера:** Менеджер кластера считывает параметр рабочего сервера *«Количество заданий на один рабочий процесс»*. Если на сервере доступно 4 процесса `rphost`, а лимит равен 5 заданиям на процесс, то одновременно на сервере могут выполняться максимум **20 фоновых задач**.
*   **Состояние ожидания:** Остальные 80 заданий переводятся кластером в статус `Ожидание выполнения` (Задание поставлено в очередь). Они будут подхватываться планировщиком строго по мере освобождения слотов в `rphost` в порядке FIFO (First In, First Out — первый пришел, первый ушел).

---

### 🗺️ Механизм Ключей (КлючИмениМетода): Защита от параллельного дублирования

Главный инструмент защиты СУБД от повторного запуска одинаковых тяжелых расчетов на уровне платформы — это свойство **Ключ** фонового задания.

```text
       [ Запуск регламентного задания: "РасчетСебестоимостиЗаГод" ]
       [ Передан параметр Ключ = "УчетныйГод_2026"                ]
                                     │
                                     ▼
     [ Менеджер кластера rmngr проверяет активную таблицу ключей ]
                                     │
                    ┌────────────────┴────────────────┐
                    ▼                                 ▼
         [ Ситуация А: Ключ СВОБОДЕН ]     [ Ситуация Б: Ключ ЗАНЯТ ]
         (Такой расчет еще не запущен)     (Процесс с этим ключом уже активен)
                    │                                 │
                    ▼                                 ▼
         [ СУБД стартует поток ]           [ ПЛАТФОРМА БЛОКИРУЕТ ЗАПУСК ]
         Задание выполняется за O(N).      - Поток НЕ создается в rphost.
                                           - Новое задание СРАЗУ сбрасывается.
                                           - Защита от дублирования блокировок.
```

*   **Как это работает:** Если ты запускаешь тяжелый расчет за год и передаешь в метод `ФоновыеЗадания.Выполнить(..., ..., Ключ)` уникальную строку, платформа регистрирует этот ключ в памяти `rmngr`. Если другой пользователь (или регламентное расписание) попытается вызвать этот же метод с таким же Ключом, платформа **не создаст** второй параллельный поток и не пустит его в очередь. Вызов проигнорируется, защищая СУБД от взаимных дедлоков.

---

### ⚡ Практика Highload: Управление зависимостями через БСП «Очередь заданий»

Чтобы реализовать жесткий сценарий: *«Не запускать фоновый отчет за год, пока выполняются критические обновления БСП или расчет оперативных итогов»*, стандартного механизма платформы мало. Архитекторы используют **Подсистему БСП "Очередь заданий"**, которая реализует паттерн **Priority Queue (Очередь с приоритетами)**.

#### Как настроить приоритеты и зависимости в коде БСП:

##### 1. Разделение по Приоритетам (Квоты важности)
БСП позволяет программно выставлять заданиям численный приоритет (обычно от 1 до 5). Планировщик БСП сначала выгребает из СУБД задачи с наивысшим приоритетом, сдвигая тяжелые аналитические отчеты в конец очереди.
```bsl
// Выставляем максимальный приоритет для критической задачи (Обновление/Обмен)
ПараметрыЗадания = Новый Структура;
ПараметрыЗадания.Вставить("ИмяМетода", "ОбменСМаркетплейсами.Выполнить");

ПараметрыИнициализации = Новый Структура;
ПараметрыИнициализации.Вставить("Приоритет", 5); // 🎯 Топовый приоритет. Выполнится без очереди.

ОчередьЗаданий.ДобавитьЗадание("СлужебныйМодульБСП", ПараметрыЗадания, ПараметрыИнициализации);
```

##### 2. Выстраивание жесткой зависимости (Запрет старта)
Чтобы тяжелый годовой отчет гарантированно ждал окончания других фоновых процессов, БСП использует механизм **Состояний** и проверку активных очередей. В обработчике старта годового отчета прописывается жесткий контроль:

```bsl
// Код внутри процедуры, которая должна выполниться в фоновом режиме
Процедура РассчитатьГодовойОтчетФон(ПараметрыЗадачи) Экспорт

    Если ОбновлениеИнформационнойБазыСлужебный.ВыполняетсяОбновлениеИнформационнойБазы() Тогда
        // ❌ База обновляется. Годовому отчету здесь делать нечего. 
        // Мы принудительно переносим старт задания на 10 минут позже, не забивая RAM.
        ВызватьИсключение "Критическая ошибка: Запуск отчета заблокирован до окончания обновления ИБ!";
    КонецЕсли;
    
    // Использовать Ключ фонового задания платформы для атомарной блокировки на уровне rmngr

    КлючЗапуска = "ОтчетЗаГод_" + Формат(ПараметрыЗадачи.Год, "ЧГ=0");
```

### 📌 Регламентные правила Эксперта по контролю очередей на проде

1. **Разделение расписаний по времени (Time Window):** Все тяжелые фоновые задачи уровня `O(N)` по времени выполнения (свертка таблиц, закрытие прошлых периодов, тяжелая аналитика за год) должны иметь расписание, жестко вынесенное за рамки рабочего дня пользователей (например, с 01:00 до 05:00 утра).
2. **Использование ОбменДанными.Загрузка в фоне:** Если фоновое задание занимается массовой записью объектов (например, фоновое обновление реквизитов), внутри его кода **обязательно** должен выставляться флаг `ОбменДанными.Загрузка = Истина`. Это отключает прикладные контроли и подписки, снижая процессорное время выполнения одной задачи в разы и быстрее освобождая слоты в очереди для других фоновых процессов.
3. **Мониторинг застрявших заданий:** Если фоновое задание ушло в бесконечный цикл или дедлок СУБД, оно будет вечно занимать свой слот в `rphost`. Тимлид обязан настроить оповещения (через замеры APDEX или парсинг ТЖ) о фоновых задачах, чье время выполнения превышает критический порог (например, более 2 часов), для их принудительного уничтожения в консоли кластера.
