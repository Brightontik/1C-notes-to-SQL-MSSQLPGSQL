# Низкоуровневый справочник PostgreSQL: Анатомия памяти, блокировок и параметров конфигурации

Этот документ описывает физику распределения ОЗУ в СУБД PostgreSQL, взаимосвязь параметров ядра и причины возникновения критических ошибок при высоких нагрузках (**RPS 15 000 – 50 000+**).

---

## 1. Анатомия ошибки «out of shared memory»

### Цепочка триггера ошибки:
1. Входящий highload-поток порождает тысячи параллельных сессий.
2. Каждая сессия выполняет запросы 1С, которые неявно затрагивают десятки таблиц, индексов и динамических временных таблиц (ВТ).
3. Для каждого изменяемого или читаемого объекта транзакция обязана захватить блокировку уровня объекта (например, `RowExclusiveLock` для таблицы).
4. Запись о блокировке объекта должна быть физически размещена в фиксированной матрице (таблице блокировок) внутри **Shared Memory** (Разделяемой памяти СУБД).
5. Если свободные слоты в этой матрице заканчиваются, СУБД не может зарегистрировать новый процесс и выбрасывает ошибку: `ERROR: out of shared memory. HINT: You might need to increase max_locks_per_transaction.`

---

## 2. Справочник критических параметров ОЗУ PostgreSQL



| Имя параметра в `postgresql.conf` | К какому слою памяти относится | Физическая суть («Что делает под капотом») | Роль в Highload-системах (RPS 15k+) |
| :--- | :--- | :--- | :--- |
| **`max_connections`** | Статический лимит системы | Задает максимальное количество одновременных процессов-клиентов (`postgres`), которые ОС Linux разрешит запустить. | **Главный балласт.** Если завысить без пулера, раздует расходы на системные переключения контекста процессора и сожрет ОЗУ. |
| **`max_locks_per_transaction`** | Статический лимит транзакции | Задает среднее количество блокировок *тяжелых объектов* (таблиц, секций, индексов), которое одна транзакция может держать одновременно. | **Запас прочности.** По умолчанию равен 64. На кривом legacy-коде 1С (с кучей ВТ) этот лимит пробивается влет, вызывая общую ошибку памяти. |
| **`shared_buffers`** | Shared Memory (Разделяемая) | Главный кэш СУБД в ОЗУ для хранения страниц данных и индексов, прочитанных с диска. | Настраивается жестко (обычно 25% от всей физической RAM сервера). Из него же выделяются служебные структуры под трекинг блокировок. |
| **`work_mem`** | Local Memory (Локальная) | Объем памяти, выделяемый **каждому** индивидуальному процессу СУБД под операции сортировки (`ORDER BY`, `Merge Sort`) и хэширования. | **Главный виновник падения сервера по OOM Killer.** Память выделяется динамически. Если 50 000 процессов одновременно потребуют по куску `work_mem`, физическая RAM сервера мгновенно испарится. |

## 3. Математика СУБД: Физические формулы расчета

При старте PostgreSQL жестко запрашивает у ядра операционной системы кусок памяти под служебные нужды (включая таблицу блокировок объектов). 

### Формула емкости таблицы блокировок в Shared Memory:

Количество доступных слотов для регистрации блокировок объектов в ОЗУ вычисляется следующим образом:

```text
Total_Lock_Slots = (max_connections + max_prepared_transactions) * max_locks_per_transaction
```

*Где `max_prepared_transactions` — это слоты под двухфазные транзакции (в кластерах 1С обычно не используются или равны нулю).*



### Почему возникает системный перекос (Парадокс Балласта):
Если админ выставил `max_connections = 10 000` и `max_locks_per_transaction = 64`, СУБД создаст в ОЗУ матрицу на **640 000 слотов**.

1. Если на сервер пришло всего 1 000 реальных параллельных клиентов, но каждый из них из-за тяжелых пакетных запросов 1С попытался заблокировать по **700 объектов** (индексов и таблиц)...
2. Суммарно им потребуется: $1\ 000 \times 700 = \mathbf{700\ 000\ \text{слотов}}$.
3. Матрица переполнена ($700\ 000 > 640\ 000$), и СУБД падает в `out of shared memory`.
4. При этом 9 000 слотов подключений (`max_connections`) простаивают мертвым балластом. Память под них зарезервирована, но перераспределить ее под нужды задыхающихся транзакций Postgres не имеет права.

---

## 4. Эталонный рецепт конфигурации под Highload

Чтобы убрать балласт и защитить СУБД от падения по памяти, конфигурация переводится на жесткий паттерн:

1. **Включение пулера (PgBouncer):** Ограничивает реальный наплыв курьеров в СУБД.
2. **Зажатие подключений:** `max_connections = 500` (этого более чем достаточно, чтобы утилизировать все ядра мощного процессора без драки за мутексы-spinlocks).
3. **Расширение лимита объектов:** `max_locks_per_transaction = 512` (дает гигантский запас для любых сложных запросов 1С и сотен временных таблиц) [1С:Предприятие].

**Итог по формуле:** $(500) \times 512 = \mathbf{256\ 000\ \text{слотов}}$. Матрица в ОЗУ стала компактнее, тратит меньше служебной памяти, но ни одна транзакция больше никогда не пробьет свой индивидуальный лимит.
